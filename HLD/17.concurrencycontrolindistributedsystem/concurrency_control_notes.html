BookMyShow problem: 3 users click Book at same time. All read Seat A1 = FREE. All write BOOKED. All get success. Same seat booked by 3 people. This is the concurrency problem.
Without concurrency control
  • User1 read → FREE
  • User2 read → FREE (same time)
  • User3 read → FREE (same time)
  • All write → BOOKED
  • 3 people get the same seat
With concurrency control
  • User1 enters critical section → books seat
  • User2 checks → already BOOKED → fail
  • User3 checks → already BOOKED → fail
  • Only 1 booking succeeds
Single machine vs distributed
Single server — easy
synchronized(bookSeat) { if(seat == FREE) { seat = BOOKED; } }
One JVM — synchronized block works. User2 and User3 wait while User1 is inside.
Distributed — problem
Load Balancer
Server1 (User1) Server2 (User2) Server3 (User3)
Each server has own memory. synchronized only works inside one JVM — useless across machines.
3 tools for distributed concurrency
Transactions
ACID guarantees — either all steps succeed or none. Prevents partial updates like A debited but B not credited.
Locks
Shared lock (multiple readers ok, no writes) or Exclusive lock (only one writer, others wait).
Isolation levels
How strict the DB is about reading concurrent changes. Higher = safer but slower.
Database locks
Shared lock (read lock)
Multiple readers allowed simultaneously. But no writes while any read lock is held.
T1 → read ✔
T2 → read ✔
T3 → read ✔
T4 → write ✗ (blocked)
Exclusive lock (write lock)
Only one transaction at a time. All others blocked — reads and writes.
T1 → write ✔ (holds lock)
T2 → blocked ✗
T3 → blocked ✗
3 concurrency problems
Dirty read
T1 updates price 100→200. T2 reads 200. T1 rolls back to 100. T2 read a value that never officially existed.
Non-repeatable read
T1 reads price=100. T2 updates to 200. T1 reads again → 200. Same query, different result within same transaction.
Phantom read
T1 queries seats WHERE price < 500 → 10 rows. T2 inserts new row. T1 same query → 11 rows. Row count changed.

Rule: Higher isolation = more safety = less concurrency = slower. Choose based on how much inconsistency your system can tolerate.
Optimistic concurrency: Conflicts are rare — so don't lock. Let both proceed. Check for conflict at commit time using a version column.
Version column mechanism
User1 wins (first to commit)
Both read: seat A1, version=1 ↓ User1 commits first
UPDATE seat SET status='BOOKED', version=2 WHERE id=A1 AND version=1
version=1 matches → success ✔ ↓ DB now version=2
User2 fails (conflict detected)
User2 tries same update
UPDATE seat SET status='BOOKED', version=2 WHERE id=A1 AND version=1
version=1 no longer matches → 0 rows updated ✗ ↓ User2 retries with fresh read
Pros
  • No locks — high concurrency
  • No deadlock possible
  • Fast reads
  • Great for high-scale systems
Cons
  • Conflict detected late (at commit)
  • Retry logic needed
  • Bad when conflicts are frequent — many retries = wasted work

Pessimistic concurrency: Conflicts are likely — lock the resource immediately. Others wait. Used when correctness is critical and conflicts are frequent.
SELECT FOR UPDATE
SELECT * FROM seat WHERE id = 'A1' FOR UPDATE;
User1 acquires row lock ↓ User2 → blocked (waits) User3 → blocked (waits) ↓ User1 commits User2 proceeds → sees BOOKED → fail
Deadlock problem
Two transactions each waiting for the other's lock.
T1: lock A → wants B
T2: lock B → wants A

T1 waits for T2
T2 waits for T1
→ Deadlock
DB resolves by detecting the cycle and killing one transaction (the victim). That transaction is rolled back and retried.
Pros
  • Immediate conflict prevention
  • No retry logic needed
  • Guarantees correctness
Cons
  • Lower concurrency — others blocked
  • Deadlock possible
  • High lock contention under load

BookMyShow interview answer: Use pessimistic locking — SELECT ... FOR UPDATE. Row locked immediately, others wait. Only 1 booking succeeds. OR use optimistic locking with version column — let both proceed, conflict detected at commit.

Mental model: Application level → synchronized / mutex (single machine only). Database level → Optimistic (version column) or Pessimistic (FOR UPDATE). Distributed system = always use DB-level solution.
