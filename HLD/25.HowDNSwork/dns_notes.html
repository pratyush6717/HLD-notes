Internet without DNS: To visit Google you'd need to remember 142.250.183.206. 100 websites = 100 IPs to memorise. IP changes = everything breaks. No CDN, no load balancing, no failover. Internet unusable.
Without DNS
  • Must memorise IPs (142.250.183.206)
  • IP changes → user can't reach site
  • No human-readable names
  • No CDN geo-routing possible
  • No load balancing via names
With DNS
  • Type google.com — DNS finds the IP
  • IP changes behind the scenes, name stays same
  • Multiple IPs → automatic load balancing
  • Geo routing → nearest server for user
  • Failover → broken server? DNS points elsewhere
Analogy: Courier system. Without DNS: you give full address "Flat 302, Building 7, Street 5." With DNS: you say "Amazon Warehouse" — courier resolves the full address. DNS is the internet's phonebook.
DNS in real systems
Load balancing (multiple IPs) Geo routing (nearest server) Failover (backup servers) CDN routing (Cloudflare)

Recursive resolution: From the client's view — 1 request, 1 response. Internally the DNS resolver does all the work: Root → TLD → Authoritative. Client sees none of this.
1
Local cache check Browser checks its cache. OS checks hosts file. If IP found → done. No network call needed.
2
DNS Resolver (ISP / 8.8.8.8) Cache miss → client asks resolver. Resolver is your ISP's DNS server or Google's 8.8.8.8. Resolver does all the heavy lifting.
3
Root Server Resolver asks: "Where is .com?" Root replies: "Ask the .com TLD server." 13 root server clusters worldwide.
4
TLD Server (.com) Resolver asks: "Where is example.com?" TLD replies: "Ask the authoritative server at ns1.example.com."
5
Authoritative Server Resolver asks: "What is the IP of www.example.com?" Server replies: 1.2.3.4. This server is the final authority.
6
IP returned to client Resolver caches the result (for TTL duration) → returns IP to client → browser loads page.
Recursive vs Iterative
Recursive
Client makes 1 request. Resolver does all the work: Root → TLD → Authoritative. Client gets final IP.
Client is lazy — resolver does everything. Most common.
Iterative
Client does all the work itself: asks Root, then TLD, then Authoritative — three separate calls.
Client is the worker. Rare — used between resolvers.

TTL — Time to Live
What TTL does
DNS response cached for TTL seconds. After expiry, resolver fetches fresh record. Balances freshness vs performance.
TTL trade-offs
Low TTL (60s) → changes propagate fast, more DNS queries
High TTL (3600s) → cached longer, changes slow to propagate
Interview answer: DNS resolves domain names to IP addresses using a hierarchical system of root, TLD, and authoritative servers — typically via recursive resolution with caching for performance.

Interview questionAnswer
Why is DNS scalable?Hierarchical delegation — root doesn't know everything, delegates to TLD, TLD delegates to authoritative. Distributed responsibility.
Why not single DNS server?Single point of failure, high latency, can't handle billions of queries. Hierarchy + caching solves this.
Why is caching important?Reduces latency (no repeated lookups), reduces load on root/TLD servers. Most queries served from cache.
What is TTL?How long a DNS record is cached. Low TTL = fast propagation, more queries. High TTL = slower propagation, fewer queries.
Recursive vs Iterative?Recursive: resolver does all work for client. Iterative: client queries each server itself.
How does DNS enable load balancing?A record returns multiple IPs. Resolver picks one (round-robin). Different users get different IPs → load spreads.
