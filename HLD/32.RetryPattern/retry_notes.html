The real pain: 90% of work done. Last service call times out. If client retries from scratch → 100% work redone. 1000 clients retrying = 1000× duplicate DB writes, CPU waste, and system overload.
Client retrying from scratch
  • 90% work already done
  • Client doesn't know → starts over
  • 100% work done again
  • 1000 clients = 1000× waste
  • DB overloaded, CPU wasted
Service-level retry (smart)
  • Order Service internally retries Product call only
  • Only the last 10% redone
  • Client sees nothing — transparent recovery
  • Minimal overhead
Mental model:
Retry ≠ blindly repeat
Retry = smart, controlled recovery

Retry ka goal success nahi hai → system ko bachana hai
Phone call analogy: No retry = call fails, give up forever. Bad retry = call 100 times instantly (spam). Good retry = wait a bit → try again → wait more → try again. Smart, respectful, gives the other side time to recover.
Most important rule: Retry galat jaga laga diya → system crash. 4xx errors pe retry = useless. Non-idempotent APIs pe retry = duplicate data. Know when NOT to retry.
Never retry — 4xx errors
400 Bad Request 401 Unauthorized 403 Forbidden 404 Not Found
Input itself is wrong. Retrying same bad input → same bad result. Pure waste. Fix the request, not the retry count.
Never retry — non-idempotent
POST /create-user
Call1 → success (but timeout)
Retry → creates ANOTHER user
Duplicate data. Retry safe only if API is idempotent (GET, PUT, DELETE).
When to retry — safe cases
5xx server errors
  • 500 Internal Server Error
  • 502 Bad Gateway
  • 503 Service Unavailable
  • Transient — server might recover
Network issues
  • Connection timeout
  • Network drop
  • DNS resolution failure
  • Transient infrastructure blip
429 Rate limited
  • Server asking you to slow down
  • Retry with delay
  • Respect Retry-After header if present

How you retry matters as much as whether you retry. Fixed interval → retry storm. Exponential backoff → better. Backoff + Jitter → production-grade.
1 — Fixed interval (don't use)
Same delay every time: 2s → 2s → 2s. Simple but dangerous.

4 — Custom retry
Define your own sequence: 1s → 1s → 2s → 3s → 5s (Fibonacci). Useful for domain-specific SLAs. More complex to maintain.

Definition: Retry handles transient failures by re-attempting requests after a delay to improve success rate without user intervention.

Best practice: Use exponential backoff with jitter to avoid retry storms.

Warning: Retry should only be used for idempotent operations and transient errors (5xx, timeouts).
ConceptOne-liner
Why retryRedo 10% failed work, not 100% → avoid full client restart
When NOT to retry4xx errors (bad input), non-idempotent POST (duplicates), already open circuit
Fixed interval dangerAll clients retry at same time → synchronized spike → thundering herd
Exponential backoffDelay doubles: 100ms → 200ms → 400ms. Gives server time to recover.
JitterRandom noise added → clients spread out → no synchronized retries
IdempotencySafe to retry only if same request = same result (GET, PUT, DELETE)
Max retriesLimit to 3 — combine with circuit breaker to stop retrying on open circuit
Connection to other patterns
Retry comes last in Resilience4j chain:
Rate Limiter → Bulkhead → Time Limiter → Circuit Breaker → Retry

Why? Circuit Breaker is checked first. If circuit is open → Retry never fires → no extra load on failing service. If Retry was before Circuit Breaker → failed retries would all count as failures → circuit opens faster but retries still pile up.
