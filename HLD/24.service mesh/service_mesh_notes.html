The problem: Order Service needs to call Payment Service. Simple. But in production you need: service discovery, load balancing, retry, circuit breaker, mTLS security, telemetry, canary deployments. Duplicated in every service. Nightmare.
Every service must implement
  • Service discovery (which URL?)
  • Load balancing (which instance?)
  • Retry logic (how many times?)
  • Circuit breaker (when to stop?)
  • mTLS security (is caller authorized?)
  • Telemetry (latency, error rate)
  • Canary deployment
Service mesh moves all this out
  • Sidecar proxy handles all of it
  • Developer just calls: call("payment-service")
  • No duplicate code across services
  • Consistent behaviour everywhere
  • Central visibility
Office analogy: Without mesh — every employee finds addresses, handles security, manages retries themselves. Chaos. With mesh — each employee has an assistant (sidecar). Employee just says "talk to payment team." Assistant handles everything.

1
Service discoveryDeveloper says "payment-service". Sidecar asks Consul/Eureka for current IP + port of a healthy instance. No hardcoded URLs, works with dynamic scaling.
2
Load balancing3 Payment Service instances. Sidecar decides which one to call — round robin, least connections, or random. Automatically updated as instances scale.
3
RetryCall failed? Sidecar retries 3 times with exponential backoff. Configurable per route. Service code doesn't need any retry logic.
4
Circuit breakerPayment service failing 50%+ of calls? Sidecar opens circuit — stops calling for 60 seconds. Prevents cascade failure. Auto-resets and tests again.
5
mTLS securityBoth sidecars verify each other's identity (mutual TLS). All service-to-service traffic encrypted. Order Service can't impersonate Payment Service. Zero-trust networking.
6
TelemetryEvery request: latency, error rate, traffic volume logged automatically. No instrumentation code in services. Visualised in Grafana/Jaeger/Kiali.
7
Deployment strategyCanary: send 10% of traffic to new version, 90% to old. Sidecar-level traffic splitting. No code changes. Also supports blue-green deployments.

Interview answer: Microservices communicate via a service mesh using sidecar proxies. Each pod has a sidecar (e.g. Envoy) that handles service discovery, load balancing, retry, circuit breaker, mTLS security, and telemetry — without any code in the service itself.
What-if sandbox
No service mesh
Every service duplicates retry, discovery, security logic. One team's retry config differs from another's. Maintenance nightmare.
Sidecar fails
Entire pod affected — service can't communicate. Sidecars must be extremely lightweight and highly reliable. Envoy is battle-tested for this.
Only API Gateway, no mesh
Gateway handles external traffic only. Internal service-to-service communication still has all the original problems — no retry, no mTLS, no circuit breaker.
