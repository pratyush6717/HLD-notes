Ultimate security rule: Trust nothing from outside. Not the browser, not user input, not any client request.
CSRF
Trusting browser blindly. Browser auto-sends cookies → attacker tricks it into making requests on your behalf.
You click evil link → money transferred without you knowing
XSS
Trusting user input blindly. Attacker injects script into page → victims' browsers execute it.
Comment box with script → steals everyone's session cookies
SQL Injection
Using raw input in query. Attacker inserts SQL into input → query logic changes → database leaked.
Search box → type SQL → full DB dumped

Root cause: Browser automatically sends cookies on every request to a domain — even if the request was triggered by a completely different website.
Attack flow
User logged into bank.com (cookie stored) ↓ Attacker sends: "Click here!" link ↓ user clicks Browser auto-adds bank.com cookie ↓ POST /transfer?amount=10000&to=attacker ↓ server thinks: valid user Money transferred. User unaware.
Restaurant analogy
You're at a restaurant (logged in). A stranger tells the waiter: "Bring 2 pizzas from that table." Waiter trusts it — table is valid. CSRF = stranger making requests using your identity.
Fix — CSRF token
Server generates random token
Embeds it in every form
Client sends token with every request
Server verifies token matches
Attacker doesn't know token → request rejected
Root cause: User-submitted content stored and displayed without sanitisation — browser treats it as code, not text.

Root cause: User input concatenated directly into SQL query — attacker can change the logic of the query itself.
Attack flow
// Attacker types into search box:
' OR 1=1

// Query becomes:
SELECT * FROM users
WHERE name = '' OR 1=1

// 1=1 is always true
// → entire users table returned
Full database leak. Or worse: DROP TABLE users.
Guard analogy
Guard: "What's your name?" Attacker: "' OR let everyone in." Guard follows instruction literally → everyone gets access.
Fix — parameterised query
// BAD — raw input in query
String query = "SELECT * FROM users "
  + "WHERE name = '" + name + "'";
// Attacker controls query logic!
// GOOD — parameterised
PreparedStatement ps = conn.prepareStatement(
  "SELECT * FROM users WHERE name = ?"
);
ps.setString(1, name);
// Input treated as data, never as code
Why it works
  • Input is treated as data — never as SQL code
  • ' OR 1=1 becomes a literal string search
  • Query structure locked before input inserted

What-if sandbox
CSRF token removed
Any website can trigger requests on behalf of logged-in users. Banking, social media, admin actions all exploitable.
Input not validated
XSS and SQL injection both become possible. One unvalidated field = full system compromise.
CORS open (*)
Any domain can call your API. Attacker's site can make authenticated requests to your backend.
