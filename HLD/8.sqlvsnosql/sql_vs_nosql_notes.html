The banking nightmare: User A transfers ₹5000 to B. DB deducts from A but crashes before crediting B. Money lost. SQL's ACID transactions prevent exactly this.
What ACID means
A — Atomicity
All steps of a transaction succeed, or none do. No partial updates. Either A loses ₹5000 AND B gains ₹5000, or neither happens.
C — Consistency
DB moves from one valid state to another. Total money before = total money after. Rules can never be violated.
I — Isolation
Concurrent transactions don't interfere. Two users withdrawing simultaneously still see correct balances.
D — Durability
Once committed, data survives crashes. Even if server dies after confirming, the transaction is not lost.

Fixed schema: Every row must have the same columns. Can't store "hobbies" as an array. Strong structure = strong guarantees.
Use SQL when
Banking Payments Order systems Inventory Accounting / ERP Any relational data
The social media scale problem: Facebook has billions of posts. SQL JOINs across billions of rows = painfully slow. NoSQL stores data denormalized — reads are instant, no JOINs needed.
How NoSQL data looks (MongoDB)
{ "id": 1, "name": "PK", "city": "Pune", "hobbies": ["cricket", "music"] ← array, fine! } { "id": 2, "name": "Rahul" ← no "city" or "hobbies", also fine! }
Flexible schema: Each document can have different fields. New fields can be added without migrating the whole DB. Perfect when data shape changes often.

When to use both (Polyglot persistence)
Real production systems often use both. Uber uses MySQL for payments (SQL) and Cassandra for trip history (NoSQL). Instagram uses PostgreSQL for user data (SQL) and Redis for caching (NoSQL). Choosing the right DB per use case is the skill.

One-line answer: SQL provides strong consistency and ACID transactions for relational data. NoSQL provides flexible schema, horizontal scalability, and high availability for large distributed systems.
Decision tree — 4 questions
1. Is data relational? Tables that JOIN with each other?
YES → SQL
2. Need strong consistency? Banking, payments, inventory?
YES → SQL
3. Massive scale? Billions of records, huge write throughput?
YES → NoSQL
4. Schema changes frequently? Flexible or evolving data structure?
YES → NoSQL

Pro tip for interviews: Don't just say SQL or NoSQL — explain WHY. "I'd use PostgreSQL for payments because ACID transactions are non-negotiable, and Cassandra for activity logs because we need horizontal scale and eventual consistency is acceptable."
