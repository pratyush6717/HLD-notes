The story: You launch on a single server. Product goes viral — 100k req/min. CPU overloads, DB slows, users worldwide experience high latency. Each pain drives the next scaling step.
The 9-step scaling progression
1
Single server
App + DB on one machine. Simple, cheap, but can't scale and is a single point of failure.
2
Separate app & database
Split app server from DB server. Now each can be scaled independently.
3
Load balancer + multiple app servers
Traffic is distributed across multiple app servers. Horizontal scaling + fault tolerance.
4
Database replication (master–slave)
Writes go to master, reads from slaves. Reduces DB bottleneck.
5
Cache (Redis)
Check cache before DB. Cache hit = 1ms vs DB = 50ms. Massive latency reduction.
6
CDN
Static assets served from nearest node to user. Global latency solved.
7
Multiple data centers
Run in India, USA, Europe. Global load balancer routes to nearest. Disaster recovery built in.
8
Message queue (Kafka / RabbitMQ)
Slow tasks (email, SMS, reports) go async via queue. User doesn't wait.
9
Database sharding
Split DB horizontally across machines. Horizontal + vertical sharding. Consistent hashing prevents hotspots.

Step 2 — Separate app & database
Architecture
Client ↓ Application Server ↓ Database Server
Why?
App scaling ≠ DB scaling. Can now give more CPU to app, more storage to DB independently. No longer coupled to one machine.

Step 9 — DB Sharding: When a single DB can't handle the load even after replication, split the data itself across multiple machines.
Vertical scaling vs horizontal scaling
Vertical scaling
Give one machine more power — more CPU, more RAM, more disk. Simple but hits a hardware ceiling.
Example: 16GB RAM → 64GB RAM
Horizontal scaling (sharding)
Add more DB machines. Split data across them. No ceiling — just keep adding nodes.
DB1DB2DB3DB4
Two types of sharding
Horizontal sharding
Split rows across DBs. Same columns, different data ranges.
ShardData
DB1Users A–M
DB2Users N–Z
Vertical sharding
Split columns across DBs. Different attributes on different machines.
ShardColumns
DB1user_id, name
DB2address, phone
Sharding problems
Uneven distribution
  • A–M names far more common than N–Z
  • DB1 gets overloaded
  • Solution: Consistent Hashing
Cross-shard joins
  • Users table in DB1, Orders table in DB2
  • SQL JOIN across shards is very hard
  • Solution: Denormalization
The full picture: Each step added because of a specific pain. Together they form the architecture used by Netflix, Amazon, Uber, and Instagram.
