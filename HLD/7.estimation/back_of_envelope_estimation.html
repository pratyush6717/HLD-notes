System design guessing game nahi hai. Har decision — load balancer, cache size, server count — numbers se driven hona chahiye. Back of the envelope = rough numbers se system capacity estimate karna.
Estimation ke bina
  • Interviewer poochta hai: kitne servers?
  • Tum bolte ho: "scalable design hai"
  • System over-engineered ya under-capacity
  • No numbers = no credibility
Estimation ke saath
  • Load balancer → QPS se decide hota hai
  • Cache size → RAM calculation se
  • Storage → data growth se
  • Servers → requests per second se
Rule: Exact numbers ki zaroorat nahi. Round numbers kaafi hain — 10M, 100M, 1B. Goal rough order of magnitude hai, not precision.
Restaurant analogy
Bina estimation ke
Kitne chefs? Kitne tables? Kitni food storage? Sab guess hoga. Restaurant ya overbuilt hoga ya customers ko wait karwayega.
Estimation ke saath
Customers/day = 1000, meals/customer = 2 → kitchen size, staff, inventory sab calculate ho jaata hai. Same logic software mein.

Goal: QPS (Queries Per Second) nikalna. Yahi decide karta hai kitne app servers chahiye aur load balancer ki zaroorat hai ya nahi.
Facebook QPS calculation
Total users1 Billion
Daily active users (25%)×0.25=250 Million
Queries per user/day5 reads + 2 writes = 7
Total queries/day=250M × 7 = 1.75B
Seconds in a day86,400 ≈ 100,000
QPS=1.75B / 100,000≈ ~18,000 req/sec
QPS rangeWhat it implies
< 1,000Single app server might work
1K – 10KLoad balancer + few servers needed
10K – 100KMultiple server clusters, caching critical
> 100KMultiple data centers, heavy sharding

Most candidates sirf architecture draw karte hain. Senior engineers numbers bhi bolte hain.
5-step answer structure
StepCalculateDrives decision for
1 — Traffic (QPS)DAU × queries/day ÷ 86,400Load balancer, number of app servers
2 — StorageDAU × data/user × daysDB type, sharding, CDN for media
3 — Cache / RAMDAU × cached items × item sizeNumber of cache servers (Redis)
4 — ServersQPS ÷ capacity per serverHorizontal scaling plan
5 — CAP tradeoffConsistency vs AvailabilityDB choice (AP → Cassandra, CP → Zookeeper)
Facebook example summary: ~18K QPS → ~180 app servers · Post storage ~250 GB/day → 500 TB in 5 years · Image storage ~8 TB/day → 16 PB in 5 years · Cache ~750 GB → 10 Redis servers · CAP: AP (availability + eventual consistency)
