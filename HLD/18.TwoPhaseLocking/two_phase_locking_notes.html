Core idea: A transaction must handle locks in exactly two phases — first only acquire (growing), then only release (shrinking). You can never acquire a new lock after releasing one. This guarantees serializable execution.

Rule: Once unlock(A) happens at Step 5 → no new locks allowed. Shrinking phase starts.

✔ Acquire new locks

✔ Release held locks

Problem 1 — Deadlock
Two transactions each waiting for the other's lock. Neither can proceed. Circular wait.
The deadlock scenario
T1: lock(A) → wants lock(B) T2: lock(B) → wants lock(A) T1 waiting for T2 T2 waiting for T1 → forever blocked
Neither can proceed — system stuck.
Detection — wait-for graph
DB builds a directed graph of who is waiting for whom.
T1 → T2 T2 → T3 T3 → T1 ← cycle detected!
Cycle in graph = deadlock. DB aborts one transaction (the victim) and retries it.
Problem 2 — Cascading abort
One transaction's rollback forces other transactions to also abort — because they already read uncommitted dirty data.
The cascading abort scenario
T1: lock(A) → update A = 11 T1: unlock(A) ← releases early T2: lock(A) → reads A = 11 T1: ROLLBACK (A back to 10) T2 has read A = 11 which no longer exists → T2 must also abort
Why Strict 2PL fixes this
In Strict 2PL, T1 holds all locks until commit. T2 cannot read A until T1 commits.
T1: lock(A) → update A = 11 T1: commit → unlock(A) ← only now T2: lock(A) → reads A = 11 ✔ (committed value — safe)
No dirty reads possible → no cascading abort.
3 variants of 2PL — each trades off concurrency vs safety differently.
Basic 2PL
Growing then shrinking. Locks can be released before commit.
lock(A) lock(B) update A update B unlock(A) ← before commit unlock(B) commit
✗ Deadlock possible
✗ Cascading abort possible
Conservative 2PL
Acquire ALL locks before transaction starts. If any unavailable — wait, don't start.
lock(A, B, C) ← all upfront update A update B unlock(A) unlock(B) unlock(C) commit
✔ No deadlock possible
✗ Low concurrency — holds locks longer
✗ Must know all resources upfront
Strict 2PL (most used)
Hold ALL locks until commit. Release only after transaction completes.
lock(A) lock(B) update A update B commit unlock(A) ← only after commit unlock(B)
✔ No cascading abort
✔ Strong consistency
✗ Deadlock still possible (DB detects it)

Interview answer: 2PL is a concurrency control technique that guarantees serializable execution using two phases — growing (only acquire locks) and shrinking (only release locks). Real databases use Strict 2PL where all locks are held until commit, preventing cascading aborts.
