The problem: You hardcode http://product-service:8080. Product Service scales to 3 instances. Your code still calls only :8080. One goes down — code fails. IP changes in containers — URL breaks. Infrastructure changes = code changes.
4 hardcoding disasters
Scaling ignored
  • Product Service: 3 instances (8080, 8081, 8082)
  • Code always calls :8080
  • Other instances idle — wasted resources
Instance down
  • :8080 crashes
  • Code keeps calling :8080
  • System fails even though :8081, :8082 alive
Dynamic infra (containers)
  • Containers auto-scale — IPs change
  • Hardcoded URL instantly stale
  • Kubernetes pod IPs change on restart
Tight coupling
  • Product Service URL changes
  • Order Service must be redeployed too
  • Every service knows every other service's address
Mental model:
Old way: Hardcoded IP → breaks on any infra change
New way: Service name → Discovery → Instance → call

Infrastructure changes → code should NOT change
Phonebook analogy: Old way — memorise every phone number. If someone changes number, you're stuck. With service discovery — you say "Call Rahul." Phonebook finds the current number. You don't care what it is.
Core aha moment: Service does NOT know the URL. It only knows the service name. Discovery resolves the current healthy instance.

Hidden magic — client-side LB
Eureka gives a list of instances. Order Service chooses which to call — round robin, random, or least connections. Load balancing happens client-side without a separate LB component.
Service discovery has 4 core mechanisms. Together they ensure services are always findable, always up-to-date, and never overloading the registry.
1
Registration Service starts → immediately registers with Eureka: "I'm product-service at 192.168.1.5:8080." Eureka adds it to the registry.
2
Heartbeat Every 30 seconds, service pings Eureka: "Still alive." If heartbeat stops → Eureka marks it DOWN → removed from discovery list. Dead instances never returned to clients.
3
Discovery Client asks: "Where is product-service?" Eureka returns list of healthy instances. Client picks one and calls directly.
4
Local cache (very important) On startup, client fetches full registry from Eureka → stores locally. Every request served from cache — Eureka NOT called each time. Cache refreshed periodically (default: 30s). This means Eureka handles 1/N of the traffic it would otherwise get.
Cache refresh trade-off
Refresh too fast
  • Constant calls to Eureka
  • High load on registry server
  • Defeats the purpose of caching
Refresh too slow
  • Stale registry — dead instances still in cache
  • Calls go to down instances
  • Retry + fallback needed
Fix for stale cache: Combine with retry logic. If cached instance is dead → retry with another from list → eventually fresh fetch from Eureka.
Problem: 1 Eureka server = single point of failure. If it goes down, services can't discover each other. Solution: Eureka cluster — 3 servers synced with each other.
Eureka cluster
Eureka 1 ↔ Eureka 2 ↔ Eureka 3
All sync each other — eventual consistency

The aha moment
Each Eureka server is also a client — it registers itself with other Eureka servers. Eureka server = both registry AND service. That's how they stay in sync.
