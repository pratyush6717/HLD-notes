Authorization Code Flow — most used and most secure. Client never touches your password. Code is exchanged server-to-server for a token.
1
Client registration Instagram registers with Google OAuth. Google issues client_id and client_secret. These identify Instagram as a trusted app.
2
User clicks "Login with Google" Instagram redirects user to Google Auth Server with client_id, redirect_uri, scope, and a random state value.
3
User authenticates + consents Google shows login page. User enters email + password. Google shows consent screen: "Instagram wants access to: email, profile." User clicks Allow.
4
Authorization code issued Google redirects back to instagram.com/callback?code=abc123&state=random123. The code is short-lived (minutes).
5
Exchange code for token (server-to-server) Instagram backend sends POST /token with code + client_id + client_secret. Google returns access_token + refresh_token. This step is server-to-server — never exposed to browser.
6
Access resource Instagram calls GET /userinfo with Authorization: Bearer access_token. Google API returns name and email. User is now logged in.
Why the two-step code → token exchange? If the token was sent directly in the browser redirect (URL), it could be leaked in browser history or logs. The code is useless without the client_secret — only Instagram's server has that.
Access token vs refresh token
Access token
Short-lived credential used to access APIs.
expires_in: 1 hour
scope: email profile
use: GET /userinfo
Short expiry limits damage if stolen.
Refresh token
Long-lived token to get new access tokens without re-login.
POST /token
grant_type=refresh_token
refresh_token=xyz...
Stored securely server-side. Never exposed to browser.
Security — state parameter (CSRF prevention)
1. Instagram sends state=random123 in the auth request
2. Google echoes it back in the redirect: callback?code=abc&state=random123
3. Instagram verifies state matches → request is legitimate
4. If state doesn't match → reject — someone forged the redirect

Different use cases need different flows. Authorization Code is the gold standard. Others exist for specific scenarios.
1
Authorization Code most used Web apps and mobile apps. Two-step: code → token. Secure — client_secret never exposed to browser. Use this by default.
2
Implicit Grant deprecated Token returned directly in URL. Faster but insecure — token in browser history. Replaced by Auth Code + PKCE for SPAs.
3
Resource Owner Password Grant Client collects username + password directly. Only for highly trusted first-party apps. Not recommended — defeats the purpose of OAuth.
4
Client Credentials M2M No user involved. Service-to-service communication. Example: your backend calling another service's API. Uses client_id + client_secret directly.
5
Refresh Token Grant When access token expires, client sends refresh token to get a new access token silently. No user re-login needed.

One-line answer: OAuth 2.0 is an authorization framework that lets third-party apps access user data without the user sharing their password — using short-lived tokens instead.
OAuth vs authentication
OAuth 2.0
Authorization — answers "what can this app access?" Grants limited access to resources. Does NOT prove who the user is.
OpenID Connect (OIDC)
OAuth 2.0 + identity layer. Adds an ID token (JWT) that proves who the user is. This is what "Login with Google" actually uses.
ConceptOne-liner
OAuth purposeDelegate limited access to data without sharing password
Authorization CodeBest flow — code exchanged server-to-server for token
Access tokenShort-lived (~1hr), used for API calls

Real systems: GoogleGitHubFacebookMicrosoft — all use OAuth 2.0 + OpenID Connect. When you click "Login with Google" on any site, Authorization Code flow runs behind the scenes.
