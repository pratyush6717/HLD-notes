← Back to Index
The core pain: 1M rows, no index → full table scan → 10,000 pages read → O(N). With B+ Tree index → 3–4 page reads → O(log N). That's the entire reason indexes exist.
Without index — full scan
  • SELECT WHERE emp_id = 915234
  • DB checks Page1, Page2 ... Page10000
  • Every row scanned — O(N)
  • 1M rows = 1M checks — very slow
With B+ Tree index
  • Index tree traversal — O(log N)
  • 1M rows → tree depth ~3 or 4
  • Only 3–4 page reads
  • Direct jump to correct page
How data lives on disk
Disk ↓ Blocks ↓ Data Pages (~8KB each, ~100 rows/page) ↓ Rows
1M rows → 10,000 pages. Without index: all 10,000 must be read.
Library analogy: Books randomly on shelves = full scan. Library catalog (index) = go directly to Shelf B3. DB index works the same — tells you exactly which page (shelf) to go to.
DB never searches rows directly. Always: Index tree → Page pointer → Load page → Binary search on slot directory → Row. Like Google Maps: City → Street → Building → Room.
Complete lookup path
Query: WHERE emp_id = 30 ↓ Step 1 B+ Tree index traversal (root → 25 → right → 30) ↓ finds 30 → Page1 ↓ Step 2 — load page Page1 loaded from disk ↓ Step 3 — slot directory Binary search on slot directory (5, 17, 25, 30, 41) ↓ finds slot3 → offset 420 bytes rowD: emp_id=30, name=D → returned
Page1 — physical storage (messy order)
rowA → emp_id 5
rowD → emp_id 30 ← we want this
rowB → emp_id 17
rowE → emp_id 41
rowC → emp_id 25
Rows stored in insertion order — not sorted. Moving rows is expensive.
Slot directory — sorted mini-index inside page
slot0 → 5 (rowA)
slot1 → 17 (rowB)
slot2 → 25 (rowC)
slot3 → 30 (rowD) ← offset 420 bytes
slot4 → 41 (rowE)
Sorted by key. Binary search applied. O(log slots) — fast.
Cinema hall analogy: Ticket counter (index) → Hall (page) → Row number (slot) → Seat (row). Index doesn't search for the slot — the page itself has a small local directory for that final hop.
One line to remember:
Clustered = data IS the index (rows stored in sorted order)
Non-clustered = index POINTS to the data (rows stay random)
Classroom analogy
Clustered index — students sit in order
Teacher arranges benches by emp_id. Rows physically sorted. Walk directly to bench 30 area.
5 17 25 30 41
Data pages arranged 1,2,4,5... Leaf nodes contain actual rows. Only 1 allowed — you can't sit in two orders simultaneously.

Students sit randomly. Teacher keeps a register: Name → Bench number. Two-step lookup.

Too many indexes = slower inserts. Each index = one more B+ Tree to update on every insert/update/delete. 5 indexes on a 1M row table = 5 tree updates per write.
What happens during insert
INSERT emp_id = 17 ↓ Find correct position in clustered index ↓ Load target data page from disk ↓ if page is full Page Split — Page1 splits into Page1 + Page2 ↓ Update all secondary indexes ↓ Insert complete

Trade-off summary
More indexes — good for reads
  • SELECT queries faster
  • Range queries efficient
  • Sort operations faster
More indexes — bad for writes
  • Every INSERT updates all trees
  • Every UPDATE may trigger page splits
  • More disk I/O on writes
  • More memory for index pages
The aha moment:

Without index: 1,000,000 checks → O(N)
With B+ Tree: ~4 checks → O(log N)

For 1M rows that's the difference between seconds and microseconds.
Complete mental model
Query arrives ↓ B+ Tree index search — O(log N), ~3-4 hops ↓ leaf node stores Index value + pointer to data page ↓ Load data page from disk ↓ Binary search on slot directory ↓ slot → byte offset Row returned
ConceptOne-liner
Index purposeTurn O(N) full scan into O(log N) tree search
B+ TreeBalanced tree — leaf nodes hold value + page pointer
Data page8KB chunk on disk — holds ~100 rows
Slot directoryMini sorted index inside a page — binary search to find exact row
Clustered indexRows physically sorted by index key — data IS the index
Non-clustered indexSeparate index pointing to rows — extra hop (key lookup)
Page splitWhen a full page inserts a new row — page splits into two

Final intuition: DB internally = Disk → Blocks → Pages → Rows. Index = B+ Tree, leaf nodes store value → page pointer. Always: Index → Page → Slot → Row. Never a direct row search.
