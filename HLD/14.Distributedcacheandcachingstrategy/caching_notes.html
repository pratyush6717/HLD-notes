Core idea: Cache = fast memory between app and DB. RAM read ~1ms vs DB read ~100ms = 100× faster. Use it when reads >> writes to reduce DB load and latency.
3 disasters without cache
Slow DB + high latency
  • 1M users × 100ms DB read = slow
  • Page load = 2–3 seconds
  • Bad UX, users leave
DB overloaded
  • Every request hits DB directly
  • DB CPU spikes → crash risk
  • No relief layer
Same data fetched repeatedly
  • Product details, user profiles, homepage — read thousands of times
  • DB hit every single time — pure waste
Library analogy: DB = storage room deep inside. Cache = front desk bookshelf. Student asks for book → check desk first (cache hit = instant). Not there → fetch from storage, keep copy at desk (cache miss → populate). Next student gets it from desk instantly.

Most important for interviews. Each strategy answers: when does the cache get populated? when does it get updated? who is responsible — app or cache?
1 Cache-Aside (most common)
App checks cache. Miss → app fetches from DB → stores in cache. App is in control.
Cache HIT
App ↓ Cache → return data
Cache MISS
App ↓ miss DB read ↓ store Cache populate
✔ Read-heavy systems. Cache failure doesn't break app. Flexible structure.
✗ Stale data risk — DB updated but cache still has old value.
Use: Product catalog, user profiles
2 Read-Through Cache
Cache itself fetches from DB on miss. App only talks to cache — never directly to DB.
App → Cache ↓ on miss Cache → DB ↓ stores and returns Cache → App
✔ App logic simplified — no DB code in app. Cache manages itself.
✗ Cache structure must match DB structure. Less customisation.
3 Write-Around Cache
Writes go directly to DB. Cache is NOT updated — marked invalid. Next read = cache miss → DB fetch → repopulate.
Write → DB directly ↓ Cache invalidated (dirty) ↓ next read Cache miss → DB → repopulate
✔ Cache consistency improves — no stale writes.
✗ Write-heavy systems see many cache misses — defeats the purpose.
4 Write-Through Cache
Every write goes to cache AND DB simultaneously. Cache always stays fresh.
App write ↓
Cache update DB update
(both simultaneously)
✔ Cache always fresh. No stale data.
✗ Write latency doubles (2 writes). Two-phase commit needed. Overhead on every write.
5 Write-Back / Write-Behind (most advanced)
Write to cache → async queue → DB updated later. Extremely fast writes. DB update is deferred.
App write ↓ Cache (instant) ↓ async Queue ↓ later Database
✔ Sub-millisecond writes. DB down = cache still serves. Great for counters, likes, analytics.
✗ Data loss risk — if cache crashes before queue flushes to DB, writes are lost.
Use: Social media likes, analytics counters, high-frequency writes
Single cache server = single point of failure + limited memory. Production me cache cluster use karte hain. Consistent hashing data distribute karta hai across nodes.
Single cache server
  • Single point of failure
  • Limited RAM (one machine)
  • Cannot scale horizontally
Distributed cache cluster
  • Data distributed across nodes
  • If one node fails — only ~1/N keys affected
  • Add nodes to scale capacity

Cache remove kar diya
App → DB for every request. High latency, DB overloaded, poor scalability. System crawls under load.
Write-back + queue fail
Cache updated, DB never gets the write. Data inconsistency. Writes lost permanently if cache also crashes before recovery.
Cache TTL too small
Data expires too fast → many cache misses → DB hit for everything. Cache becomes useless. Defeats the purpose.

Final rule: Cache use karo when reads >> writes. Goal: reduce DB load, reduce latency, increase scalability. TTL properly set karo — too small = many misses, too large = stale data.
