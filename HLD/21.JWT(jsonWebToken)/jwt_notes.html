Session pain: 10 app servers, 3 DB clusters. Every API request = DB query to validate session. DB becomes bottleneck. Horizontal scaling breaks. JWT solves this by putting session data inside the token itself — no DB needed.
Session-based auth problems
  • Server stores session in DB on every login
  • Every request = DB lookup to validate
  • DB becomes bottleneck at scale
  • Session replication across servers = complex
  • Horizontal scaling is hard
JWT solution
  • Session data stored inside the token
  • Server just verifies signature + expiry
  • No DB call needed
  • Stateless — any server can verify
  • Perfect for microservices

JWT = 3 base64-encoded parts separated by dots. HEADER.PAYLOAD.SIGNATURE — Encoded, NOT encrypted. Payload is readable by anyone (paste on jwt.io). Never store passwords in JWT.

Part 1 — Header
{ "alg": "HS256", ← signing algorithm "typ": "JWT" ← token type }
Metadata only. Which algorithm was used to sign.
Part 2 — Payload (claims)
{ "sub": "12345", ← subject (user id) "email": "pk@gmail.com", ← public claim "role": "admin", ← public claim "iss": "auth.myapp.com", ← issuer "exp": 1710000000, ← expiry timestamp "iat": 1709990000 ← issued at }
Readable by anyone — base64 not encrypted. Never put passwords here.
Part 3 — Signature
HMACSHA256( base64(header) + "." + base64(payload), secret_key )
Ensures token wasn't tampered. If payload changes → signature breaks → server rejects. Only server with the secret_key can create valid signatures.

Problem 1 — Token revocation
User banned but token still valid until expiry. JWT is stateless — server has no way to say "ignore this token."
Solution: blacklist
Store revoked token IDs in DB. Check on every request.
Problem: Requires DB lookup again — defeats JWT's stateless benefit.
Solution: short-lived tokens (recommended)
Access token: 10–15 min
Refresh token: 7 days
Damage limited to 10 mins. Refresh token revokable — stored in DB.
Problem 2 — Payload visible
JWT is base64 encoded — not encrypted. Anyone can decode and read the payload. Solution: JWE (JSON Web Encryption) — encrypts the payload. Rarely needed unless very sensitive data in token.
Problem 3 — Secret key leak
If secret_key leaks, attacker can forge any JWT. Solution: kid (key ID) — maintain multiple public/private key pairs, rotate them. Each JWT includes which key was used to sign it.
Access token vs refresh token flow
Login → Auth Server ↓ issues both
Access token (10 min) Refresh token (7 days)
↓ access token expires Client sends refresh token to /token endpoint ↓ New access token issued — no re-login needed
