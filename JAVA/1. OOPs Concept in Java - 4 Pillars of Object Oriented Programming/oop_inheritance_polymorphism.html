OOP has 4 pillars. Interviewer asks "explain OOP" — say all 4 in one breath:
1. Abstraction
Hide complexity. Show only what's needed.
Car.drive() — you don't see engine internals
2. Encapsulation
Protect data. Private fields + public getters/setters.
account.balance — can't set directly, use deposit()
3. Inheritance
Child acquires parent's features + adds own. Code reuse.
Car extends Vehicle — gets engine, adds carType
4. Polymorphism
Same method, different behaviour based on context.
animal.sound() → Dog barks, Cat meows
Inheritance: Child = Parent features + Own features. Child can access parent. Parent CANNOT access child.
class Vehicle { boolean hasEngine; boolean getEngine() { return hasEngine; } } class Car extends Vehicle { String type; String getCarType() { return type; } } Car c = new Car(); c.getEngine(); // ✔ from parent c.getCarType(); // ✔ from child Vehicle v = new Vehicle(); v.getCarType(); // ✗ NOT allowed
4 types of inheritance
Single: A → B
Multilevel: A → B → C
Hierarchical: A → B, C
Multiple: ❌ Not in Java
Diamond problem
class C extends A, B { } // ❌ // c.getEngine() — A or B? Ambiguous.
Fix: Use interfaces — no implementation, so C provides the single implementation. No ambiguity.
Interview line: Java does not support multiple inheritance with classes to avoid ambiguity (diamond problem). Multiple interface implementation is allowed because interfaces have no implementation — the class provides the single concrete implementation.
Polymorphism: One method behaving differently based on context. Two types — compile-time (overloading) and runtime (overriding).
Compile-time — Overloading
Same name, different parameters. Decided at compile time.
int sum(int a, int b) {} int sum(int a, int b, int c) {} String sum(String a, String b) {}
int sum(int a, int b) {} String sum(int a, int b) {} // ❌ only return type different // Compiler checks params only, not return type
Interview line: Overloading depends on method signature (name + params), not return type.
Runtime — Overriding
Same name + same params + same return type. Different implementation. Decided at runtime.
class A { void show() { print("A"); } } class B extends A { @Override void show() { print("B"); } } B obj = new B(); obj.show(); // "B" — runtime decision
Key insight: Which show() runs = decided by the actual object type at runtime, not the reference type.

Most important LLD concept. Every class relationship is IS-A or HAS-A. Getting this right = clean design. Getting it wrong = bad inheritance.
IS-A (Inheritance)
Child IS a type of parent. Use extends.
class Dog extends Animal {} // Dog IS-A Animal class Car extends Vehicle {} // Car IS-A Vehicle
Test: "Dog IS an Animal" → true ✔ → use inheritance
HAS-A (Association)
One object contains another. Use field.
class Car { Engine engine; // Car HAS-A Engine } // ❌ Car extends Engine is WRONG // Car is not "a type of" Engine
Test: "Car IS an Engine" → false ✗ → don't use inheritance
HAS-A has 2 types
Aggregation — weak
Objects can exist independently.
School → Students
School closes → Students still exist ✔
Student removed → school list just empty ✔
Student class deleted → compile error ✗
Composition — strong
Child's lifecycle tied to parent.
House → Rooms
House destroyed → Rooms destroyed too ✗
Rooms can't exist without the House
Created inside parent, not passed in
Senior-level insight:
In real systems: prefer HAS-A (composition) over IS-A (inheritance).
Bad: Car extends Engine ← Car is not a type of Engine
Good: Car has Engine ← Car contains an engine
