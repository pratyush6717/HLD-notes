Class vs Object
Class — the blueprint
class Student { int age; String address; void updateAddress() {} }
Template only. No data yet.
Object — the instance
Student s1 = new Student(); Student s2 = new Student(); // Two separate real students
Real data. Independent state.

Abstraction: Show WHAT to do. Hide HOW it works.
Real-world examples
🚗 Car brake → user presses pedal. Internal hydraulics? Hidden.
📱 ATM → you see withdraw button. Core banking logic? Hidden.
☕ Coffee machine → press button. Brewing mechanism? Hidden.
In code — interface / abstract class
// Abstract — hides implementation interface Payment { void pay(int amount); // WHAT } class UpiPayment implements Payment { public void pay(int amount) { // HOW — hidden from caller callUpiServer(amount); } }
Caller only knows pay(). Doesn't know UPI/card/bank internals.
Why abstraction matters: Simplicity (user sees simple interface), Security (internal logic hidden), Flexibility (change HOW without affecting callers).
Most misunderstood OOP concept. Encapsulation ≠ just grouping. Encapsulation = CONTROLLED ACCESS to data.
// BAD — no encapsulation class Dog { String color; // public by default } Dog d = new Dog(); d.color = null; // anyone can break it
// GOOD — encapsulation class Dog { private String color; // protected public String getColor() { return color; } public void setColor(String color) { if (color != null) { // controlled this.color = color; } } }
Common confusion fixed
Q: If Cat can call dog.getColor(), is encapsulation broken?
A: No!
Encapsulation ≠ No access
Encapsulation = Controlled access

Cat → dog.getColor() → ✔ allowed (public method)
Cat → dog.color directly → ✗ not allowed (private field)
Why it matters
Control — you decide how data flows
Security — hide sensitive fields
Flexibility — change setter logic; callers unaffected
Loose coupling — classes use methods, not fields
Access modifiers are the tools of encapsulation. They decide who can see what.

Typical encapsulation pattern: Fields = private. Getters/setters = public. Logic inside setters = validation. This is Java Beans pattern — used everywhere in Spring entities.

If asked "What is Encapsulation?"
Encapsulation is the concept of binding data and methods together and restricting direct access using access modifiers to ensure controlled interaction.

If asked "What is Abstraction?"
Abstraction is the concept of exposing only essential features while hiding internal implementation details.
