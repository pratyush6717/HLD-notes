LMS / Loan system as one codebase: User + Loan + Payment + Notification + Accounting all together. Change notification logic → redeploy everything. Payment overloaded → scale all services. 10 devs → merge conflicts. One bug → cascades everywhere.
4 monolith disasters
Small change = full risk
  • Changed notification logic only
  • Must redeploy entire app
  • Loan + Payment at risk too
  • Fear of every release
Scaling is wasteful
  • Payment service needs 10× resources
  • Must scale entire monolith
  • User Service also gets 10× (unnecessary)
  • Expensive and inefficient
Team chaos
  • 10 developers, 1 codebase
  • Merge conflicts every day
  • Slow releases, dependency hell
  • One team blocks another
Hidden coupling
  • User change breaks Loan
  • Loan change breaks Payment
  • System is one tangled ball
  • Afraid to touch anything
Monolith — 1 kitchen
1 kitchen makes pizza, burgers, desserts, drinks. Burger section overloaded → whole kitchen slows. Chef changes pizza recipe → affects everything.
Microservices — food court
Pizza kitchen, Burger kitchen, Dessert kitchen. Each independent, scalable, fast. Burger stall busy → only burger stall scales. Others unaffected.
Feynman summary:
Monolith = one big brain (slow, risky, tightly coupled)
Microservices = multiple small expert brains (each does one thing well)
The real interview question: "Kitne microservices banoge?" Wrong answer: "5 services." Right answer: "Depends on domain boundaries — I'd use Domain Driven Design."
DDD — 3 steps to find boundaries
1
Identify the domain What is the system? Chat app, LMS, e-commerce? Name the business domain.
2
List all events Write down everything that happens: User Registered, Message Sent, Notification Sent, Payment Processed, Order Created, etc.
3
Group related events Events that belong to the same business capability → 1 microservice. This group = bounded context.
Chat app example — event grouping
Group 1 — User events
User Registered User Login User Logout
→ User Service
Group 2 — Message events
Message Sent Message Delivered Message Deleted
→ Message Service
Group 3 — Notification events
Notify User Push Alert
→ Notification Service
The aha moment: Microservice = group of tightly related events that belong to the same business capability. Not "what's small enough" but "what belongs together."
What-if sandbox — 3 common mistakes
Merge User + Notification
Different business capabilities combined. Can't scale independently — notification traffic spikes force user service to scale too. Tight coupling returns.
Too many micro-services
Send Service + Deliver Service + Delete Service = 3 services for messaging. Too many network calls, high latency, debugging nightmare. Called a "distributed monolith."
Shared database
All services use one DB = not real microservices. Hidden coupling through data layer. Schema change breaks everything. Can't scale DBs independently.
The biggest mistake: More microservices ≠ better system. Too many → distributed monolith (worse than the original monolith). The right number = number of bounded contexts in the domain.

4 golden rules for interviews
1
Microservice = business capability Not a technical layer (not "database service" or "auth service"). One business function = one service.
2
Use DDD → bounded context Domain events grouped by related capability = bounded context = microservice. Never split by technical concern.
3
Avoid distributed monolith Too many small services with tight coupling = worse than monolith. Services should be loosely coupled, highly cohesive.
4
Each service scales independently Own codebase, own deployment, own DB (ideally), communicates via API or Kafka. Not sharing state.
