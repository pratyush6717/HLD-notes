Core idea: Load balancer = traffic controller. Sits between users and servers, distributes requests so no single server gets overwhelmed. It's the reason large systems stay alive during traffic spikes.
4 disasters without load balancer
Server overload
  • 100,000 users → 1 server
  • CPU 100%, memory full
  • Server crash → website down
Traffic spike
  • Flipkart Big Billion Day: 500k req/sec
  • Single server can't handle
  • Sale fails — revenue loss
Uneven server usage
  • 3 servers — random routing
  • S1: 70k, S2: 10k, S3: 20k
  • S1 crashes, others idle
Scaling chaos
  • Add Server4, Server5 manually
  • Clients don't know which to use
  • No central coordination
Restaurant analogy: Customers = clients. Tables = servers. Host = load balancer. Customer enters → host assigns the best available table. Host knows which tables are free, which are busy. Exactly load balancing.

Two types: L4 works at network layer (IP + port) — fast but dumb. L7 works at application layer (HTTP headers, URL, cookies) — slower but intelligent routing.
L4 — Network Layer
Sirf IP aur port dekhta hai. Content nahi dekhta. Extremely fast.
✔ Very fast — no deep inspection
✔ Works for any TCP/UDP traffic
✗ Can't route based on content
✗ No SSL termination
Example: AWS Network Load Balancer
L7 — Application Layer
HTTP headers, URL, cookies, payload padhta hai. Smart routing possible.
/images → Server1
/api → Server2
✔ SSL termination, content routing
✗ Slightly more overhead
Examples: Nginx, HAProxy, AWS ALB

Round Robin
Requests sequentially cycle through all servers. Equal distribution regardless of capacity.
Req 1 → S1   Req 2 → S2   Req 3 → S1   Req 4 → S2
✔ Simple, equal distribution
✗ Ignores server capacity — weak server gets same load as powerful one
Weighted Round Robin
Each server gets a weight. Higher weight = more requests. Accounts for different server capacities.
S1 weight=3, S2 weight=1
Req 1→S1 · Req 2→S1 · Req 3→S1 · Req 4→S2
✔ Powerful servers handle more traffic proportionally
✗ Static weights — doesn't adapt to real-time load
IP Hash
Server chosen based on hash(client IP). Same user always goes to same server — sticky sessions.
hash(UserA IP) → S1 (always)
hash(UserB IP) → S2 (always)
✔ Session persistence — useful when server holds session state
✗ If users behind proxy → all same IP → all traffic hits one server
Least Connection
New request goes to server with fewest active connections. Dynamic — adapts to real load.
S1: 5 conn · S2: 2 conn → new req → S2
✔ Fair under varying request durations
✗ Slightly more overhead (must track connections)
Weighted Least Connection
Formula: active connections ÷ weight. Lower score wins — combines capacity and real-time load.
S1: 2 conn / weight 10 = 0.2
S2: 1 conn / weight 1 = 1.0 → new req → S1
✔ Best of both — capacity + real-time awareness
✗ More complex to configure
Least Response Time
Formula: active connections × response time. Picks the fastest server available right now.
S1: 3×2ms = 6 · S2: 1×4ms = 4 · S3: 2×0ms = 0 → new req → S3
✔ Optimises for user experience — always picks fastest
✗ Requires response time monitoring overhead

What-if sandbox
Load balancer remove kar diya
All users hit one server. CPU 100%, memory full, crash. No fault tolerance, no scaling. Back to disaster 1.
Round robin with unequal servers
S1 powerful, S2 weak — both get same traffic. S2 crashes under load S1 could easily handle. Fix: use Weighted Round Robin.
IP Hash with proxy users
All users behind corporate proxy → same IP. All traffic → one server → overload. IP Hash breaks. Fix: use Least Connection instead.
