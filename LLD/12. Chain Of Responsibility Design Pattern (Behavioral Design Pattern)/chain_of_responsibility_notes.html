Why better than if-else
// Rigid if-else if(level == DEBUG) ... else if(level == INFO) ... else if(level == ERROR) ...
COR → each handler is independent class. Add new handler → zero changes to existing code (OCP).
Key design insight: Handler IS-A Handler (substitution) + HAS-A next Handler (chain link). This IS-A + HAS-A combo is what makes the chain work.
// Abstract Handler abstract class LogHandler { protected LogHandler next; void setNext(LogHandler next) { this.next = next; } abstract void handle(String level, String message); // Helper — pass to next if set void passToNext(String level, String message) { if (next != null) next.handle(level, message); } }
// Concrete Handlers class DebugHandler extends LogHandler { void handle(String level, String msg) { if (level.equals("DEBUG")) System.out.println("[DEBUG] " + msg); else passToNext(level, msg); // forward if not mine } } class InfoHandler extends LogHandler { void handle(String level, String msg) { if (level.equals("INFO")) System.out.println("[INFO] " + msg); else passToNext(level, msg); } } class ErrorHandler extends LogHandler { void handle(String level, String msg) { System.out.println("[ERROR] " + msg); // handles everything else } }
// Chain setup — constructor chaining LogHandler chain = new DebugHandler( new InfoHandler( new ErrorHandler(null) ) ); // Chain built inside → outside // Client just calls the first chain.handle("INFO", "Server started"); // → InfoHandler handles it
Aha on constructor chaining: Chain builds from inside → outside (last handler first). But requests flow outside → inside (first handler first). The nesting is the chain — deepest node has null as next.
Two distinct variants of Chain of Responsibility. Most people only know Type 1. Type 2 is what Spring Security and Servlet Filters use.
Type 1 — Single handler (Stop)
First handler that can process it → handles → stops. Rest don't see the request.
H1 ✗ → H2 ✔ STOP → H3 never runs
Examples: logging (log4j), ATM dispensing, approval workflows
Type 2 — All handlers process (Pass-through)
Every handler runs. Each adds behaviour, then passes forward. Pipeline style.
H1 ✔ → H2 ✔ → H3 ✔ → done
Examples: Spring Security filters, Servlet filters, Node.js middleware
Spring Security is Type 2: Request → Authentication Filter → Authorization Filter → CORS Filter → Controller. Every filter runs in sequence. This is COR as a pipeline.
COR in production systems
Logging — log4j/slf4j
DEBUG → INFO → WARN → ERROR → FATAL. Each level is a handler. Request flows until matching level handles it.
Servlet Filters
HTTP request → Filter1 (auth) → Filter2 (logging) → Filter3 (compression) → Controller. All filters run (Type 2).
Spring Security
Authentication chain — multiple filters handle different security concerns. Each passes request along after processing.
Node.js Express middleware (Type 2)
HTTP Request → Auth Middleware → Rate Limit → Logger → Route Handler
Each middleware calls next() to pass to the next. COR pattern exactly.
One-liner: Chain of Responsibility decouples sender from receiver by passing the request through a chain until a handler processes it.

Category: Behavioural
Advantages
  • Loose coupling — client doesn't know who handles
  • Extensible — new handler = new class (OCP)
  • Clean — no giant if-else in one place
  • Dynamic — chain configurable at runtime
Disadvantages
  • Request may go unhandled if no match
  • Debugging tricky — flow is indirect
  • Long chain = performance impact
