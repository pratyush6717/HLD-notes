One line: Builder constructs complex objects step-by-step — separating construction from representation.

Trigger: Constructor is messy? Too many optional fields? → Use Builder. 
// BAD — messy constructor
new Student(1, "PK", 25, "Dad name",
  "Mom name", null, null, "address", ...);
// Which param is what? No idea.
// GOOD — Builder Pattern
Student s = new Student.Builder(1)
    .setName("PK")
    .setAge(25)
    .build();
// Readable. Only set what you need.
Without Builder
  • Huge constructor with 10+ params
  • Wrong order = wrong object (no compile error)
  • Constructor explosion for optional combos
  • Hard to read what's being set
With Builder
  • Mandatory fields in Builder constructor
  • Optional fields via fluent setters
  • Readable — each line tells you what it sets
  • build() creates the final immutable object
Method chaining aha: Each setter returns this (the Builder) — that's what enables fluent chaining. build() at the end assembles and returns the final object.
Real Java example — StringBuilder IS Builder
new StringBuilder()
    .append("Hello")   // step
    .append(" PK")     // step
    .append("!")        // step
    .toString();        // build() — returns final string
// Product — private constructor
class Student {
  int roll;      // mandatory
  String name;   // optional
  int age;       // optional
  String father; // optional

  // Private — only Builder can create
  private Student(Builder b) {
    this.roll   = b.roll;
    this.name   = b.name;
    this.age    = b.age;
    this.father = b.father;
  }

  // Inner Builder class
  static class Builder {
    int roll;      // mandatory
    String name;
    int age;
    String father;

    public Builder(int roll) {    // mandatory in constructor
      this.roll = roll;
    }

    public Builder setName(String name) {
      this.name = name;
      return this; // 🔥 returns Builder for chaining
    }

    public Builder setAge(int age) {
      this.age = age;
      return this;
    }

    public Builder setFather(String father) {
      this.father = father;
      return this;
    }

    public Student build() {
      return new Student(this); // assemble
    }
  }
}
// Client — clean, readable, flexible
Student s1 = new Student.Builder(1)
    .setName("PK")
    .setAge(25)
    .build();

// Different student — different optional fields
Student s2 = new Student.Builder(2)
    .setName("Rahul")
    .setFather("Mr. Sharma")
    .build(); // age not set — fine!
Key design decision: Mandatory fields go in Builder's constructor (can't skip). Optional fields are setters (skip freely). build() creates a valid object regardless of which optionals were set.
90% of people confuse these. They look similar because both involve "adding things step by step" — but they solve completely different problems.
Builder — Construction
Creates a new object step by step. Once built, object is fixed. Used at object creation time.
// Static structure — set at build time
Pizza p = new Pizza.Builder()
    .size("large")
    .cheese(true)
    .mushroom(true)
    .build();
// Pizza is now immutable
Problem: PizzaWithCheese + PizzaWithMushroom + PizzaWithCheeseAndMushroom = class explosion
Decorator — Enhancement
Adds features to an existing object dynamically at runtime. Object keeps its interface. Infinite combinations.
// Dynamic stacking — flexible
Pizza p = new Mushroom(
    new Cheese(
        new BasePizza()
    )
);
p.getCost(); // 200+20+40 = 260

Both are Creational patterns — but they handle different levels of complexity.
Factory — One-shot creation
Hides which class is created. Simple objects. Client passes type, gets object.
Shape s = ShapeFactory.getShape("CIRCLE");
// Done in one call. No steps.
Builder — Step-by-step creation
Handles complex objects with many optional fields. Client controls every step.
Student s = new Student.Builder(1)
    .setName("PK").setAge(25).build();
// Multiple steps, full control.
