The key line that makes Composite powerful: Directory.ls() just calls child.ls() for each child — it doesn't care if the child is a File or another Directory. Recursion is automatic.
// Component interface — the contract interface FileSystem { void ls(); } // Leaf — no children, does actual work class File implements FileSystem { String name; File(String name) { this.name = name; } public void ls() { System.out.println("📄 " + name); } }
// Composite — contains children, delegates class Directory implements FileSystem { String name; List<FileSystem> children = new ArrayList<>(); Directory(String name) { this.name = name; } void add(FileSystem f) { children.add(f); } public void ls() { System.out.println("📁 " + name + "/"); for (FileSystem child : children) { child.ls(); // 🔥 no instanceof — just call } } }
// Usage — builds the tree Directory root = new Directory("root"); Directory docs = new Directory("Documents"); docs.add(new File("resume.pdf")); docs.add(new File("cover.docx")); root.add(docs); root.add(new File("notes.txt")); root.ls(); // recursively prints entire tree
HAS-A insight: Directory IS-A FileSystem (implements interface) + HAS-A list of FileSystem objects. The children can be Files or Directories — it doesn't care. Same IS-A + HAS-A combo as Decorator and Proxy.
Expression trees are Composite Pattern. Number = Leaf (no sub-expressions). Operation = Composite (has left + right sub-expressions). evaluate() recurses through the tree.

// Interface interface Expression { int evaluate(); } // Leaf — just a number class Number implements Expression { int val; Number(int val) { this.val = val; } public int evaluate() { return val; } } // Composite — an operation class Add implements Expression { Expression left, right; Add(Expression l, Expression r) { this.left = l; this.right = r; } public int evaluate() { return left.evaluate() + right.evaluate(); } } class Multiply implements Expression { Expression left, right; Multiply(Expression l, Expression r) { this.left = l; this.right = r; } public int evaluate() { return left.evaluate() * right.evaluate(); } } // Build: 2 * (1 + 7) Expression expr = new Multiply( new Number(2), new Add(new Number(1), new Number(7)) ); System.out.println(expr.evaluate()); // 16

Both use IS-A + HAS-A same interface trick.

Composite: HAS-A list of children → tree
Decorator: HAS-A one wrapped object → chain

Composite vs COR (deep)
Both pass work along a structure.

Composite: ALL nodes participate (ls() recurses everywhere)
COR: ONE node handles (stops when matched)

Composite = full traversal
COR = first match wins
Definition: Composite Pattern allows you to treat individual objects and compositions of objects uniformly — removing instanceof checks using polymorphism.

Ultimate trigger: If the problem looks like a TREE → think Composite.
When to use
Use Composite when
  • Tree / hierarchy structure in the problem
  • Objects contain same-type objects (recursive composition)
  • Need to treat whole and part the same way
  • Want to eliminate instanceof checks with polymorphism
Real-world examples
File system (File + Directory) UI components (Button inside Panel inside Window) HTML DOM (element contains elements) Expression trees (calculator, compilers) Menu systems (MenuItem inside Menu) Org chart (Employee inside Department)
