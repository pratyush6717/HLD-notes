One line: Bridge decouples abstraction from implementation so that both can vary independently.

The real problem it solves: When both the abstraction hierarchy AND the implementation hierarchy need to grow — without being tightly coupled to each other.

Add new animals → abstraction grows independently. Add new breathe processes → implementation grows independently. No class explosion.
IS-A + HAS-A again: LivingThing IS-A abstraction (abstract class) + HAS-A BreatheImplementer (reference to implementation). Client injects the right implementer at construction. Abstraction delegates to it.
Simple inheritance works fine — until you need to add a new breathing process independently of the animal hierarchy.
Simple inheritance (problem)
LivingThing (abstract)
└── breatheProcess() abstract
├── Dog → breatheProcess() { nose }
├── Fish → breatheProcess() { gills }
└── Tree → breatheProcess() { leaves }
The tightly coupled problem
  • Breathing process baked into each animal class
  • Want a new breathing process (e.g. Bird via beak)?
  • Must add Bird class first — no Bird, no Bird breathe
  • Breathe process can't exist independently
  • Both hierarchies grow together — can't scale independently
Root problem: Implementation (how to breathe) is tightly coupled to abstraction (what animal). One cannot change without the other. Bridge pattern separates them into independent hierarchies.
Extract the implementation into its own interface. Abstraction holds a reference to it. Both hierarchies now grow independently.
// Implementation hierarchy — grows independently interface BreatheImplementer { void breathe(); } class LandBreathe implements BreatheImplementer { public void breathe() { System.out.println("Nose → O2 in, CO2 out"); } } class WaterBreathe implements BreatheImplementer { public void breathe() { System.out.println("Gills → absorb O2 from water"); } } class TreeBreathe implements BreatheImplementer { public void breathe() { System.out.println("Leaves → CO2 in, O2 out"); } } // Add XYZBreathe? Add independently. No animal changes needed.
// Abstraction hierarchy — holds reference to implementer abstract class LivingThing { BreatheImplementer breatheImpl; // HAS-A implementer LivingThing(BreatheImplementer impl) { this.breatheImpl = impl; } abstract void breatheProcess(); } class Dog extends LivingThing { Dog(BreatheImplementer impl) { super(impl); } public void breatheProcess() { breatheImpl.breathe(); } // delegates } class Fish extends LivingThing { Fish(BreatheImplementer impl) { super(impl); } public void breatheProcess() { breatheImpl.breathe(); } }
// Client — injects the right implementer LivingThing fish = new Fish(new WaterBreathe()); fish.breatheProcess(); // "Gills → absorb O2 from water" LivingThing dog = new Dog(new LandBreathe()); dog.breatheProcess(); // "Nose → O2 in, CO2 out" // Change breathe method at runtime — same as Strategy! LivingThing tree = new Tree(new XYZBreathe()); // new process, no Tree changes

Strategy — Change behaviour at runtime
Intent: swap the algorithm/behaviour of a single object dynamically. Focus = one object + swappable behaviour.

Intent: design for long-term growth. Both abstraction AND implementation expand independently over time. Focus = scalability of two hierarchies.

The honest truth from the lecture: UML-wise and code-wise — Bridge and Strategy are almost the same. The difference is intent only.

Strategy = "I want to change this behaviour now"
Bridge = "I want both hierarchies to grow without coupling"
Definition: Bridge pattern decouples an abstraction from its implementation so that the two can vary independently.

Category: Structural — separates a complex class into two independent hierarchies.
When to use
Use Bridge when
  • Both abstraction and implementation need to grow independently
  • Tight coupling prevents adding new processes without new classes
  • Want to avoid a Cartesian product of classes (3 animals × 4 breathe = 12 classes)
  • Implementation should be swappable at runtime
