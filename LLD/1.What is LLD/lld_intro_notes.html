Design patterns — 3 categories. Every pattern fits one of these. Know the category → know the problem it solves.
Creational
Object kaise banega?
Controls how objects are created. Hides the complexity of instantiation from the caller.
Singleton — one object only
Factory — decide which object
Builder — step-by-step creation
Structural
Objects ka arrangement kaise hoga?
How objects combine to form larger structures. Defines the skeleton of the system.
Adapter — bridge incompatible interfaces
Decorator — add behaviour dynamically
Composite — tree structure
Behavioural
Objects baat kaise karenge?
How objects interact and communicate. Defines the flow of the system.
Observer — notify on change
Strategy — swap algorithms
Command — encapsulate action
Ultimate aha moment:
Creational → banega kaise (creation)
Structural → judega kaise (structure)
Behavioural → chalega kaise (interaction / flow)
The most important LLD concept. Every class relationship is either IS-A (inheritance) or HAS-A (association). Get this right → diagram draws itself.
IS-A (Inheritance)
Child IS a type of parent
Car IS-A Vehicle
Dog IS-A Animal
Admin IS-A User
class Vehicle {} class Car extends Vehicle {}
When: child has all parent properties + adds more. "Is a kind of" relationship.
HAS-A (Association)
One object contains another
Car HAS-A Engine
Library HAS-A Books
Order HAS-A Payment
class Car { Engine engine; // HAS-A }
When: one object uses/contains another. "Is made of" or "uses" relationship.
Diagram shortcut (interview hack)
Car ──extends──▷ Vehicle     ← IS-A (inheritance arrow)
Car ──────────◇ Engine      ← HAS-A (diamond = contains)
Car ──────────◇ Wheels      ← HAS-A
Instead of memorising UML symbols — just think IS-A or HAS-A, then draw the arrow accordingly.
HAS-A has 2 types based on how tightly the objects are coupled. Key question: "If the parent dies, does the child die too?"
Aggregation — Weak HAS-A
Objects are independent
Library HAS-A Books
Library closes → books still exist
Books can belong to another library
class Library { List<Book> books; // Books exist independently }
Loose relation. Objects have independent lifecycles.
Composition — Strong HAS-A
Objects are dependent
House HAS-A Rooms
House demolished → rooms gone too
Rooms can't exist without the house
class House { Room room = new Room(); // Created inside House // Room dies when House dies }
Tight relation. Child's lifecycle is fully controlled by parent.
FeatureAggregationComposition
Relationship strengthWeak (loose)Strong (tight)
LifecycleIndependent — child survives parent's deathDependent — child dies with parent
ExampleLibrary → BooksHouse → Rooms
Code signalObject passed in (constructor/setter)Object created inside the class
UML symbolEmpty diamond ◇Filled diamond ◆
Quick test: "If the container object is deleted, does the contained object still exist?"
Yes → Aggregation (weak). No → Composition (strong).

Interview approach — always follow this order
1
Identify classes — what are the nouns in the problem? Each major entity = a class.
2
Define relationships — IS-A or HAS-A? Aggregation or Composition? Draw before you code.
3
Apply patterns — which category? Which pattern fits? Then write the code.
