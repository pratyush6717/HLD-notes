IS-A + HAS-A combo: Adapter IS-A adapter interface (same contract as client expects) + HAS-A adaptee (holds existing system to delegate to). Both relationships in one class.
Scenario: Weight machine returns pounds. Client only understands kilograms. Can't change the existing weight machine. Add an adapter.
// Adaptee — existing system, returns pounds
// Can't change this — it's an external/legacy product
interface WeightMachine {
  double getWeightInPound();
}

class WeightMachineForBabies implements WeightMachine {
  public double getWeightInPound() {
    return 28.0; // always returns in pounds
  }
}
// Adapter interface — what CLIENT expects
interface WeightMachineAdapter {
  double getWeightInKg();
}

// Concrete adapter — IS-A adapter interface, HAS-A adaptee
class WeightMachineAdapterImpl implements WeightMachineAdapter {
  WeightMachine weightMachine; // HAS-A existing system

  WeightMachineAdapterImpl(WeightMachine wm) {
    this.weightMachine = wm;
  }

  public double getWeightInKg() {
    double weightInPound = weightMachine.getWeightInPound(); // talk to adaptee
    return weightInPound * 0.45;  // translate — pounds to KG
  }
}
// Client — only knows WeightMachineAdapter
WeightMachineAdapter adapter = new WeightMachineAdapterImpl(
  new WeightMachineForBabies()
);

double weightInKg = adapter.getWeightInKg(); // ✔ gets KG
// Client never knew pounds existed
Adapter is everywhere — in hardware, software, and system integration. Any time two incompatible systems need to talk, an adapter is the solution.
Power adapter
Socket (square pins) + Power plug (oval pins) = incompatible. Adapter bridges them. Classic real-world example.
XML → JSON
Server returns XML. Client needs JSON. Adapter parses XML and outputs JSON. Client never sees XML.
External API shield
Your company reads from adapter. External API changes format? Adapter absorbs the change. Your 10 internal components are unaffected.
External API change — Adapter absorbs it
Before: External API returns format=1 → Your 10 components read format=1
API changes to format=1# → All 10 components break

With Adapter: 10 components read from adapter → adapter reads format=1# → converts to format=1
API changes → only adapter changes. 10 components untouched.
Java standard library examples: Arrays.asList() adapts an array to a List interface. InputStreamReader adapts a byte InputStream to a character Reader. These are Adapter pattern in action.

Adapter vs Proxy
Both have IS-A + HAS-A. Key difference: Proxy uses SAME interface as the real object. Adapter CONVERTS to a DIFFERENT interface. If interfaces match → Proxy. If interfaces differ → Adapter.
Adapter vs Decorator
Decorator wraps and enhances — same interface, adds features. Adapter translates — different interface, enables compatibility. Decorator = "power-up." Adapter = "translator."
Definition: Adapter pattern bridges the gap between two incompatible interfaces — allowing an existing class to be used where a different interface is expected, without modifying it.
