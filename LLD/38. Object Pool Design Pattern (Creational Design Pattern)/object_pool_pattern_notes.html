One line: Object Pool manages a fixed set of reusable objects — clients borrow, use, and return. Create once, reuse many times.

Category: Creational pattern.
Pool state — free vs in-use
Free pool
conn-1 conn-2 conn-3
In-use pool
conn-4 conn-5
Client calls getResource() → moves from free to in-use
Client calls releaseResource() → moves back to free
Real-world examples
💻 DB Connection Pool (HikariCP)
🧵 Thread Pool (ExecutorService)
🔌 Socket Pool
🎮 Game object pooling (bullets, particles)
Why it exists
Creating DB connections is expensive (100ms+)
Without pool: create → use → destroy per request
With pool: borrow → use → return → reuse
Performance boost + controlled resources
Interviewers don't care about theory — they care whether you mention these two. Missing either = red flag.
❌ Wrong implementation
// Multiple pool managers — uncontrolled resources Pool pool1 = new Pool(); // 3 connections Pool pool2 = new Pool(); // 3 more connections Pool pool3 = new Pool(); // 3 more connections // Total: 9 connections — no control
Also wrong: no thread safety → two threads get same connection → data corruption
✔ Must-have 1 — Singleton Pool Manager
Only ONE pool exists globally. Prevents uncontrolled resource multiplication.
class ConnectionPool { private static ConnectionPool instance; // singleton private Queue<Connection> freePool = new LinkedList<>(); private ConnectionPool(int size) { for (int i = 0; i < size; i++) { freePool.add(new Connection()); } } public static synchronized ConnectionPool getInstance() { if (instance == null) instance = new ConnectionPool(5); return instance; } }
✔ Must-have 2 — Thread Safety
Without it: Thread A and Thread B both see the same connection as "free" → both borrow it → data corruption.
public synchronized Connection getConnection() { if (freePool.isEmpty()) return null; // or block Connection c = freePool.poll(); inUse.add(c); return c; } public synchronized void releaseConnection(Connection c) { inUse.remove(c); freePool.add(c); }
// Full Object Pool — Singleton + Thread-safe class ConnectionPool { private static ConnectionPool instance; private Queue<Connection> freePool = new LinkedList<>(); private Set<Connection> inUse = new HashSet<>(); private int maxSize; private ConnectionPool(int size) { this.maxSize = size; for (int i = 0; i < size; i++) { freePool.add(createConnection()); } } public static synchronized ConnectionPool getInstance(int size) { if (instance == null) instance = new ConnectionPool(size); return instance; } public synchronized Connection getConnection() { if (!freePool.isEmpty()) { Connection c = freePool.poll(); inUse.add(c); return c; } // All connections busy return null; // or block until available } public synchronized void releaseConnection(Connection c) { inUse.remove(c); freePool.add(c); // back to free pool } private Connection createConnection() { return new Connection("jdbc:mysql://..."); } } // Client usage ConnectionPool pool = ConnectionPool.getInstance(5); Connection conn = pool.getConnection(); // use conn... pool.releaseConnection(conn); // MUST return — or leak
Say any of these in interview → interviewer impressed. These distinguish a senior engineer from a junior.
1
Block instead of returning null
When pool is exhausted, use BlockingQueue or wait()/notify() to make client wait until a connection is released, instead of returning null.
2
Timeout support
Wait max N seconds for a connection. If none available → fail with exception. Prevents threads waiting forever.
3
Health check before reuse
DB connection may have gone stale (timeout). Validate before giving to client: conn.isValid(). Discard + create new if dead.
4
Resource leak prevention
If client forgets releaseConnection() → connection lost forever. Use try-with-resources or AutoCloseable wrapper to auto-return on scope exit.
5
Use libraries in real systems
HikariCPApache Commons PoolExecutorService — battle-tested pool implementations. Say this in interview for instant +points.
Definition: Object Pool is a creational pattern that manages a fixed number of reusable objects, allowing clients to borrow and return them — reducing the cost of frequent object creation.

Senior-level bonus: "In real systems we use HikariCP for DB pools and ExecutorService for thread pools rather than building our own."
4 common interview mistakes
Mistakes = red flags
  • Forget Singleton → multiple pools → uncontrolled resources
  • No thread safety → two threads get same object
  • Return null when pool empty → client NPE
  • No resource leak handling → connections lost
Correct answers
  • Singleton → ONE pool globally
  • synchronized / BlockingQueue → thread safe
  • Block with timeout → no null return
  • AutoCloseable / try-with-resources → no leak
