Golden rule: View = dumb. Model = dumb. Controller = smart. No logic in View or Model — Controller connects everything.
Model = Data Layer — "WHAT data is"
Entities / POJOs, DB logic, Repository / DAO. No business logic here.
class ParkingSpot { int id; boolean occupied; } class Car { String number; }
Controller = Brain — "WHAT to do"
Business logic, validation, flow control. Receives request, processes, calls Model, returns to View.
void parkCar(Car car) { validate(car); // validation ParkingSpot s = assignSpot(car); // logic spotRepo.save(s); // calls Model return spotResponse(s); // sends to View }
View = UI Layer — "HOW it is shown"
Frontend HTML/React OR JSON API response in backend. No logic — just display.
{ "spotId": 10, "status": "occupied" } // In backend = JSON response // In frontend = React component

MVC vs Layered (Spring nuance): Spring's Controller → Service → Repository is layered architecture that fits inside MVC. Service layer = more granular Controller. Repository = Model's DB access layer. Both work together.
Key correction to open with: "Technically, MVC is an architectural pattern, not a design pattern. It organises code into Model (data), View (display), and Controller (logic) for separation of concerns."
Why MVC is used
Advantages
✔ Separation of concerns
✔ Change DB → only Model changes
✔ Change UI → only View changes
✔ Team parallel work (frontend + backend)
✔ Scalable independently
Disadvantages
✗ Overhead for small apps
✗ More classes/files to manage
✗ Controller can become fat (God class)
