One line: Objects communicate through a central mediator instead of directly with each other.

Memory trick: "Don't talk to each other — talk to the boss." Boss = Mediator.
Without Mediator — spaghetti
Bidder A ↔ Bidder B
Bidder A ↔ Bidder C
Bidder B ↔ Bidder C
Bidder C ↔ Bidder D
N objects = N(N-1)/2 connections
With Mediator — star topology
Bidder A → Auction
Bidder B → Auction
Bidder C → Auction
Auction → notifies all
N objects = N connections only
Without Mediator
  • Every bidder knows every other bidder
  • 10 bidders = 45 connections
  • Add one bidder → update 10 others
  • Tight coupling everywhere
With Mediator
  • Bidders only know the auction (mediator)
  • 10 bidders = 10 connections
  • Add one bidder → only add to mediator
  • Loose coupling
Real-world examples: Online AuctionAir Traffic Control (tower)Chat App (server)Multiplayer Game Lobby — in all of these, objects talk to a central coordinator, not to each other.
4 components — always the same structure.
1
Mediator interface
Contract for central coordinator. Defines addBidder() and placeBid(). Multiple mediator types possible (e.g. Dutch auction, English auction).
2
Concrete Mediator (Auction)
Maintains list of colleagues. When a bid comes in — notifies all other bidders. All business rules live here (not in bidders).
3
Colleague interface (Bidder)
Contract for all participants. Defines placeBid() and receiveBidNotification().
4
Concrete Colleague (Bidder impl)
HAS-A mediator reference. placeBid() delegates to mediator. receiveBidNotification() reacts to others' bids. Knows ONLY the mediator — never other bidders.
// 1. Mediator interface interface AuctionMediator { void addBidder(Colleague bidder); void placeBid(Colleague bidder, int amount); } // 2. Concrete Mediator — all rules live here class Auction implements AuctionMediator { List<Colleague> bidders = new ArrayList<>(); public void addBidder(Colleague b) { bidders.add(b); } public void placeBid(Colleague bidder, int amount) { for (Colleague b : bidders) { if (b != bidder) { // don't notify the bidder themselves b.receiveBidNotification(bidder.getName(), amount); } } } }
// 3. Colleague interface interface Colleague { void placeBid(int amount); void receiveBidNotification(String bidder, int amount); String getName(); } // 4. Concrete Colleague class Bidder implements Colleague { String name; AuctionMediator mediator; // HAS-A mediator, NOT other bidders Bidder(String name, AuctionMediator mediator) { this.name = name; this.mediator = mediator; mediator.addBidder(this); // self-register } public void placeBid(int amount) { System.out.println(name + " placed bid: " + amount); mediator.placeBid(this, amount); // goes through mediator } public void receiveBidNotification(String bidder, int amount) { System.out.println(name + " notified: " + bidder + " bid " + amount); } public String getName() { return name; } }
// 5. Client AuctionMediator auction = new Auction(); Bidder alice = new Bidder("Alice", auction); Bidder bob = new Bidder("Bob", auction); Bidder carol = new Bidder("Carol", auction); alice.placeBid(1000); // Output: Alice placed bid: 1000 // Bob notified: Alice bid 1000 // Carol notified: Alice bid 1000 bob.placeBid(2000); // Output: Bob placed bid: 2000 // Alice notified: Bob bid 2000 // Carol notified: Bob bid 2000
Mediator and Observer look similar — both involve objects being notified. The difference is the role of the central object.
Mediator — coordination
Central object controls how colleagues interact. Contains interaction logic. Bidder A's action triggers Mediator to decide what to tell Bidder B.
Bidder → Auction → Bidder
Auction decides the rules
Colleagues don't know each other
Observer — broadcast
Subject broadcasts an event. Observers react independently. No coordination logic — just notification.
Subject → notify all observers
No back-communication
One-way event push

Definition: Mediator is a behavioural pattern that reduces coupling between objects by making them communicate through a central mediator instead of directly interacting.

Final model: Colleagues = Users. Mediator = Server. Users talk to server, server tells others.
