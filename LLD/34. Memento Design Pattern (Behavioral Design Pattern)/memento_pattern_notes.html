// Originator — decides what to save and restore class Configuration { int height, width; void setConfig(int h, int w) { this.height = h; this.width = w; } Memento createMemento() { return new Memento(height, width); // snapshot } void restore(Memento m) { this.height = m.height; this.width = m.width; } } // Memento — immutable snapshot class Memento { final int height, width; // final = immutable Memento(int h, int w) { this.height = h; this.width = w; } } // Caretaker — manages history, never reads memento content class Caretaker { Stack<Memento> history = new Stack<>(); void save(Memento m) { history.push(m); } Memento undo() { if (!history.isEmpty()) return history.pop(); return null; } }
// Client — save + undo flow Configuration config = new Configuration(); Caretaker caretaker = new Caretaker(); // State 1 — 5×10 config.setConfig(5, 10); caretaker.save(config.createMemento()); // snapshot saved // State 2 — 7×12 config.setConfig(7, 12); caretaker.save(config.createMemento()); // snapshot saved // State 3 — 9×14 (not saved) config.setConfig(9, 14); System.out.println(config.height); // 9 // UNDO — back to state 2 config.restore(caretaker.undo()); System.out.println(config.height); // 7 // UNDO again — back to state 1 config.restore(caretaker.undo()); System.out.println(config.height); // 5

The most important insight — why not just copy the object directly?
Store object directly (bad)
  • Caretaker gets full reference to Configuration
  • Can read and modify internal fields
  • Breaks encapsulation
  • Internals exposed to outside world
Memento pattern (good)
  • Originator decides WHAT to save
  • Originator decides HOW to restore
  • Caretaker only holds Memento — can't read it
  • Internal state never exposed outside
The encapsulation rule:
Originator = only one who can read/write its own state
Memento = opaque to Caretaker (black box)
Caretaker = only stores + returns — never inspects

This is why Memento fields are often package-private in Java — Caretaker is in a different package and can't see them.
Memento and Command both support undo — but they store completely different things.
Memento — stores STATE
Takes a snapshot of the object's state. Restoring reverses to that snapshot.
// What's stored: Memento { height=5, width=10 } // Undo = restore snapshot config.height = 5 config.width = 10
State-based undo
Command — stores ACTION
Records what action was taken. Undo = reverse that action.
// What's stored: TurnOnACCommand { execute(), undo() } // Undo = call reverse action ac.turnOff()
Action-based undo
