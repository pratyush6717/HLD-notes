One line: Move operations OUT of the class into separate visitor classes — object stays clean, operations grow freely.

Mental model: Object = Data. Visitor = Operations. Room = data, PricingVisitor = what to do with it.
Without Visitor — class explosion
  • Room.getPrice() + maintain() + reserve() + clean() + tax()...
  • Add new feature → modify Room → retest everything
  • 50 methods in one class = nightmare
  • Violates OCP + SRP
With Visitor — clean separation
  • Room only has accept(visitor)
  • Add CleaningVisitor → Room untouched
  • Add TaxVisitor → Room untouched
  • OCP + SRP satisfied
Before vs after
// BAD — class bloat
class Room {
  void getPrice()   { ... } // op 1
  void maintain()   { ... } // op 2
  void reserve()    { ... } // op 3
  void clean()      { ... } // op 4 (NEW)
  void calcTax()    { ... } // op 5 (NEW)
  // Room grows every time a new operation added
}
// GOOD — Visitor Pattern
class Room {
  void accept(RoomVisitor v) { v.visit(this); }
  // Room never changes. Operations = visitor classes.
}

class PricingVisitor   { visit(SingleRoom), visit(DoubleRoom) }
class MaintenanceVisitor { ... }
class CleaningVisitor   { ... } // new op — Room untouched
// 1. Element interface
interface Room {
  void accept(RoomVisitor visitor);
}

// 2. Concrete Elements
class SingleRoom implements Room {
  public void accept(RoomVisitor visitor) {
    visitor.visit(this); // 🔥 the magic — pass self
  }
}
class DoubleRoom implements Room {
  public void accept(RoomVisitor visitor) {
    visitor.visit(this);
  }
}
// 3. Visitor interface — all room types listed
interface RoomVisitor {
  void visit(SingleRoom room);
  void visit(DoubleRoom room);
}

// 4. Concrete Visitors — operations separated
class PricingVisitor implements RoomVisitor {
  public void visit(SingleRoom r) { System.out.println("Single = ₹1000"); }
  public void visit(DoubleRoom r) { System.out.println("Double = ₹2000"); }
}

class MaintenanceVisitor implements RoomVisitor {
  public void visit(SingleRoom r) { System.out.println("Maintain single"); }
  public void visit(DoubleRoom r) { System.out.println("Maintain double"); }
}

// 5. Client — same room, different visitors
Room room = new SingleRoom();
room.accept(new PricingVisitor());      // Single = ₹1000
room.accept(new MaintenanceVisitor());  // Maintain single
// Add new operation? Just add new Visitor class. Room unchanged.
Double dispatch = the method called is determined by TWO types — the visitor type AND the element type. This is what makes Visitor powerful and unusual.
Single dispatch (normal Java)
Method determined by ONE type — the object you call on.
room.someOp(visitor);
// Only room's type decides the method
// Visitor's type = ignored
Can't vary behaviour based on both room AND visitor type simultaneously.
Double dispatch (Visitor)
Step 1: room.accept(visitor) → calls based on room type. Step 2: visitor.visit(this) → calls based on visitor type.
room.accept(visitor);   // dispatch 1: room type
// inside accept():
visitor.visit(this);   // dispatch 2: visitor type
Method = f(room type) × f(visitor type)
Dispatch matrix
VisitorRoom = SingleRoomRoom = DoubleRoom
PricingVisitorvisit(SingleRoom) → ₹1000visit(DoubleRoom) → ₹2000
MaintenanceVisitorvisit(SingleRoom) → maintain singlevisit(DoubleRoom) → maintain double
CleaningVisitorvisit(SingleRoom) → clean singlevisit(DoubleRoom) → clean double
The MAGIC line: Inside accept(), calling visitor.visit(this) — this is the concrete room type. Java overload resolution picks the right visit() method based on the concrete type. Two dispatches → two types → one correct method.
Visitor vs Strategy — most common confusion. Both separate behaviour from objects. Different purpose.
Strategy — change HOW
Swap the algorithm used by an object. One algorithm at a time. Same type of operation, different implementation.
Payment → UPI / Card / NetBanking
Same operation (pay), different method
Object behaviour changes at runtime
Visitor — add WHAT
Add entirely new operations without changing the object. Multiple different operations on same object structure.
Room → Pricing / Maintenance / Cleaning
Different operations, same objects
Object structure is stable

Definition: Visitor is a behavioural pattern that allows adding new operations to existing object structures without modifying those objects — by separating operations into visitor classes and using double dispatch.
The key trade-off — memorise this
Visitor is GOOD when
✔ Elements are STABLE (rarely add new room types)
✔ Operations keep growing (pricing, tax, maintenance...)
✔ Want OCP — add operations without touching objects
Visitor is HARD when
✗ New element types added frequently
✗ Add TripleRoom → update ALL visitors
✗ Violates OCP on the element axis
