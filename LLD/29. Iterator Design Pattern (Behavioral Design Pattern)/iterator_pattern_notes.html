One line: Iterator provides a way to traverse a collection without exposing its internal structure.

Memory trick: Collection = Data. Iterator = Cursor moving over data.
You use this every day: Iterator<Integer> it = list.iterator(); — Java's entire collection framework is built on this pattern. ArrayList, LinkedList, HashSet, Stack — all expose the same Iterator interface.
// BAD — client must know internals if (collection is ArrayList) { for (int i = 0; i < list.size(); i++) ... } if (collection is LinkedList) { ... } if (collection is HashSet) { ... } // Different traversal for each type
// GOOD — Iterator Pattern Iterator<Book> it = collection.iterator(); while (it.hasNext()) { Book b = it.next(); // Works for ANY collection type }
Two key rules
Who knows traversal logic?
✔ Iterator knows how to traverse
✗ Client does NOT know structure
✗ Client doesn't know if it's array, hash, or list
Who stores data?
✔ Aggregate (Collection) stores data
✗ Iterator just traverses — doesn't own data
Iterator is a cursor, not a container
// 1. Iterator interface interface Iterator<T> { boolean hasNext(); T next(); } // 2. Aggregate (Collection) interface interface Aggregate<T> { Iterator<T> createIterator(); } // 3. Concrete Aggregate class Library implements Aggregate<Book> { List<Book> books = new ArrayList<>(); void addBook(Book b) { books.add(b); } public Iterator<Book> createIterator() { return new BookIterator(books); // gives out iterator } }
// 4. Concrete Iterator class BookIterator implements Iterator<Book> { List<Book> books; int index = 0; BookIterator(List<Book> books) { this.books = books; } public boolean hasNext() { return index < books.size(); } public Book next() { return books.get(index++); } } // 5. Client — no knowledge of Library internals Library lib = new Library(); lib.addBook(new Book("Clean Code")); lib.addBook(new Book("System Design")); Iterator<Book> it = lib.createIterator(); while (it.hasNext()) { System.out.println(it.next().title); }
Who creates the iterator? The Aggregate (Library) — not the client. This is important: collection knows its own structure best, so it creates the right iterator for itself.
Same collection → multiple iterators possible. Each iterator walks the data differently. This is why Iterator is more powerful than just using for loops.
Forward iterator
Normal — index starts at 0, increments.
int index = 0; hasNext() → index < size next() → books.get(index++)
Reverse iterator
Index starts at end, decrements. Same interface!
int index = size - 1; hasNext() → index >= 0 next() → books.get(index--)
Filtered iterator
Skip items that don't match. Still same interface!
// Only return books with rating > 4 hasNext() → find next match next() → return matched item
// All three used identically by client Iterator<Book> forward = lib.createForwardIterator(); Iterator<Book> reverse = lib.createReverseIterator(); Iterator<Book> filtered = lib.createFilteredIterator(4.0); // Same while loop works for all three! while (forward.hasNext()) { System.out.println(forward.next()); } while (reverse.hasNext()) { System.out.println(reverse.next()); } while (filtered.hasNext()) { System.out.println(filtered.next()); }

ForEach internally uses Iterator. for (Book b : books) compiles to it = books.iterator(); while(it.hasNext()) { b = it.next(); }. So when you write for-each, you're using Iterator Pattern without knowing it.
When to prefer explicit Iterator over for-each: Use explicit iterator when you need it.remove() during traversal. for-each doesn't expose remove() safely.
