One line: Encapsulate a request as an object — so you can parameterize, queue, log, and undo it.

Memory trick: "Button doesn't know what it does" → Button → Command → Receiver
Without Command — 3 problems
  • Client knows AC internals (tight coupling)
  • Add Fan + Bulb → client becomes huge
  • No history → can't undo anything
With Command — fixed
  • Client only knows Remote (Invoker)
  • Each device gets its own Command class
  • Stack of commands → undo/redo trivial
  • Commands can be logged, queued, replayed

Invoker doesn't know about AC. Command doesn't know about Remote. Each layer is decoupled.
5 components — always the same structure regardless of use case.
1
Receiver — does the actual work
AirConditioner, TV, Bulb. Contains real business logic. Completely unaware of Command/Invoker.
2
Command interface
execute() + optional undo(). Contract all concrete commands implement.
3
Concrete Command — HAS-A receiver
TurnOnACCommand, TurnOffACCommand. Holds receiver reference. execute() delegates to receiver. undo() reverses it.
4
Invoker — HAS-A command
RemoteControl. Holds command reference + command history (stack). Calls execute(). Knows nothing about receivers.
5
Client — wires everything
Creates receiver + command + invoker. Sets command on invoker. Calls invoker methods. Never calls receiver directly.
The most powerful benefit of Command. Because each action is an object, you can store them in a stack and reverse them — undo/redo for free.
Undo flow
Press ON → execute() → push(TurnOnCmd)
Press OFF → execute() → push(TurnOffCmd)
Press UNDO → pop() → TurnOffCmd.undo() → AC ON
Press UNDO → pop() → TurnOnCmd.undo() → AC OFF
// In Invoker Stack<Command> history = new Stack<>(); Stack<Command> redoStack = new Stack<>(); void pressButton() { command.execute(); history.push(command); redoStack.clear(); // new action clears redo } void undo() { if (!history.isEmpty()) { Command last = history.pop(); last.undo(); redoStack.push(last); // enable redo } } void redo() { if (!redoStack.isEmpty()) { Command cmd = redoStack.pop(); cmd.execute(); history.push(cmd); } }
Interview trap: Where is undo stored? → In Invoker (history stack). Who knows the undo logic? → Command class. Can we do redo? → Yes, two stacks (undo stack + redo stack).
// 1. Receiver class AirConditioner { public void turnOn() { System.out.println("AC ON"); } public void turnOff() { System.out.println("AC OFF"); } } // 2. Command interface interface Command { void execute(); void undo(); } // 3. Concrete Commands class TurnOnACCommand implements Command { AirConditioner ac; TurnOnACCommand(AirConditioner ac) { this.ac = ac; } public void execute() { ac.turnOn(); } public void undo() { ac.turnOff(); } } class TurnOffACCommand implements Command { AirConditioner ac; TurnOffACCommand(AirConditioner ac) { this.ac = ac; } public void execute() { ac.turnOff(); } public void undo() { ac.turnOn(); } }
// 4. Invoker class RemoteControl { Command command; Stack<Command> history = new Stack<>(); void setCommand(Command c) { this.command = c; } void pressButton() { command.execute(); history.push(command); } void pressUndo() { if (!history.isEmpty()) { history.pop().undo(); } } } // 5. Client AirConditioner ac = new AirConditioner(); RemoteControl remote = new RemoteControl(); remote.setCommand(new TurnOnACCommand(ac)); remote.pressButton(); // AC ON remote.setCommand(new TurnOffACCommand(ac)); remote.pressButton(); // AC OFF remote.pressUndo(); // AC ON (undo TurnOff) remote.pressUndo(); // AC OFF (undo TurnOn)
Definition: Command pattern encapsulates a request as an object, allowing parameterization, queuing, logging, and undo/redo.

Category: Behavioural.
