The deepest aha moment: LSP violation ka problem child me nahi hota — problem parent ke galat design me hota hai.
The mistake everyone makes: "Bicycle doesn't have an engine — it's the child's fault." Wrong. Why does Vehicle have hasEngine() at all? Not all vehicles have engines. Parent assumed too much.
Bad hierarchy
Vehicle← hasEngine() — WRONG
├──Motorcycle ✔
├──Car ✔
└──Bicycle ✗→ null.toString() → NPE
Vehicle v = new Bicycle();
v.hasEngine() → crash at runtime
Why this is wrong
Parent contract says: "I have an engine"
Bicycle says: "I don't"
Contract broken → LSP violation

Root cause: hasEngine() doesn't belong in Vehicle. It belongs only in vehicles that actually have engines.
Root cause formula
Wrong abstraction in parent = LSP violation
2-step fix: Step 1 — identify what's truly common to ALL vehicles. Step 2 — move specific behaviour to a specialised subclass. Never put capabilities in the parent that not all children support.
Before — LSP violation
Vehicle
└── hasEngine() ← not universal
├──Motorcycle ✔
├──Car ✔
└──Bicycle ✗ crashes
After — LSP satisfied
Vehicle
└── getNumberOfWheels() ← universal
├──Bicycle ✔only wheels
└──EngineVehicle
└── hasEngine() ← specific
├──Car ✔
└──Motorcycle ✔
Golden rule: Parent me sirf wahi methods rakho jo sab children support karte hain.

Common → Vehicle (getNumberOfWheels)
Specific → EngineVehicle (hasEngine)

Result: Runtime error → compile time error. Bicycle can never be assigned to EngineVehicle — compiler stops it.
// BAD — runtime NPE guaranteed class Vehicle { String hasEngine() { return null; } // Bicycle returns null } class Bicycle extends Vehicle { String hasEngine() { return null; } // meaningless } Vehicle v = new Bicycle(); v.hasEngine().toString(); // NullPointerException at runtime 💥
// GOOD — hierarchy matches reality class Vehicle { int getNumberOfWheels() { ... } // ALL vehicles have this } class Bicycle extends Vehicle { // Fine — only uses getNumberOfWheels } class EngineVehicle extends Vehicle { boolean hasEngine() { return true; } // SPECIFIC to engine vehicles } class Car extends EngineVehicle { ... } // ✔ class Motorcycle extends EngineVehicle { ... } // ✔
Safe substitutions
Vehicle v = new Bicycle(); v.getNumberOfWheels(); // ✔ EngineVehicle e = new Car(); e.hasEngine(); // ✔
Compile-time protection
EngineVehicle e = new Bicycle(); // ❌ Compile error! // Bicycle is not an EngineVehicle // Runtime error became compile error 🔥
The key shift: Before fix — bad design meant problems only showed up at runtime (NPE, null, unexpected behaviour). After fix — compiler catches invalid assignments before code even runs. Safer system with zero extra effort at runtime.
Interview killer line: LSP violation usually happens due to incorrect abstraction in the parent class — not because the child class is wrong.

One line summary: Design your hierarchy so all subclasses can safely replace the parent without breaking behaviour.
