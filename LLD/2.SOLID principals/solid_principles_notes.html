SOLID = clean, maintainable, scalable code likhne ke rules. Design patterns are the tools — SOLID is the foundation they're built on.
Without SOLID
Spaghetti code. Hard to change. One edit breaks 5 things. Bugs everywhere. New feature = nightmare.
With SOLID
Clean code. Easy to change. Reusable. Testable. New feature = add a class, nothing breaks.

Restaurant analogy — all 5 in one place
S — Chef cooks, Waiter serves, Cleaner cleans. One job each.
O — New dish added → chef learns it. Old recipes untouched.
L — Veg burger behaves exactly like any burger — no surprises.
I — Waiter doesn't need to know how to cook. Separate interfaces.
D — Customer orders "Food" (interface), not a specific dish hardcoded.
S
Single Responsibility Principle
1 class = 1 responsibility
// BAD — class doing 3 jobs class Invoice { calculate() {} // business logic saveToDB() {} // persistence print() {} // presentation }
// GOOD — each class has 1 job class Invoice { calculate() {} } class InvoiceDAO { saveToDB() {} } class InvoicePrinter { print() {} }
Aha moment: Change notification logic → only NotificationService changes. DB logic change → only DAO changes. No ripple effects.

Common confusion: 1 class = 1 responsibility, NOT 1 class = 1 method. A class can have many methods as long as they all serve the same single purpose.
O
Open/Closed Principle
Open for extension · Closed for modification
// BAD — add DB2? Must modify existing code if (type == "DB") saveDB(); if (type == "FILE") saveFile();
// GOOD — add new behaviour via extension interface InvoiceDAO { save(); } class DBInvoiceDAO implements InvoiceDAO { save(){...} } class FileInvoiceDAO implements InvoiceDAO { save(){...} } // Add S3InvoiceDAO? Zero changes to existing code.
Aha moment: Old code never touched. New feature = new class. Existing system untouched = existing tests still pass.
L
Liskov Substitution Principle
Child class = perfect replacement for parent
// BAD — Bicycle breaks the contract class Vehicle { turnOnEngine() {} } class Bike extends Vehicle { turnOnEngine() { /* works */ } } class Bicycle extends Vehicle { turnOnEngine() { throw Exception; } } // Replace Vehicle with Bicycle → code explodes
// GOOD — separate hierarchies class EngineVehicle { turnOnEngine() {} } class Bike extends EngineVehicle { ... } class Vehicle { accelerate() {} } class Bicycle extends Vehicle { ... } // No engine — no problem
Aha moment: Anywhere you use a parent class, you should be able to swap it with a child class and nothing breaks. Child class should only add behaviour, never contradict parent's contract.
I
Interface Segregation Principle
Many small interfaces > one fat interface
// BAD — Waiter forced to implement cook() interface RestaurantWorker { serve(); cook(); // Waiter doesn't cook! clean(); // Waiter doesn't clean! } class Waiter implements RestaurantWorker { cook() { /* useless */ } }
// GOOD — each interface is focused interface Servable { serve(); } interface Cookable { cook(); } interface Cleanable { clean(); } class Waiter implements Servable { serve() {...} } class Chef implements Cookable { cook() {...} }
Aha moment: No class should be forced to implement methods it doesn't use. Small, focused interfaces = better composition and no "empty" or "throws UnsupportedOperation" methods.
D
Dependency Inversion Principle
Depend on abstraction, not concrete class
// BAD — MacBook locked to WiredKeyboard class MacBook { WiredKeyboard keyboard = new WiredKeyboard(); // hardcoded // Can NEVER use wireless keyboard without changing MacBook }
// GOOD — MacBook depends on interface interface Keyboard { type(); } class WirelessKeyboard implements Keyboard { type() {...} } class WiredKeyboard implements Keyboard { type() {...} } class MacBook { Keyboard keyboard; // inject any keyboard MacBook(Keyboard k) { this.keyboard = k; } }
Aha moment: Loose coupling = flexibility. Change keyboard type without touching MacBook. This is the foundation of Dependency Injection in Spring — you inject interfaces, not concrete implementations.

Connection to real code: @Autowired in Spring follows this — Spring injects the right implementation at runtime.
Interview killer line: SOLID principles ensure low coupling and high cohesion — making the system easy to change without breaking existing behaviour.
