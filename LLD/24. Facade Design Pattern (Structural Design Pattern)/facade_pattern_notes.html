Important: Facade is optional
Client CAN still access subsystems directly if needed. Facade just adds a convenient layer — it doesn't block access.
Facade using Facade: CheckoutFacade → OrderFacade + PaymentFacade. Very common in real systems.
Order system — the real power
// BAD — client knows everything productService.getProduct(id); paymentService.pay(amount); invoiceService.generate(order); notificationService.send(email); // Client coupled to 4 services + their sequence
Payment service changes? Client updates. Invoice logic changes? Client updates. Any service changes → client breaks.
// GOOD — Facade hides complexity + flow class OrderFacade { void createOrder(int id, double amount) { productService.getProduct(id); // step 1 paymentService.pay(amount); // step 2 invoiceService.generate(order); // step 3 notificationService.send(email); // step 4 } } // Client orderFacade.createOrder(1, 999.0); // that's it
Facade hides BOTH: complexity and sequence. Client is decoupled from all 4 services.
DAO facade example
// Without facade — client calls 5 DAO methods directly dao.insert(); dao.updateName(); dao.updateAddress(); dao.getById(); dao.getByEmail(); // With facade — clean 2-method interface class EmployeeFacade { void createEmployee(Employee e) { dao.insert(e); dao.updateAddress(e.getAddress()); // orchestrates } Employee getEmployee(int id) { return dao.getById(id); } } // Client facade.createEmployee(emp); // ✔ clean facade.getEmployee(101); // ✔ clean
Facade hides two things: (1) How many services/classes are involved. (2) In what sequence they must be called. Client just calls one method.

One-liner shortcuts — memorise:
Facade → simplify (hide complexity, hide workflow)
Proxy → control (same interface, gatekeeper)
Adapter → translate (incompatible interfaces bridged)
Definition: Facade pattern provides a unified, simplified interface to a set of complex subsystems — reducing coupling and hiding implementation complexity from the client.
ConceptOne-liner
Facade definitionSimple interface to complex system — hides both complexity and workflow sequence
Key benefitClient decoupled from all subsystems — changes to any service don't break client
Facade is optionalClient can still access subsystems directly — Facade just adds convenience
Facade of FacadesCheckoutFacade uses OrderFacade + PaymentFacade — composing at higher level
vs ProxyFacade = simplify (new interface, many classes). Proxy = control (same interface, one class).
vs AdapterFacade = hide complexity (same system). Adapter = fix incompatibility (different systems).
Real examplesAPI Gateway (Facade over microservices), Service layer (Facade over DAO layer)
Full structural patterns — final recap
Facade → simple interface to complex system (simplify)
Proxy → controlled access to same interface (control)
Adapter → bridge between incompatible interfaces (translate)
Decorator → add behaviour by wrapping (enhance)
Composite → tree of uniform objects (recurse)
Builder → step-by-step object construction (creational)
