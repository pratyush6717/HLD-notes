One line: Reduce memory by sharing common (intrinsic) data across millions of objects — pass the varying (extrinsic) data in each call instead of storing it.

Memory trick: Don't store duplicate data → share it.
The problem — memory explosion
Without Flyweight
  • Game: 10 lakh robot objects
  • Each stores type + sprite (30KB) + x + y
  • 10,00,000 × 30KB = 30 GB RAM 💣
  • But 5 lakh robots are SAME type → same sprite
  • Storing same sprite 5 lakh times = pure waste
With Flyweight
  • Create only 2 objects (humanoid + robotic)
  • Sprite stored ONCE per type in cache
  • x, y position passed in each call
  • 2 × 30KB = 60KB RAM ✔
  • 10 lakh → 2 objects created

Pattern recognition trigger: Millions of similar objects + memory constraint + repeated heavy data (sprites, fonts, icons) → immediately think Flyweight.
The most important concept. Split every object's data into what's shared (intrinsic) and what's unique per instance (extrinsic). Only store intrinsic inside the object.
Intrinsic — stored inside, shared
Same across many objects. Cached once. Never changes after creation.
🎮 Robot: type, sprite/image
📝 Text editor: font, style, character shape
🌳 Game trees: texture, model, color
🗺️ Map: icon type, icon image
// Stored IN the object String type; Sprite sprite;
Extrinsic — passed in, unique per call
Changes per object/call. NOT stored inside. Passed as method parameter.
🎮 Robot: x position, y position
📝 Text editor: row, column
🌳 Game trees: x, y, z coordinate
🗺️ Map: lat, lng coordinate
// Passed to method void display(int x, int y);
Golden rule:
Intrinsic → shared → store inside flyweight object
Extrinsic → unique per instance → pass from outside

The flyweight object itself is immutable (intrinsic never changes). The caller provides the context (extrinsic) each time.
// Flyweight object — stores only intrinsic data class Robot { String type; // intrinsic — shared Sprite sprite; // intrinsic — shared (heavy, 30KB) Robot(String type, Sprite sprite) { this.type = type; this.sprite = sprite; } void display(int x, int y) { // extrinsic passed in System.out.println(type + " at (" + x + "," + y + ")"); // render sprite at position } }
// Flyweight Factory — cache + reuse class RobotFactory { static Map<String, Robot> cache = new HashMap<>(); static Robot getRobot(String type) { if (!cache.containsKey(type)) { // create ONCE per type — not per instance Sprite sprite = new Sprite(type); // heavy 30KB load cache.put(type, new Robot(type, sprite)); System.out.println("Created: " + type); } return cache.get(type); // ✔ reuse existing } }
// Client — 10 lakh calls, only 2 objects created for (RobotData data : gameData) { // 10 lakh items Robot r = RobotFactory.getRobot(data.type); // cache hit after first r.display(data.x, data.y); // extrinsic passed each time } // Console shows: "Created: humanoid", "Created: robotic" — only 2 times!
Text editor version: Letter l = LetterFactory.get('T'); l.display(row, col); — 1000 'T' characters in document → 1 Letter object, reused 1000 times with different row/col.

Flyweight vs Singleton
Singleton = ONE object, globally. Flyweight = ONE object per TYPE (2 robot types = 2 flyweights). Singleton has one; Flyweight has as many as there are distinct intrinsic states.
Flyweight vs Object Pool
Object Pool = borrow/return full mutable objects (DB connections). Flyweight = shared immutable objects never "checked out" — just reused simultaneously by all callers.
Definition: Flyweight is a structural pattern that reduces memory by sharing intrinsic (common) data among many objects — externalising the varying (extrinsic) data to callers.

Quick example: Game with 10 lakh robots → share sprite per type → pass position. 10 lakh objects → 2 objects. Memory: 30GB → 60KB.
