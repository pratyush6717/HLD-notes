One line: Strategy Pattern defines multiple algorithms, encapsulates each in a dedicated class, and lets you swap them at runtime without touching the context class.
Category

Starting point: Vehicle has a default drive(). Child classes override it. Seems fine — until you have 10–15 subclasses and notice the same code appearing in multiple places.
Bad design — drive logic scattered everywhere
Vehicle→ normalDrive()
├──SportsVehicle→ sportsDrive() ← duplicate
├──OffRoadVehicle→ sportsDrive() ← same duplicate
├──PassengerVehicle→ normalDrive() (reuse parent)
└──GoodsVehicle→ normalDrive() (reuse parent)
Problem 1 — Code duplication
  • sportsDrive() written in SportsVehicle
  • Same sportsDrive() in OffRoadVehicle
  • 10 subclasses = 10 scattered copies
  • Bug fix → update in 10 places
Problem 2 — Tight coupling
  • Drive logic baked into each subclass
  • Change OffRoad to sports → modify the class
  • Can't change behaviour at runtime
  • New drive type = new subclass or modify existing
Key insight: Extract the algorithm (drive behaviour) into its own hierarchy. Vehicle doesn't own drive logic anymore — it holds a reference to a DriveStrategy and delegates to it.

Vehicle
├── DriveStrategy strategy ← injected

├──SportsVehiclepasses SportsDrive
├──OffRoadVehiclepasses SportsDrive
└──GoodsVehiclepasses NormalDrive
Aha moment:
Vehicles scale independently — add new vehicle without touching strategies.
Strategies scale independently — add new algorithm without touching vehicles.

Same sportsDrive logic in ONE place. No duplication. Change it once → all vehicles using it get the fix.
// Step 1 — Strategy interface interface DriveStrategy { void drive(); } // Step 2 — Concrete strategies class NormalDriveStrategy implements DriveStrategy { public void drive() { System.out.println("Normal drive"); } } class SportsDriveStrategy implements DriveStrategy { public void drive() { System.out.println("Sports drive"); } }
// Step 3 — Context (Vehicle) holds strategy class Vehicle { DriveStrategy driveStrategy; Vehicle(DriveStrategy driveStrategy) { this.driveStrategy = driveStrategy; // injected } void drive() { driveStrategy.drive(); // delegates — doesn't care which } } // Step 4 — Child classes inject the right strategy class SportsVehicle extends Vehicle { SportsVehicle() { super(new SportsDriveStrategy()); // ← passes strategy } } class GoodsVehicle extends Vehicle { GoodsVehicle() { super(new NormalDriveStrategy()); // ← different strategy } }
// Step 5 — Usage (and runtime swap!) SportsVehicle sv = new SportsVehicle(); sv.drive(); // → "Sports drive" // Runtime swap — change strategy after creation sv.driveStrategy = new NormalDriveStrategy(); sv.drive(); // → "Normal drive" — same object, new behaviour
Runtime swap = the magic. No new subclass. No class modification. Just inject a different strategy object and behaviour changes instantly.
Definition: Strategy is a behavioural pattern that defines a family of algorithms, encapsulates each in a class, and makes them interchangeable — allowing behaviour to change at runtime without modifying the context.
ConceptOne-liner
Strategy (interface)Algorithm contract — one abstract method all strategies implement
Concrete strategyOne specific implementation of the algorithm
ContextUses strategy via interface — doesn't know/care which concrete one
Solves code duplicationAlgorithm in one class — not scattered across 10 subclasses
Solves tight couplingInject strategy via constructor — swap at runtime, no class change
Follows OCPNew algorithm = new strategy class. Existing code untouched.
Connection to DIPContext depends on DriveStrategy interface, not NormalDriveStrategy concrete
When to use Strategy pattern
Use when
  • Multiple classes differ only in behaviour
  • Same algorithm used in different classes (duplication)
  • Need to switch algorithms at runtime
  • Want to isolate algorithm from the code that uses it
Real-world examples
Payment: CreditCard/UPI/Wallet strategies
Sorting: BubbleSort/MergeSort strategies
Compression: ZIP/RAR/LZ4 strategies
Navigation: Walking/Driving/Cycling routes
