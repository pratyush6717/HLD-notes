3 components
Observable (Publisher)
Holds state. Maintains list of observers. Notifies all when state changes.
addObserver()

Observer (Subscriber)
Subscribes to the observable. Receives updates. Runs its own logic on change.
update()
The key method
notifyObservers() calls update() on every subscribed observer. The entire pattern in one line.

Key property — loose coupling
WeatherStation doesn't know what TVDisplay or MobileApp do with the data. It just calls update(). Observers can be added/removed without touching the Observable.

Most important distinction. Push: publisher decides what data to send. Pull: observer decides what data to fetch.
Push model — eager
Publisher sends data directly in the update call.
// Publisher pushes data notifyObservers(temp, humidity, pressure); // Observer receives it void update(float temp, float humidity, float pressure) { display(temp); // uses what it needs }
✔ Simple — one call, data delivered
✔ Fast — no extra round trip
✗ Observer may receive data it doesn't need
Pull model — lazy
Publisher just signals a change. Observer fetches what it needs.
// Publisher signals only notifyObservers(); // no data // Observer pulls what it needs void update() { float temp = weatherStation.getTemp(); // fetch only this }
✔ Flexible — observer fetches only what it needs
✗ Extra method calls
✗ Observer needs reference to observable

// Observer interface interface Observer { void update(float temp, float humidity); } // Observable interface interface Observable { void addObserver(Observer o); void removeObserver(Observer o); void notifyObservers(); }
// Concrete Observable class WeatherStation implements Observable { List<Observer> observers = new ArrayList<>(); float temp, humidity; void addObserver(Observer o) { observers.add(o); } void removeObserver(Observer o) { observers.remove(o); } void notifyObservers() { for (Observer o : observers) { o.update(temp, humidity); // calls each observer } } void setWeather(float temp, float humidity) { this.temp = temp; this.humidity = humidity; notifyObservers(); // triggers automatically } }
// Concrete Observers class TVDisplay implements Observer { public void update(float temp, float humidity) { System.out.println("TV: Temp=" + temp); } } class MobileApp implements Observer { public void update(float temp, float humidity) { System.out.println("App notification: " + temp + "°C"); } } // Usage WeatherStation ws = new WeatherStation(); ws.addObserver(new TVDisplay()); ws.addObserver(new MobileApp()); ws.setWeather(38.0f, 75.0f); // → both observers notified
removeObserver() is critical. Always unsubscribe when observer is no longer needed. Forgotten observers = memory leak — observable holds reference forever.
Killer lines:
Observer pattern provides loose coupling between publisher and subscribers — neither needs to know implementation details of the other.

It is the foundation of event-driven and reactive systems — Kafka producers/consumers, Spring events, and reactive streams all follow this pattern.
Common mistakes
Tight coupling
  • Observable calls methods directly on concrete observers
  • Fix: always use Observer interface
No unsubscribe
  • Observers never removed
  • Memory leak — old observers live forever
  • Fix: always call removeObserver()
Too many observers
  • 1000 observers → notifyAll() is slow
  • Fix: async notification / Kafka topics
