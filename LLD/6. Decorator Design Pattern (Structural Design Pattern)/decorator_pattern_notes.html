One line: Decorator adds behaviour to an object dynamically by wrapping it — without changing its original class.

Aha: Object ko modify nahi karte, usse wrap karke powerful banate hain.
The pizza intuition
Mushroom Decorator — adds ₹40
Cheese Decorator — adds ₹20
Farmhouse (Base) — ₹200
p.getCost() → Mushroom → Cheese → Farmhouse → 200+20+40 = ₹260
The most important concept
IS-A Pizza
Decorator implements Pizza interface. It IS a Pizza. Can be used anywhere a Pizza is expected.
HAS-A Pizza
Decorator holds a reference to the Pizza it wraps. It delegates to the inner Pizza and adds on top.
Inheritance + Composition combo: IS-A lets it be substituted. HAS-A lets it delegate and wrap. This is what makes Decorator powerful.
The problem without Decorator: Each pizza + topping combination = a new class. 3 toppings on 2 base pizzas = 8+ classes. 5 toppings = dozens of classes.

With Decorator
2 base classes (Margherita, Farmhouse)
3 topping decorators (Cheese, Mushroom, Paneer)
Total: 5 classes → infinite combinations

✔ Add new topping → 1 new class
✔ Stack any way at runtime
✔ Zero existing code changes
Shortcut: Layering dikhe → Decorator socho. Multiple combinations dikhe → Decorator socho. Class explosion ka dar ho → Decorator socho.

Mushroom.getCost()

Execution: inside → outside (deepest layer first)
Step-by-step wrapping
Pizza p = new Farmhouse(); // Step 1: base p = new Cheese(p); // Step 2: wrap with cheese p = new Mushroom(p); // Step 3: wrap with mushroom p.getCost(); // Mushroom(Cheese(Farmhouse)) → 260 p.getDesc(); // "Farmhouse, Cheese, Mushroom"
// Base interface interface Pizza { String getDesc(); int getCost(); } // Concrete bases class Margherita implements Pizza { public String getDesc() { return "Margherita"; } public int getCost() { return 150; } } class Farmhouse implements Pizza { public String getDesc() { return "Farmhouse"; } public int getCost() { return 200; } }
// Abstract Decorator — IS-A Pizza + HAS-A Pizza abstract class ToppingDecorator implements Pizza { protected Pizza pizza; // HAS-A ToppingDecorator(Pizza pizza) { this.pizza = pizza; } } // Concrete Decorators class Cheese extends ToppingDecorator { Cheese(Pizza pizza) { super(pizza); } public String getDesc() { return pizza.getDesc() + ", Cheese"; } public int getCost() { return pizza.getCost() + 20; } // delegates + adds } class Mushroom extends ToppingDecorator { Mushroom(Pizza pizza) { super(pizza); } public String getDesc() { return pizza.getDesc() + ", Mushroom"; } public int getCost() { return pizza.getCost() + 40; } }
// Usage — stacking at runtime Pizza p = new Farmhouse(); // 200 p = new Cheese(p); // 220 p = new Mushroom(p); // 260 System.out.println(p.getDesc()); // Farmhouse, Cheese, Mushroom System.out.println(p.getCost()); // 260
Java real example: new BufferedReader(new FileReader("file.txt")) — BufferedReader wraps FileReader. That IS the Decorator pattern in Java's standard library.
Killer lines:
Decorator adds responsibilities to objects dynamically without modifying existing code.
It avoids class explosion by using composition over inheritance.
