One line: Template Method defines a fixed workflow in the parent — subclasses customize individual steps.

Memory trick: Recipe. Steps are fixed (Step 1 → Step 2 → Step 3). Cooking style differs per chef.
Core idea: Same process. Same steps. Same order. But different implementations per step.

Why it matters: The order (validate → debit → fees → credit) must NEVER change — wrong order = money lost. Template Method enforces this with final keyword. Child classes customize the steps but can never reorder them.
// Abstract class — owns the template abstract class PaymentFlow { // TEMPLATE METHOD — final, child can NEVER override public final void sendMoney() { validate(); // step 1 debit(); // step 2 calculateFees(); // step 3 credit(); // step 4 // Order is locked. Always. No exceptions. } // Steps = abstract → child MUST implement abstract void validate(); abstract void debit(); abstract void calculateFees(); abstract void credit(); }
// Concrete class 1 — Pay to Friend class PayToFriend extends PaymentFlow { void validate() { System.out.println("Basic validation"); } void debit() { System.out.println("Debit full amount"); } void calculateFees() { System.out.println("No fees (0%)"); } void credit() { System.out.println("Credit full amount"); } } // Concrete class 2 — Pay to Merchant class PayToMerchant extends PaymentFlow { void validate() { System.out.println("Strict KYC validation"); } void debit() { System.out.println("Debit amount"); } void calculateFees() { System.out.println("Apply 2% fee"); } void credit() { System.out.println("Credit after fee deduction"); } }
// Client — same interface, different behaviour PaymentFlow p1 = new PayToFriend(); p1.sendMoney(); // Basic validation → Debit full amount → No fees → Credit full amount PaymentFlow p2 = new PayToMerchant(); p2.sendMoney(); // Strict KYC → Debit amount → 2% fee → Credit after deduction // Both follow: validate → debit → fees → credit. Always.
3 design decisions that make Template Method work — all matter.
1
Template method = final
Child classes CANNOT override sendMoney(). This guarantees the order. Without final — child could override the whole method and bypass the flow. That breaks the pattern.
2
Steps = abstract
Each step is abstract — child MUST provide implementation. No default = no lazy subclass. If a step has a sensible default, make it non-abstract (hook method).
3
Control stays in parent
Parent = algorithm skeleton (what and in what order). Child = step implementation (how). Child never controls the flow — only fills in the blanks.
Real-world usage
JdbcTemplate (Spring) RestTemplate (Spring) JUnit @Before → @Test → @After Order processing (validate → pay → invoice → notify) Game loops (init → play → cleanup)
Template Method and Strategy both allow varying behaviour — but through completely different mechanisms.
Template Method — Inheritance
Parent defines the flow. Subclasses fill in steps. Flow is fixed. Decided at compile time.
abstract class PaymentFlow { final void sendMoney() { validate(); debit(); credit(); } abstract void validate(); // child fills in }
Inheritance-based. Parent controls flow.
Strategy — Composition
Client injects algorithm object. Algorithm is interchangeable. Flow NOT enforced.
class Context { Strategy strategy; // injected void execute() { strategy.run(); } // Client can change strategy anytime }
Composition-based. Client controls algorithm.

Definition: Template Method is a behavioural pattern where a base class defines the skeleton of an algorithm, and subclasses override specific steps without changing the overall flow.

Category: Behavioural. Uses inheritance.
Trigger words → Template Method
Same process, different implementations Steps must follow same order Enforce workflow structure Consistency + customisation
Common mistake
// WRONG — child overrides the whole method class PayToFriend extends PaymentFlow { @Override void sendMoney() { // ❌ defeats the pattern credit(); // skips validate + debit! } }
// CORRECT — template method is final, child overrides only steps class PayToFriend extends PaymentFlow { void validate() { ... } // ✔ only steps void debit() { ... } // ✔ // sendMoney() is final — can't override }
