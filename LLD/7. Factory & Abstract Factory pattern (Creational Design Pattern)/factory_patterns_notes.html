The core problem: Object creation scattered everywhere. Change the class name → update 50 places.
Factory solution: Centralize creation. Caller just asks for the object — factory decides how to make it.
3 factory patterns — when to use which
Simple Factory
One class creates all objects. Best for small systems.
Use when: few types, simple logic
Factory Method
One factory per type. Delegates creation to subclasses.
Use when: complex creation, SRP needed
Abstract Factory
Creates families of related objects. Consistency guaranteed.
Use when: multiple related objects, product families
Final mental model:
Simple → centralized (one factory, all types)
Factory Method → distributed (one factory per type)
Abstract Factory → grouped (one factory, full family)

Decision rule: Need ONE object → Factory. Need MANY related objects → Abstract Factory.
Core idea: One central class creates all objects. Caller passes a type string. Factory decides which concrete class to instantiate.
// BAD — creation everywhere
Shape s1 = new Circle();
Shape s2 = new Square();
// Change Circle → update 50 places
// GOOD — Simple Factory
Shape s = ShapeFactory.getShape("CIRCLE");

class ShapeFactory {
  static Shape getShape(String type) {
    if(type.equals("CIRCLE"))  return new Circle();
    if(type.equals("SQUARE"))  return new Square();
    throw new IllegalArgEx();
  }
}

Core idea: Instead of one fat factory — one factory per product type. Creation logic delegated to subclasses. SRP restored.
// Interface
interface ShapeFactory {
  Shape createShape();
}

// One factory per type
class CircleFactory implements ShapeFactory {
  public Shape createShape() { return new Circle(); }
}
class SquareFactory implements ShapeFactory {
  public Shape createShape() { return new Square(); }
}

// Usage
ShapeFactory factory = new CircleFactory();
Shape s = factory.createShape();

├── CircleFactory

Core idea: Factory of factories. Each factory creates a complete family of related objects — guaranteeing they match each other.
The problem it solves: Without Abstract Factory — you could mix Economy interior with Luxury exterior. The mismatch is valid code but wrong product. Abstract Factory makes mismatch impossible.
Structure
CarFactory (interface)
├── createInterior()
└── createExterior()

EconomyCarFactory
├── → EconomyInterior

LuxuryCarFactory
├── → LuxuryInterior
└── → LuxuryExterior 
// Abstract Factory interface
interface CarFactory {
  Interior createInterior();
  Exterior createExterior();
}

class LuxuryCarFactory implements CarFactory {
  public Interior createInterior() {
    return new LuxuryInterior();
  }
  public Exterior createExterior() {
    return new LuxuryExterior();
  }
}

// Usage — always consistent
CarFactory factory = new LuxuryCarFactory();
Interior i = factory.createInterior(); // Luxury
Exterior e = factory.createExterior(); // Luxury
// Mismatch impossible
