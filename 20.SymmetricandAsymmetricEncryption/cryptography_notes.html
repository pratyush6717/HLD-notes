The nightmare without crypto: You type password on bank website. Data travels PK → Router → ISP → Internet → Bank. Anyone with Wireshark on the network reads: username=pk123&password=mypassword. HTTPS, JWT, WhatsApp — all useless. Internet = open jungle.
Without encryption
  • Passwords travel as plain text
  • Anyone on the network can read it
  • Bank accounts, credit cards stolen
  • JWT tokens intercepted and reused
With encryption
  • Data converted to unreadable form
  • Hacker sees: X3$9#@K
  • Only intended receiver can decrypt
  • Sniffing gives nothing useful
Core concept
HELLO → Encrypt → X3$9#@K → Decrypt → HELLO
Hacker intercepts the middle part — gets X3$9#@K — useless without the key.
4 tools of cryptography
Symmetric
Same key encrypts + decrypts. Fast.
Asymmetric
Public + private key pair. Secure key exchange.
Hashing
One-way. Integrity check. Can't reverse.
Digital sig
Authentication + integrity combined.
Symmetric encryption = same key to lock and unlock. Fast. But the problem: how do you securely share the key in the first place?
How it works
Plaintext: HELLO ↓ encrypt(key=5) Ciphertext: 9JFKL@! ↓ send over network Hacker sees: 9JFKL@! (useless) ↓ receiver decrypt(key=5) Plaintext: HELLO

✔ Very fast (128-bit key)
✗ Key distribution problem — how to share the key securely?
Locked box analogy: You and your friend have the same key. You lock a box, friend opens it. But how do you give friend the key without someone intercepting it? If you mail the key, the hacker intercepts the key AND the box.
Asymmetric encryption = two keys. Public key encrypts. Only private key can decrypt. Public key can be shared with anyone. Private key stays secret forever.
Encryption flow
Client: HELLO ↓ encrypt with server's public key Ciphertext: 7G#@2L! ↓ travels over network Hacker sees: 7G#@2L! (can't decrypt) ↓ server decrypts with private key Server reads: HELLO
Algorithms
RSA ECC Diffie-Hellman
✔ Solves key distribution — public key can be given to anyone
✗ 10–100× slower than symmetric
✗ RSA key = 2048 bits vs AES = 128 bits
Public lock analogy: Your friend has a padlock (public key) and gives copies to everyone. You put a message in a box and lock it with their padlock. Only your friend's private key opens it. Even the courier (hacker) can't open it — they only have copies of the padlock, not the key.

HTTPS combines both. Asymmetric encryption is used once to securely exchange a symmetric key. Then symmetric encryption handles all actual data — fast and secure.
TLS handshake (HTTPS)
1
Client connects to server TLS handshake begins. Client sends supported cipher suites.
2
Asymmetric encryption — key exchange RSA or Diffie-Hellman used to securely establish a shared secret. Neither side ever sends the key directly over the network.
3
Symmetric encryption — all actual data AES used from this point. Fast. Every HTTP request/response encrypted with the shared AES key established in step 2.
Why both? RSA/DH = secure but slow. AES = fast but needs secure key sharing. Use RSA to securely share the AES key. Then switch to AES for everything else. Best of both worlds.
Diffie-Hellman key exchange (simplified)
Both agree on public numbers: g=3, p=7
Client picks secret: a=4 — never shared
Server picks secret: b=5 — never shared
They exchange derived public values → both compute the same shared_key = 2
Hacker saw the exchange but can't compute the key without a or b.
Digital signature = authentication + integrity. Proves the message came from the right sender AND hasn't been tampered with. Uses private key to sign, public key to verify.
Sender signs
Message: HELLO ↓ hash(HELLO) Hash: 5G78K2 ↓ encrypt(hash, private_key) Signature: @X9#2Z ↓ send both HELLO + signature
Receiver verifies
Receive: HELLO + signature ↓ hash(HELLO) Compute hash: 5G78K2 ↓ decrypt(signature, public_key) Extracted hash: 5G78K2 ↓ compare Match ✔ → message authentic + unchanged
Used in: JWT (signature ensures token wasn't tampered), HTTPS certificates (CA signs server's public key), SSH (authenticate without password), software releases (verify binary hasn't been modified).
Final mental model:

Symmetric → fast bulk encryption (AES)
Asymmetric → secure key exchange (RSA, DH)
Hashing → one-way integrity check
Digital signature → authentication + integrity

Trade-off sandbox
Only symmetric
Key distribution problem. Hacker intercepts the key during sharing. All encrypted data can now be decrypted.
Only asymmetric
10–100× slower. RSA encrypting gigabytes of HTTPS traffic = unusable. Too much CPU overhead.
No encryption at all
HTTPS impossible. JWT insecure. Password sniffing trivial. Bank fraud rampant. Internet unusable for anything sensitive.
