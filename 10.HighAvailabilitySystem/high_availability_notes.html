Single Point of Failure: Ek DB pe poora system depend hai. DB down → login fail, payment fail, order fail. System completely down. Ye interview ka sabse important term hai.
Single point of failure
  • User → Server → ❌ DB
  • Login fail, payment fail, order fail
  • Zero redundancy
  • One crash = full outage
High availability solution
  • Multiple DBs — ek crash karo, doosra le lega
  • Multiple data centers — region fail karo, doosra active
  • Replication — data dono jagah copy
  • Failover — automatic traffic shift
Key terms (interview mein zaroori)
Replication
Primary DB se data Replica DB me copy hota rehta hai. Dono me same data.
Failover

Active–Passive = 1 active + 1 backup. Primary handles everything. Replica stays in sync but idle. Failover tabhi hota hai jab primary crash kare.

Agar interviewer bole: "Design High Availability Architecture"

4 steps mein answer do — SRFA: Single point removal → Replication → Failover → Active choice.
4-step answer structure
1
Remove single point of failure — never depend on one server or DB
2
Multiple data centers — geographic redundancy (Mumbai + Pune)
3
Replication — data continuously synced across both DCs
4
Active–Passive or Active–Active — choose based on consistency requirement
Quick decision rule
If system needs…ChooseBecause
Strong consistency (banking, payments)Active–PassiveSingle source of truth — no conflict possible
Maximum throughput + availability (Netflix, social media)Active–ActiveBoth DCs active, zero idle resource, no downtime on failover
Pro tip: Mention both architectures and then say which you'd choose and why. "For a payment system I'd use Active–Passive because data conflicts in financial systems are unacceptable. For a social media feed I'd use Active–Active because brief inconsistency is acceptable and we need maximum throughput."
