The shift: Start with everything in one codebase (monolith). As the system grows, it becomes impossible to scale, deploy, or debug. The solution is splitting it into independent microservices.
Monolith
One giant application
Login, Orders, Products, Payments, Billing, Notifications — all in one codebase, one deployment, one database.
Analogy: One kitchen making pizza, burgers, desserts, and drinks — if burgers get busy, everything slows
Microservices
Independent services
Each feature is its own service — own codebase, own deployment, own database, own scaling.
Analogy: A food court — Pizza stall, Burger stall, Coffee stall each operate independently
4 phases of microservices design
1
Decomposition Break the monolith into services using business capability or DDD patterns.
2
Database design Decide: shared database or database-per-service.
3
Communication Services talk via REST APIs, events (Kafka), or message queues.
4
Integration & governance API gateway, service discovery, and monitoring connect external systems.
E-commerce monolith: Login, Product, Order, Inventory, Payment, Billing, Notifications — all in one codebase. Works fine at first, then breaks down at scale.
Small change → huge deployment
  • Fix one line in Payment
  • Must build entire application
  • Run full regression tests
  • Deploy everything
Scaling is expensive
  • Orders → heavy traffic
  • Login → low traffic
  • Must scale both together
  • Can't scale independently
Codebase becomes unmanageable
  • 5M+ lines of code
  • Slow IDE loading
  • High coupling between modules
  • Debugging is a nightmare
One bug breaks everything
  • Payment bug → Orders fail
  • Notifications fail
  • Inventory fails
  • Everything is tightly coupled
Result: Development slows down. Scaling becomes expensive. Maintenance becomes painful. This is the pain that created microservices.
Core idea: Split into independent services — Login, Order, Product, Payment, Billing. Each has its own codebase, deployment pipeline, and scaling unit.
Benefits
Independent deploy
  • Fix bug in Payment
  • Deploy only Payment service
  • No full system redeploy
Independent scaling
  • Orders → 1000 req/sec
  • Login → 20 req/sec
  • Scale only Order service
  • Much cheaper
Smaller codebases
  • Monolith: 5M LOC
  • Each service: ~50k LOC
  • Easier to maintain
  • Faster onboarding
New problems microservices introduce
Latency
Services communicate over the network. Each hop adds latency.
Client → Order → Payment → Inventory Each arrow = a network call
Debugging complexity
A failure could be in any of 4+ services. Tracing a request across services is hard without proper monitoring.
Distributed transactions
Order DB and Payment DB are separate. If payment fails after order is created, rolling back across two services is complex.
The key interview question: How do you decide how to split a monolith? Most candidates say "just break it up" — senior engineers explain the pattern. Two main approaches exist.
Pattern 1 — Business capability
One service per business function
Divide by what the business does — not how the code is structured.
Order Service Product Service Payment Service Account Service Login Service
Best for: Simpler systems, faster initial design
Pattern 2 — Subdomain / DDD
Domain → multiple sub-services
Define domains first, then split each domain into smaller services based on bounded contexts.
Order domain
Placement Tracking Cancellation
Payment domain
Forward payment Refund
Best for: Complex systems, large teams

What if services are too small?
Over-decomposed
  • Too many network calls between services
  • High latency on every user request
  • Complex debugging across dozens of hops
  • Deployment overhead multiplies
What if services are too big?
Under-decomposed
  • Effectively back to a monolith
  • Hard to scale individual features
  • Hard to deploy independently
  • Tight coupling returns
What if services share a database?
Shared database anti-pattern
  • Strong coupling through the data layer
  • Data ownership conflicts between teams
  • Can't scale databases independently
  • Schema changes break multiple services
The sweet spot: Each service owns exactly one business capability or bounded context, has its own database, and communicates with others only through well-defined APIs or events.
Most candidates say: "Just break the system into microservices."

Senior engineers explain HOW: Decomposition strategy → Database strategy → Communication pattern → Integration pattern.
Answer structure for HLD interviews
1
Decomposition strategy Business capability or DDD? Why? Explain how you'd split this specific system.
2
Database strategy Database-per-service (preferred) or shared? How will you handle distributed transactions?
3
Communication pattern Sync (REST) for reads, async (Kafka/queues) for writes. Explain the trade-off.
4
Integration & governance API gateway for external traffic, service discovery, distributed tracing for monitoring.
