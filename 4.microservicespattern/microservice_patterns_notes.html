Strangler patternGradual monolith → microservices migration
The pain: Rewriting an entire monolith in one shot takes years, risks unknown bugs, and you can't stop 1000 req/sec production traffic. You need a safer way to migrate.
How it works
An API gateway sits in front. Traffic is gradually rerouted from the monolith to new microservices, one feature at a time.
Client ↓ API Gateway / Controller
Monolith (90%)
Microservices (10%)
Traffic migration over time
Phase 1
10% MS
Phase 2
50% MS
Phase 3
100% MS
Monolith is deleted once traffic = 0%
Analogy: Replacing an old bridge — build the new one beside it, gradually move traffic across, then tear down the old one. Zero downtime, zero big-bang risk.
Saga patternDistributed transactions with compensation
The pain: In microservices, each service has its own DB. A single user action spans multiple services. If step 3 fails after steps 1 and 2 succeeded, the data is now inconsistent with no rollback.
Happy path — order placement
1. Order Service → create order ↓ 2. Inventory Service → reduce stock ↓ 3. Payment Service → process payment
Failure + compensation
1. Order created ✓ ↓ 2. Inventory reduced ✓ ↓ 3. Payment FAILED ✗ ↓ compensation triggers Inventory Service → restore stock ↓ Order Service → cancel order
Two types of Saga
Choreography
Services talk via events
Each service listens for events and reacts. No central controller — services coordinate themselves.
Order Created event ↓ Inventory Service reacts ↓ Inventory Updated event ↓ Payment Service reacts
Pros
Simple, no central dependency
Cons
Hard to debug, circular dependencies possible
Orchestration
Central orchestrator controls flow
One coordinator calls each service in order and handles rollback on failure.
Orchestrator ↓ calls Order Service ↓ calls Inventory Service ↓ calls Payment Service
Pros
Easy to monitor, clear control flow
Cons
Central coordinator is a single point of concern
Classic interview example: User A sends ₹10 to User B. Balance deducted → Payment record fails. Saga compensation: add ₹10 back to balance. System returns to consistent state.
CQRSCommand Query Responsibility Segregation
The pain: In microservices, each service owns its own DB. You can't do a SQL JOIN across Order DB, Inventory DB, and Payment DB — that violates service boundaries. Queries that need combined data become impossible.
Write side (commands)
Each service writes to its own database. Normal microservices flow.
Order Service → Order DB Inventory Service → Inventory DB Payment Service → Payment DB
Operations: Create, Update, Delete
Read side (queries)
A separate read database is built with pre-joined, combined data. Synced via events or ETL.
Order DB Inventory DB Payment DB
↓ events / ETL sync Read DB (combined view) ↓ Client queries here
Analogy: A bank's transaction ledger (write DB) and its analytics dashboard (read DB). The dashboard pulls from the ledger but never affects transactions. Fast reads, clean writes, no coupling.
Write DBRead DB
OperationsCreate, Update, DeleteRead-only queries
Optimised forData integrity, transactionsFast reads, joins, aggregation
Sync mechanism—Events, Kafka, ETL triggers
ConsistencyStrong (immediate)Eventual
Real production systems combine all three. Strangler migrates you in safely. Saga keeps transactions consistent. CQRS makes reads fast and scalable. They're designed to work together.
Combined architecture flow
Client request ↓ API Gateway ↓ routes to microservices (via Strangler during migration) Microservices (Order, Inventory, Payment) ↓ transactions managed by Saga (Orchestrator or Choreography) ↓ publishes events to Kafka / Event Bus ↓ consumed by CQRS Read DB (combined view)

Trade-off sandbox
Without Saga
Inventory reduced, payment fails, order still active. System is permanently inconsistent with no recovery path.
Without CQRS
Cross-service joins are impossible. APIs become slow and complex. Read queries require orchestrating multiple service calls.
Without Strangler
Big-bang rewrite takes years, introduces unknown bugs, and risks production downtime. Migration becomes a high-stakes gamble.
