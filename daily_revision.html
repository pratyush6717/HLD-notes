■ High-Level Design

HLD 1Network Protocols
  • OSI 7 layers: Physical → Data Link → Network → Transport → Session → Presentation → Application
  • TCP: reliable, ordered, 3-way handshake (SYN→SYN-ACK→ACK); UDP: fast, unreliable, connectionless
  • HTTP/1.1: persistent conn; HTTP/2: multiplexing, header compression; HTTP/3: QUIC (UDP-based)
  • TLS: asymmetric handshake → negotiate symmetric session key → encrypt traffic
  • Use TCP for reliability (web, email); UDP for speed (gaming, video, DNS)
HLD 2CAP Theorem
  • Only 2 of 3 guaranteed: Consistency, Availability, Partition Tolerance
  • Network partitions always happen → must choose C or A
  • CP: MongoDB, ZooKeeper, HBase — consistent, may be unavailable during partition
  • AP: Cassandra, DynamoDB, CouchDB — always available, eventually consistent
  • PACELC: extends CAP → adds Latency vs Consistency tradeoff when no partition
HLD 3Microservices
  • vs Monolith: Monolith = single deploy unit; Microservices = independent services with own DBs
  • Pros: independent deploy, scale, tech diversity, fault isolation
  • Cons: network latency, distributed tracing, eventual consistency, complex ops
  • Sync comm: REST/gRPC; Async comm: Kafka/RabbitMQ (decoupled)
  • Each service owns its DB — no shared database
HLD 4Microservices Patterns
  • Saga: distributed txn via chain of local transactions + compensating txns on failure
  • CQRS: separate read model and write model
  • Event Sourcing: store events, not current state; rebuild state by replaying events
  • Outbox Pattern: write to DB + outbox table atomically; relay publishes to Kafka
  • Strangler Fig: gradually replace monolith by routing traffic to new microservices
HLD 5Scaling (0 to 1M Users)
  • Vertical: bigger machine (SPOF, limits); Horizontal: more machines (need LB + stateless)
  • Read scaling: read replicas; Write scaling: sharding (partition by key)
  • CDN for static assets; Redis cache to reduce DB load
  • Stateless app servers → store session in Redis, not server memory
  • Async jobs for heavy tasks (email, reports) → message queue
HLD 6Consistent Hashing
  • Hash ring: servers placed at hash positions; request → nearest clockwise server
  • Virtual nodes: each server has multiple ring positions → balanced load distribution
  • Adding/removing server → only K/N keys remapped (vs all keys in naive hashing)
  • Used by: Cassandra, DynamoDB, Memcached, Chord DHT
  • Benefits: minimal key movement on topology changes
HLD 7Back-of-Envelope Estimation
  • Storage: char=1B, int=4B, long=8B, UUID=16B, image=~300KB, video=~10MB/min
  • Throughput: 1Gbps=125MB/s; 10K QPS × 1KB = 10MB/s traffic
  • Latency: L1=1ns, RAM=100ns, SSD=100μs, Network=1-10ms, HDD=10ms
  • Peak QPS ≈ 2-3× average; 10M DAU → ~116 QPS average
  • 1 day = 86,400s ≈ 100K; 1 month = 2.5M; 1 year = 30M seconds
HLD 8SQL vs NoSQL
  • SQL: ACID, schema-fixed, joins, vertical scale (PostgreSQL, MySQL)
  • NoSQL types: Document (MongoDB), Key-Value (Redis/DynamoDB), Column (Cassandra), Graph (Neo4j)
  • BASE: Basically Available, Soft state, Eventual consistency
  • Use SQL: financial, relational, complex queries; Use NoSQL: unstructured, high velocity, massive scale
  • NoSQL: horizontal scale, flexible schema, partition-friendly
HLD 9Idempotent API
  • Same request N times = same result (no side effects on repeat)
  • Idempotent by default: GET, PUT, DELETE; Not idempotent: POST
  • Make POST idempotent: client sends Idempotency-Key header
  • Server stores key→result in DB; if key seen before, return cached result
  • Critical for payment APIs, order creation to prevent duplicate charges
HLD 10High Availability
  • 99.9% = 8.7h downtime/yr; 99.99% = 52min; 99.999% = 5min
  • Active-Passive: standby takes over on failure; Active-Active: both serve traffic
  • Replication: leader-follower or multi-leader; avoid SPOF at every layer
  • Health checks + auto-failover (LB marks unhealthy, stops routing)
  • Geographic distribution for disaster recovery
HLD 11Kafka (Distributed Messaging Queue)
  • Topic → Partitions → Offsets; consumer tracks its own offset
  • Producer → Broker cluster → Consumer Groups (each partition → one consumer per group)
  • Messages retained for configurable time (not deleted on consume)
  • Guarantees: at-least-once (default), exactly-once (transactions), at-most-once
  • Use cases: event streaming, log aggregation, decoupling, CDC
HLD 12Proxy vs Reverse Proxy
  • Forward Proxy: sits in front of clients; hides client IP (VPN, content filtering)
  • Reverse Proxy: sits in front of servers; hides servers (Nginx, HAProxy)
  • Reverse proxy features: load balancing, SSL termination, caching, compression, A/B testing
  • Clients see one IP; reverse proxy routes internally to many servers
HLD 13Load Balancer Algorithms
  • Round Robin: rotate equally; Weighted: by server capacity
  • Least Connections: route to server with fewest active connections
  • IP Hash: same client → same server (session affinity/sticky sessions)
  • L4 (transport): route by IP/port; L7 (application): route by URL, headers, content
  • Health checks: remove unhealthy backends from rotation
HLD 14Distributed Cache & Strategy
  • Cache-Aside (Lazy): app checks cache → miss → DB → populate cache
  • Write-Through: write cache + DB together; consistent but slower writes
  • Write-Behind: write cache; async write to DB; risk of data loss
  • Eviction: LRU (least recently used), LFU (least frequently used), TTL
  • Redis: single-threaded, in-memory, pub/sub, persistence (RDB snapshots / AOF log)
HLD 15Distributed Transactions
  • 2PC: prepare → commit; coordinator failure = blocking; not suitable for microservices
  • Saga (Choreography): each service emits event; next service listens; compensate on failure
  • Saga (Orchestration): central orchestrator tells each service what to do
  • Outbox Pattern: write to DB + outbox atomically; relay (CDC/Debezium) publishes to Kafka
  • Eventual consistency is the practical answer for microservices
HLD 16Database Indexing
  • B-Tree: balanced, good for range queries; default in most DBs
  • Hash index: exact match only, O(1); no range queries
  • Composite index (a,b,c): usable for queries on (a), (a,b), (a,b,c) — leftmost prefix rule
  • Covering index: all columns in index → no table lookup needed
  • Over-indexing slows writes + increases storage; index selectively
HLD 17Concurrency Control (Distributed)
  • Optimistic: no lock; check version before commit; good for low-conflict
  • Pessimistic: lock before access; good for high-conflict; risk of deadlock
  • MVCC: readers don't block writers; each txn sees snapshot (PostgreSQL, MySQL InnoDB)
  • Vector clocks: track causality in distributed systems; detect concurrent writes
HLD 18Two Phase Locking (2PL)
  • Growing phase: acquire locks only (no releases)
  • Shrinking phase: release locks only (no new acquires)
  • Strict 2PL: hold all locks until transaction commits/aborts
  • Prevents dirty reads + non-repeatable reads; but can cause deadlocks
  • Deadlock prevention: timeout, wait-die, wound-wait schemes
HLD 19OAuth 2.0
  • Auth Code Flow: most secure; for web apps with backend server
  • Client Credentials: machine-to-machine; no user involved
  • PKCE: for mobile/SPA (proof key for code exchange); prevents code interception
  • OAuth = Authorization; OpenID Connect = Authentication (adds ID token)
  • Implicit flow deprecated; never put tokens in URL
HLD 20Encryption (Symmetric & Asymmetric)
  • Symmetric (AES): same key for encrypt/decrypt; fast; key distribution problem
  • Asymmetric (RSA): public key encrypts, private key decrypts; slow
  • TLS: asymmetric handshake → agree on symmetric session key → encrypt with AES
  • Hashing: one-way, no key (SHA-256); bcrypt/Argon2 for passwords (with salt)
  • Digital signature: sign with private key, verify with public key
HLD 21JWT (JSON Web Token)
  • Structure: header.payload.signature (each Base64URL encoded)
  • Signed with secret (HS256) or RSA private key (RS256); anyone can verify with public key
  • Stateless: server doesn't store sessions; any server can verify
  • Downside: can't invalidate before expiry → use short-lived access + refresh token
  • Claims: iss, sub, exp, iat, aud; never put passwords in payload
HLD 22Thundering Herd Problem
  • Many clients request same resource simultaneously (cache expires / server restarts)
  • Mutex/lock: only one thread queries DB; others wait for result
  • Probabilistic early expiry: randomly refresh before actual TTL expires
  • Jitter on TTL: add random offset so caches don't expire simultaneously
  • Cache warming: pre-populate cache before expected spike
HLD 23API Gateway
  • Single entry point for all microservices
  • Features: auth, rate limiting, routing, load balancing, SSL termination, request aggregation
  • BFF (Backend for Frontend): separate gateway per client type (mobile/web)
  • Examples: AWS API Gateway, Kong, Netflix Zuul, Spring Cloud Gateway
  • Logging, tracing, and monitoring at one place
HLD 24Service Mesh
  • Dedicated infra layer for service-to-service communication
  • Sidecar proxy (Envoy) injected alongside each service container
  • Features: load balancing, retries, circuit breaking, mTLS, distributed tracing
  • Control plane (Istio): manages config; Data plane: handles actual traffic
  • Moves cross-cutting concerns out of application code
HLD 25How DNS Works
  • Browser cache → OS cache → Recursive Resolver → Root NS → TLD NS → Authoritative NS
  • Records: A (IPv4), AAAA (IPv6), CNAME (alias), MX (mail), TXT (verification/SPF)
  • TTL determines how long records are cached by resolvers
  • DNS load balancing: multiple A records (round-robin); Geo DNS for routing by region
HLD 26Monolith to Microservices
  • Strangler Fig: gradually route new functionality to microservices; coexist with monolith
  • Decompose by business capability or DDD bounded contexts
  • Anti-corruption layer: translator between old and new models
  • Start with 2-pizza team rule: team should be fed by 2 pizzas = right service size
  • Avoid micro-monolith: too granular = chatty services, worse than monolith
HLD 27Security Attacks
  • SQL Injection: use prepared statements / parameterized queries
  • XSS: encode output, Content-Security-Policy header, sanitize input
  • CSRF: CSRF tokens, SameSite cookie attribute, check Origin/Referer header
  • CORS: server sets Access-Control-Allow-Origin; browser enforces
  • HTTPS everywhere; input validation at boundaries; least privilege principle
HLD 28Dual Write Problem
  • Writing to two systems (DB + Kafka) atomically is impossible without coordination
  • Outbox Pattern: write to DB + outbox table in same transaction; relay picks up
  • CDC (Change Data Capture): Debezium reads DB transaction log → publishes to Kafka
  • Never write to Kafka then DB (or vice versa) without outbox — risk of partial failure
HLD 29Service Discovery
  • Client-side: client queries registry (Eureka), picks server, calls directly
  • Server-side: load balancer queries registry, routes (AWS ALB + Route53)
  • Services self-register on startup; send heartbeats; deregister on shutdown
  • Registry removes instances that miss heartbeats
  • Examples: Eureka, Consul, etcd, ZooKeeper
HLD 30Rate Limiter Algorithms
  • Token Bucket: tokens added at fixed rate; burst allowed up to bucket capacity
  • Leaky Bucket: queue requests; process at fixed rate; overflow = drop
  • Fixed Window: count per window; edge case: 2× traffic at window boundary
  • Sliding Window Log: accurate, stores all timestamps; memory-intensive
  • Sliding Window Counter: fixed window + weighted previous window; balanced
HLD 31Bulkhead Pattern
  • Isolate failures to prevent cascading (like ship bulkhead compartments)
  • Thread pool bulkhead: each downstream service gets dedicated thread pool
  • Semaphore bulkhead: limit concurrent calls to a service
  • Prevents one slow service from exhausting all threads/resources
  • Combine with circuit breaker for full resilience
HLD 32Retry Pattern
  • Retry failed requests: fixed delay, exponential backoff, jitter
  • Exponential backoff: delay = base × 2^n + random jitter
  • Only retry idempotent operations (GET, PUT); not POST without idempotency key
  • Set max retries + max timeout to prevent infinite loops
  • Jitter prevents thundering herd on retry storms
HLD 33Circuit Breaker
  • CLOSED: normal; requests pass through; track failure rate
  • OPEN: failure threshold exceeded; reject all requests immediately; return fallback
  • HALF-OPEN: after timeout, let test requests through; if success → CLOSED, else → OPEN
  • Prevents cascading failures; gives failing service time to recover
  • Libraries: Resilience4j (Java), Polly (.NET), Hystrix (deprecated)
HLD 34Design: URL Shortener
  • Key generation: Base62(auto-increment ID) or first 7 chars of MD5 hash
  • Storage: DB (id, short_key, long_url, created_at) + Redis cache for hot URLs
  • 301 (permanent, browser caches) vs 302 (temporary, always hits server)
  • Pre-generate keys with KGS (Key Generation Service) for high throughput
  • Expiry, analytics, custom alias as extensions
HLD 35Design: WhatsApp
  • WebSocket for persistent real-time connection; fallback to long-polling
  • Message queue per user; messages stored until delivered + acknowledged
  • Group messages: fanout-on-write (small groups) vs fanout-on-read (large groups)
  • E2E encryption: Signal Protocol (double ratchet algorithm)
  • Media: upload to object storage (S3), send CDN URL in message
HLD 36Design: Rate Limiter
  • Centralized: Redis INCR + EXPIRE per user/IP per time window
  • Atomic with Lua script; sliding window via Redis Sorted Sets (ZADD / ZCOUNT)
  • Response headers: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset
  • Return 429 (Too Many Requests) with Retry-After header
  • Distribute across nodes: each node tracks locally + sync via Redis
HLD 37Design: Key-Value Store
  • Consistent hashing for key distribution across nodes
  • Replication factor N; data replicated to N consecutive nodes on ring
  • Quorum: W + R > N for strong consistency; W=1,R=1 for availability
  • Gossip protocol for failure detection and membership changes
  • Versioning with vector clocks; conflict resolution: last-write-wins or merge

■ Low-Level Design

LLD 1SOLID Principles
  • Single Responsibility: one class = one reason to change
  • Open/Closed: open for extension, closed for modification (use abstraction)
  • Liskov Substitution: subclass must be usable wherever superclass is used
  • Interface Segregation: no client forced to depend on unused methods; prefer small interfaces
  • Dependency Inversion: depend on abstractions (interfaces), not concretions
LLD 2Strategy Pattern
  • Behavioral; define family of algorithms, encapsulate each, make interchangeable at runtime
  • Context holds reference to Strategy interface; swap strategy without changing Context
  • Example: PaymentStrategy (CreditCard/UPI/PayPal), SortStrategy
  • Prefer over if-else/switch chains that grow with each new algorithm
  • Enables Open/Closed principle: add strategy without modifying context
LLD 3Observer Pattern
  • Behavioral; one-to-many: Subject notifies all Observers on state change
  • Subject: addObserver(), removeObserver(), notifyObservers()
  • Example: stock ticker, notification service, Java EventListener
  • Push vs Pull model: push sends data; pull observer fetches from subject
  • Beware: memory leaks if observers not deregistered (weak references help)
LLD 4Decorator Pattern
  • Structural; add behavior to objects dynamically without subclassing
  • Wraps original object; implements same interface; chains decorators
  • Example: Coffee → MilkDecorator → SugarDecorator; Java I/O streams
  • Alternative to inheritance explosion (many feature combinations)
  • Each decorator adds one responsibility (SRP-friendly)
LLD 5Factory & Abstract Factory
  • Factory Method: subclass decides which object to create; encapsulates instantiation
  • Abstract Factory: creates families of related objects without specifying concrete classes
  • Example: ShapeFactory.create("CIRCLE"); UIFactory → WindowsUI or MacUI
  • Hides new keyword; makes code open to extension
  • Use when: object creation logic is complex or varies by config/platform
LLD 6Parking Lot Design
  • Classes: ParkingLot, ParkingFloor, ParkingSpot, Vehicle, Ticket, Payment
  • Spot types: Compact, Large, Handicapped, Motorcycle
  • Strategy for spot assignment; observer for spot availability updates
  • Ticket holds: spot, vehicle, entry time; Payment calculates fee by duration
  • Singleton ParkingLot; concurrent access with locks on spots
LLD 7Chain of Responsibility
  • Behavioral; pass request along handler chain; each handler decides to process or pass
  • Decouples sender from receivers; handler has reference to next handler
  • Example: Logger levels (DEBUG→INFO→ERROR), ATM cash dispenser, middleware
  • Request stops at first handler that handles it, OR passes through all
  • Easy to add/remove handlers without changing client
LLD 8Proxy Pattern
  • Structural; placeholder/surrogate controls access to another object
  • Virtual Proxy: lazy initialization (Hibernate lazy loading)
  • Protection Proxy: access control (check permissions before forwarding)
  • Caching Proxy: cache results of expensive operations
  • Remote Proxy: represents object in different address space (RMI, gRPC stub)
LLD 9State Pattern
  • Behavioral; object changes behavior based on internal state
  • State interface; ConcreteState classes; Context delegates behavior to current state
  • Example: VendingMachine (IdleState, HasMoneyState, DispenseState, OutOfStockState)
  • Eliminates large if-else/switch for state-dependent behavior
  • States can transition to other states; Context unaware of specific state logic
LLD 10Composite Pattern
  • Structural; tree structure; Leaf and Composite treated uniformly via Component interface
  • Component interface; Leaf (no children); Composite (has list of Components)
  • Example: FileSystem (File/Folder), UI component tree, org chart
  • Client calls same operation on leaf or composite transparently
  • Makes recursive tree operations simple and elegant
LLD 11Adapter Pattern
  • Structural; convert one interface to another that client expects
  • Object Adapter: wraps adaptee via composition (preferred)
  • Class Adapter: extends adaptee (requires multiple inheritance)
  • Example: XML→JSON adapter, legacy payment adapter, Java InputStreamReader
  • Use when: integrating incompatible legacy or third-party code
LLD 12Builder Pattern
  • Creational; construct complex objects step by step; separate construction from representation
  • Builder interface; ConcreteBuilder; Director (optional); Product
  • Example: Pizza.builder().size("L").cheese(true).build()
  • Solves telescoping constructor problem; makes optional params explicit
  • Lombok @Builder auto-generates builder for POJOs
LLD 13Facade Pattern
  • Structural; simplified interface to complex subsystem; single entry point
  • Hides complexity of multiple classes behind one clean API
  • Example: HomeTheatreFacade (controls TV, Sound, Lights together)
  • Spring's JdbcTemplate is a facade over JDBC boilerplate
  • Does NOT prevent direct access to subsystem; just provides convenience
LLD 14Bridge Pattern
  • Structural; decouple abstraction from implementation; both vary independently
  • Abstraction holds reference to Implementor interface
  • Example: Shape (abstraction) + Color (implementor); Device + RemoteControl
  • Prevents Cartesian product of subclasses (e.g., RedCircle, BlueCircle, RedSquare...)
  • Use when both abstraction and implementation should be extensible
LLD 15Flyweight Pattern
  • Structural; share common state among many fine-grained objects to reduce memory
  • Intrinsic state: shared, stored in flyweight; Extrinsic state: unique, passed in by client
  • Example: character glyphs in text editor, trees in forest game
  • FlyweightFactory caches and reuses existing flyweights
  • Use when: many similar objects consume too much memory
LLD 16Command Pattern
  • Behavioral; encapsulate request as object; supports undo/redo, queuing, logging
  • Command interface: execute() + undo(); Invoker; Receiver
  • Example: text editor undo/redo, remote control buttons, transaction log replay
  • Invoker doesn't know what Receiver does; just calls execute()
  • Macro commands: composite of multiple commands executed together
LLD 17Iterator Pattern
  • Behavioral; sequential access to collection without exposing internal structure
  • Iterator: hasNext(), next(); Iterable: iterator()
  • Java: Iterable + Iterator, enables enhanced for-each loop
  • Can have multiple iterators on same collection simultaneously
  • Fail-fast iterator throws ConcurrentModificationException on structural modification
LLD 18Mediator Pattern
  • Behavioral; central mediator handles communication; objects don't reference each other directly
  • Reduces many-to-many → one-to-many relationships; centralizes coordination
  • Example: ATC (Air Traffic Control), ChatRoom mediator, EventBus
  • Objects call mediator.notify(); mediator decides who to inform
  • Risk: mediator can become a God Object if over-used
LLD 19Visitor Pattern
  • Behavioral; add new operations to existing object structure without modifying it
  • Visitor: visit(ConcreteElement); Element: accept(Visitor) — double dispatch
  • Example: AST traversal, tax calculation on different item types, export (JSON/XML)
  • Adding new operation = new Visitor class; adding new element = update all visitors
  • Use when: many unrelated operations on stable object structure
LLD 20Template Method Pattern
  • Behavioral; define algorithm skeleton in base class; subclasses fill in specific steps
  • Base class calls abstract methods in fixed order; subclass implements them
  • Hook methods: optional steps subclass may override (with default impl)
  • Example: DataMiner (openFile→extractData→parseData→analyzeData→report)
  • Hollywood Principle: "Don't call us, we'll call you" — base calls subclass
LLD 21Memento Pattern
  • Behavioral; capture and restore object state without violating encapsulation
  • Originator creates Memento; Caretaker stores list of Mementos; Memento is opaque
  • Example: text editor undo, game save states, transaction rollback
  • Memento only exposes state to Originator (narrow interface to others)
LLD 22MVC Pattern
  • Model: data + business logic; View: UI rendering; Controller: handles input, updates model/view
  • MVP: Presenter between Model and View; View is passive
  • MVVM: ViewModel exposes data bindings; View auto-updates (Angular, SwiftUI)
  • Spring MVC: @Controller → @Service → @Repository
  • Separation of concerns; testability of Model/Controller independent of UI
LLD 23Object Pool Pattern
  • Creational; reuse expensive objects instead of creating/destroying repeatedly
  • Pool manages: available objects, borrowed objects, max size
  • Client borrows from pool, uses, then returns (not discards)
  • Example: DB connection pool, thread pool, HTTP connection pool
  • HikariCP is a high-performance JDBC connection pool
LLD 24Null Object Pattern
  • Return NullObject instead of null; NullObject implements interface but does nothing
  • Eliminates null checks; client treats null and real object uniformly
  • Example: NullLogger implements Logger but logs nowhere
  • Reduces NullPointerException risk; cleaner client code
LLD 25Design: BookMyShow
  • Classes: Theatre, Screen, Show, Seat, Booking, User, Payment
  • Seat states: AVAILABLE, LOCKED (TTL), BOOKED
  • Concurrent booking: optimistic locking on seat; only one booking succeeds
  • Payment timeout releases seat lock back to AVAILABLE
  • Observer for seat availability notifications; strategy for pricing
LLD 26Design: Splitwise
  • Classes: User, Group, Expense, Split (Equal/Exact/Percentage), Balance, Settlement
  • Split strategies: Equal, Exact amount, Percentage (Strategy Pattern)
  • Balance graph: who owes whom; simplify debts (greedy: max creditor ← max debtor)
  • Observer for expense added → update all member balances
LLD 27Design: Elevator System
  • Classes: Elevator, Request (floor, direction), ElevatorController, Door
  • States: IDLE, MOVING_UP, MOVING_DOWN; transitions via State Pattern
  • Scheduling: SCAN (go up/down servicing all floors) or LOOK algorithm
  • Multiple elevators: dispatcher assigns based on direction + proximity
LLD 28Design: ATM
  • Classes: ATM, Card, Account, Transaction, CashDispenser, Receipt
  • State Pattern: IdleState → CardInsertedState → PINVerifiedState → TransactionState → EjectCardState
  • Chain of Responsibility for cash dispensing (₹500 → ₹200 → ₹100 handlers)
  • Singleton ATM; thread-safe withdrawal with account balance locking

■ Java Core

Java 1OOPs — 4 Pillars
  • Encapsulation: bundle data + methods; expose via getters/setters; hide internals
  • Inheritance: is-a relationship; extends; single inheritance (use interfaces for multiple)
  • Polymorphism: compile-time (overloading by signature) vs runtime (overriding + dynamic dispatch)
  • Abstraction: hide implementation; expose contract via abstract class or interface
Java 2JVM, JRE, JDK
  • JDK = JRE + dev tools (javac, javadoc, jdb); needed for development
  • JRE = JVM + standard libraries; needed to run Java apps
  • JVM: Class Loader → Bytecode Verifier → Interpreter + JIT Compiler + GC
  • JIT: identifies hot code paths, compiles to native machine code for performance
  • Write once, run anywhere: .java → .class (bytecode) → JVM executes
Java 3Primitive & Reference Types
  • Primitives: byte(1), short(2), int(4), long(8), float(4), double(8), char(2), boolean
  • References stored on stack; objects on heap; GC manages heap
  • Autoboxing: int → Integer; Unboxing: Integer → int; beware NullPointerException
  • String pool: literals interned; "abc"=="abc" true; new String("abc")=="abc" false
  • Use .equals() for value comparison; == compares references
Java 4Memory & Garbage Collection
  • Stack: method frames, local variables (LIFO, thread-private); Heap: objects (shared, GC-managed)
  • Young Gen: Eden → Survivor (S0/S1) → Old Gen (after N minor GCs)
  • Minor GC = Young Gen; Major GC = Old Gen; Full GC = entire heap (stop-the-world)
  • Algorithms: Serial, Parallel, G1 (default Java 9+), ZGC, Shenandoah (low-latency)
  • GC roots: static fields, local vars, active threads; unreachable objects collected
Java 5Singleton & Immutable Classes
  • Singleton: private constructor + static instance + getInstance()
  • Thread-safe: double-checked locking with volatile, OR enum singleton (safest)
  • Enum singleton: enum Singleton { INSTANCE; } — handles serialization + reflection
  • Immutable: final class, final fields, no setters, defensive copy in constructor/getters
  • String, Integer are immutable; safe to share across threads without synchronization
Java 6Interfaces (Java 8+)
  • All methods public+abstract by default (before Java 8)
  • Default methods (Java 8): concrete methods in interface; can be overridden
  • Static methods (Java 8): belong to interface; called via InterfaceName.method()
  • Private methods (Java 9): helper methods not exposed outside interface
  • Class can implement multiple interfaces; interface can extend multiple interfaces
Java 7Functional Interfaces & Lambdas
  • @FunctionalInterface: exactly one abstract method
  • Built-in: Function<T,R>, Predicate<T>, Consumer<T>, Supplier<T>, BiFunction<T,U,R>
  • Lambda: (params) -> expr; captures effectively-final variables from enclosing scope
  • Method reference: Class::staticMethod, instance::method, Class::new
  • Enables functional programming style; used heavily in Streams API
Java 8Exception Handling
  • Checked: must handle (IOException); extends Exception
  • Unchecked: runtime exceptions (NPE, IndexOutOfBounds); extends RuntimeException
  • Error: JVM errors (OutOfMemoryError); don't catch
  • try-with-resources: auto-closes AutoCloseable; finally always runs even with exception
  • Multi-catch: catch(IOException | SQLException e); exception variable is effectively final
Java 9Collections Hierarchy
  • Collection → List (ArrayList, LinkedList), Set (HashSet, TreeSet), Queue (PriorityQueue, ArrayDeque)
  • Map: HashMap, TreeMap, LinkedHashMap — NOT a Collection
  • Fail-fast iterators throw ConcurrentModificationException; fail-safe (CopyOnWrite) don't
  • Unmodifiable: List.of(), Map.of(), Collections.unmodifiableList()
  • Thread-safe: ConcurrentHashMap, CopyOnWriteArrayList, Collections.synchronizedList()
Java 10HashMap Internal Working
  • Array of buckets; key.hashCode() → bucket index; linked list for collisions
  • Java 8+: bucket > 8 entries → convert to TreeMap (O(log n) vs O(n))
  • Load factor 0.75; resizes (doubles + rehashes) when 75% full
  • null key allowed (always goes to bucket 0); null value allowed
  • Contract: if a.equals(b) then a.hashCode() == b.hashCode() (reverse not required)
Java 11Streams API (Java 8)
  • Lazy evaluation: intermediate ops not executed until terminal op called
  • Intermediate: filter, map, flatMap, sorted, distinct, limit, skip, peek
  • Terminal: collect, forEach, reduce, count, min, max, findFirst, anyMatch, allMatch
  • Collectors: toList(), groupingBy(), joining(), counting(), partitioningBy(), toMap()
  • Parallel: .parallel() → uses ForkJoinPool.commonPool(); good for CPU-bound, large data
Java 12Multithreading Basics
  • Thread.start() → JVM creates OS thread; run() executes in new thread
  • Runnable (no return), Callable<T> (returns Future<T>)
  • synchronized: intrinsic lock (monitor); method or block level
  • volatile: visibility guarantee (no CPU cache); NOT atomicity for compound ops
  • Problems: race condition, deadlock, livelock, starvation, memory visibility
Java 13Thread Lifecycle
  • States: NEW → RUNNABLE → {BLOCKED | WAITING | TIMED_WAITING} → TERMINATED
  • wait() releases lock; notify()/notifyAll() wakes waiting threads (inside synchronized)
  • join(): calling thread waits for target thread to finish
  • interrupt(): sets interrupt flag; thread checks with isInterrupted()
  • BLOCKED: waiting for monitor lock; WAITING: waiting for signal (wait/join/park)
Java 14Locks & Conditions
  • ReentrantLock: same thread can lock multiple times; tryLock(), lockInterruptibly()
  • ReadWriteLock: multiple concurrent readers OR one exclusive writer
  • StampedLock (Java 8): optimistic read → validate stamp after read; best performance
  • Condition: await() / signal() / signalAll() — more flexible than wait/notify
  • Always unlock in finally block: lock.lock(); try{...} finally{lock.unlock();}
Java 15Atomic & Lock-Free (CAS)
  • CAS (Compare-And-Swap): atomic HW instruction; no lock needed
  • AtomicInteger, AtomicLong, AtomicReference: CAS-based, lock-free
  • AtomicInteger.incrementAndGet() is atomic; volatile int i++ is NOT (3 ops)
  • volatile: ensures visibility (no CPU register caching); not compound atomicity
  • ABA problem with CAS: use AtomicStampedReference to track version
Java 16Thread Pools (ExecutorService)
  • newFixedThreadPool(n): n threads, unbounded queue
  • newCachedThreadPool(): unlimited threads; idle 60s; good for short I/O tasks
  • newSingleThreadExecutor(): one thread; sequential task execution
  • ThreadPoolExecutor: configure corePool, maxPool, keepAlive, queue, RejectionPolicy
  • Rejection policies: AbortPolicy(throw), CallerRunsPolicy, DiscardPolicy, DiscardOldestPolicy
Java 17CompletableFuture
  • Async computation: supplyAsync(() -> result), runAsync(() -> void)
  • Chain: thenApply (transform), thenAccept (consume), thenCompose (flatMap), thenCombine
  • Error handling: exceptionally(ex -> fallback), handle((r,ex) -> ...), whenComplete
  • allOf(): wait for all; anyOf(): first to complete
  • Non-blocking alternative to Future.get(); uses ForkJoinPool by default
Java 18Virtual Threads (Java 21)
  • JVM-managed lightweight threads; millions possible (vs thousands for OS threads)
  • Thread.ofVirtual().start(runnable) or Executors.newVirtualThreadPerTaskExecutor()
  • Mounted on platform (carrier) threads; unmounted on blocking I/O
  • Best for I/O-bound tasks (web servers, DB calls); NOT CPU-bound tasks
  • Don't pool virtual threads; create as needed — pooling defeats the purpose
Java 19ThreadLocal
  • Per-thread variable; each thread has own independent copy
  • get() / set(value) / remove(); override initialValue() for defaults
  • Use cases: request context in web apps, transaction management, SimpleDateFormat
  • Always call remove() in finally — thread pools reuse threads; stale data persists
  • InheritableThreadLocal: child threads inherit parent's value
Java 20Records & Sealed Classes
  • Records (Java 16): immutable data carrier; auto-generates constructor, getters, equals, hashCode, toString
  • record Point(int x, int y) {} → point.x(), point.y()
  • Sealed Classes (Java 17): restrict which classes can extend
  • sealed class Shape permits Circle, Rectangle; subclasses: final/sealed/non-sealed
  • Enable exhaustive pattern matching in switch
Java 21Pattern Matching & Switch Expressions
  • instanceof pattern (Java 16): if (obj instanceof String s) { s.length() }
  • Switch expression (Java 14): switch(x) { case A -> ...; } — no fall-through
  • Pattern switch (Java 21): case Integer i when i > 0 -> ...
  • yield to return value from a block in switch expression
  • Sealed classes + pattern switch = exhaustiveness checking by compiler
Java 22Optional (Java 8)
  • Container that may or may not contain a value; avoids null returns
  • Create: Optional.of(val), Optional.ofNullable(val), Optional.empty()
  • Use: orElse(), orElseGet(), orElseThrow(), map(), flatMap(), filter(), ifPresent()
  • Don't use as field type or parameter; only as return type
  • orElse() always evaluates arg; orElseGet(() -> ...) is lazy — prefer it for expensive defaults
Java 23Generics & Wildcards
  • Type safety at compile time; type erasure at runtime
  • Bounded: <T extends Number> (upper bound), <? super Integer> (lower bound)
  • Wildcard <?>: unknown type; read-only (can't add elements except null)
  • PECS: Producer Extends (<? extends T> for reading), Consumer Super (<? super T> for writing)
  • Covariant: List<Dog> is NOT a subtype of List<Animal> — use List<? extends Animal>
Java 24Comparator & Comparable
  • Comparable: natural ordering; compareTo() in class itself; one sort order
  • Comparator: external; compare(a,b); multiple sort strategies
  • Comparator.comparing(Person::getName).thenComparing(Person::getAge).reversed()
  • Return negative (a<b), zero (a==b), positive (a>b)
  • TreeMap/TreeSet use Comparable/Comparator for ordering
Java 25Java Annotations
  • Metadata on code elements; processed at compile-time (APT) or runtime (Reflection)
  • Meta-annotations: @Target (where), @Retention (SOURCE/CLASS/RUNTIME), @Inherited, @Repeatable
  • Custom: public @interface MyAnno { String value() default ""; }
  • Used by: Spring (DI), JUnit (test discovery), Lombok (code gen), JPA (ORM)
  • Runtime annotations readable via Reflection: class.getAnnotation(MyAnno.class)

SB 1Spring Boot Fundamentals
  • @SpringBootApplication = @Configuration + @EnableAutoConfiguration + @ComponentScan
  • Auto-configuration: detects classpath, configures beans automatically (e.g., DataSource if H2 present)
  • Embedded server (Tomcat/Jetty/Undertow); no WAR deployment needed
  • Starter POMs: spring-boot-starter-web, spring-boot-starter-data-jpa etc.
  • Convention over configuration; opinionated defaults; override with application.properties
SB 2Layered Architecture
  • Controller (HTTP) → Service (business logic) → Repository (data access) → DB
  • DTO: Data Transfer Object (between Controller/Service; not entity)
  • Entity: JPA-managed DB object; maps to table
  • Controller should be thin; business logic in Service
  • Repository extends JpaRepository<Entity, Id> for CRUD + pagination
SB 3IoC & Bean Lifecycle
  • IoC container instantiates, assembles, and manages beans
  • Lifecycle: Instantiate → Inject Dependencies → @PostConstruct → Ready → @PreDestroy → Destroy
  • Stereotypes: @Component, @Service, @Repository, @Controller (all create beans)
  • @Bean in @Configuration class: explicit bean definition
  • @Repository adds exception translation (SQL → Spring exceptions)
SB 4Dependency Injection
  • Constructor injection (preferred): immutable fields, easy to test, explicit dependencies
  • Setter injection: for optional dependencies
  • Field injection: @Autowired on field — avoid (hard to test, hidden deps)
  • @Qualifier("beanName"): disambiguate when multiple beans of same type
  • @Primary: default bean; @Lazy: initialize on first use
SB 5Bean Scopes
  • Singleton (default): one instance per ApplicationContext; shared
  • Prototype: new instance per injection/lookup; Spring doesn't destroy it
  • Request: one per HTTP request (web only)
  • Session: one per HTTP session (web only)
  • Injecting prototype/request bean into singleton: use proxyMode = ScopedProxyMode.TARGET_CLASS
SB 6Controller Annotations
  • @RestController = @Controller + @ResponseBody
  • Mapping: @GetMapping, @PostMapping, @PutMapping, @DeleteMapping, @PatchMapping
  • Params: @PathVariable, @RequestParam, @RequestBody, @RequestHeader, @CookieValue
  • @ResponseStatus(HttpStatus.CREATED) sets default response code
  • @Valid on @RequestBody triggers Bean Validation (@NotNull, @Size, etc.)
SB 7AOP (Aspect-Oriented Programming)
  • Cross-cutting concerns: logging, security, transactions, auditing
  • @Aspect, @Pointcut; advice types: @Before, @After, @AfterReturning, @AfterThrowing, @Around
  • Pointcut expression: execution(* com.example.service.*.*(..))
  • @Around: most powerful; call pjp.proceed() to execute method; wrap with try-catch
  • Works via proxy (JDK or CGLIB); self-invocation bypasses proxy = no AOP
SB 8@Transactional
  • Auto-rollback on RuntimeException; use rollbackFor=Exception.class for checked
  • Propagation: REQUIRED (default, joins existing), REQUIRES_NEW (new txn), NESTED, NOT_SUPPORTED
  • Isolation: READ_UNCOMMITTED, READ_COMMITTED (default), REPEATABLE_READ, SERIALIZABLE
  • readOnly=true: hints JPA to skip dirty checking → performance boost
  • Self-invocation doesn't work (proxy issue) — call from another bean
SB 9@Async & Async Config
  • @EnableAsync on config; @Async on method → runs in separate thread
  • Return void or CompletableFuture<T> from async methods
  • Default executor: SimpleAsyncTaskExecutor (creates new thread per call)
  • Configure ThreadPoolTaskExecutor bean named taskExecutor for pooling
  • Self-invocation doesn't work; must be called from different bean
SB 10Filters vs Interceptors
  • Filter (Servlet level): runs before DispatcherServlet; implement javax.servlet.Filter
  • Interceptor (Spring MVC): inside DispatcherServlet; has access to Handler + ModelAndView
  • Filter: doFilter(req, res, chain); call chain.doFilter() to continue
  • Interceptor: preHandle() (before), postHandle() (after), afterCompletion() (cleanup)
  • Use Filter for: auth, encoding, CORS; Interceptor for: logging, role-checking
SB 11Exception Handling
  • @ExceptionHandler in controller or @ControllerAdvice (global)
  • @RestControllerAdvice = @ControllerAdvice + @ResponseBody
  • Return ResponseEntity with appropriate HTTP status code
  • ProblemDetail (RFC 7807 / Spring 6): standardized error response structure
  • Order: local @ExceptionHandler wins over global @ControllerAdvice
SB 12JPA — Entity Lifecycle & Caching
  • Lifecycle: Transient → Managed (persist/find) → Detached (close/detach) → Removed (remove)
  • 1st level cache: per EntityManager (within same transaction); automatic
  • Calling find() twice → 2nd call hits cache; no DB query
  • 2nd level cache: shared across EntityManagers; configure with Ehcache/Redis
  • Dirty checking: JPA detects changed managed entities → auto-updates on flush
SB 13JPA Relationships & Fetch
  • Annotations: @OneToOne, @OneToMany, @ManyToOne, @ManyToMany
  • Default fetch: @ManyToOne/@OneToOne = EAGER; @OneToMany/@ManyToMany = LAZY
  • N+1 problem: 1 query for parents + N queries for each child collection
  • Fix N+1: JOIN FETCH in JPQL, @EntityGraph, or @BatchSize
  • CascadeType: PERSIST, MERGE, REMOVE, ALL; orphanRemoval=true for child cleanup
SB 14Spring Security Architecture
  • Filter chain: ordered SecurityFilters process each request
  • AuthenticationManager → AuthenticationProvider → UserDetailsService
  • SecurityContextHolder stores Authentication (thread-local)
  • Configure via SecurityFilterChain bean (lambda DSL)
  • @EnableWebSecurity; CSRF enabled by default (disable for stateless REST)
SB 15JWT Authentication

  • Generate: Jwts.builder().subject(user).expiration(date).signWith(key).compact()
  • Validate: parse → check expiry → check signature; return UsernamePasswordAuthenticationToken
  • Set in SecurityContext: SecurityContextHolder.getContext().setAuthentication(auth)
  • Stateless: sessionCreationPolicy(STATELESS); no JSESSIONID
SB 16Method Security

  • @PostAuthorize("returnObject.owner == principal.username"): after method
  • @Secured("ROLE_ADMIN"): simple role check; no SpEL

SB 22@Transactional Propagation Deep Dive
  • REQUIRED: use existing txn; create new if none (default)
  • REQUIRES_NEW: always new txn; suspends existing; independent commit/rollback
  • NESTED: savepoint in existing txn; rollback to savepoint on failure
  • SUPPORTS: join if exists; no txn if none; NOT_SUPPORTED: suspend existing, run without
  • NEVER: throw if txn exists; MANDATORY: throw if NO txn exists
