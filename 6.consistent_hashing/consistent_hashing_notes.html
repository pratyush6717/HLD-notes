The pain: Simple hashing uses hash(key) % N. When you add a server (N changes), almost every key remaps to a different server. For millions of records this is catastrophic — massive data movement.
Simple hashing — disaster on resize
  • userId 25 → DB1 (with 3 servers)
  • Add 1 server → N becomes 4
  • userId 25 now hashes to DB2
  • Almost all keys need to move
  • Millions of records = hours of downtime
Consistent hashing — minimal movement
  • Keys and servers placed on a ring
  • Add/remove a server → only ~1/N keys move
  • 3 servers → 4 servers: ~25% keys move
  • No cascading remapping
  • System stays stable
Core idea in 3 steps
Step 1 — Build ring
Create a circular hash space (0 → 2³²). Hash each server to a position on the ring.
hash(DB1) → position 2
hash(DB2) → position 6
hash(DB3) → position 10
Step 2 — Place keys
Hash each key to a position on the same ring.
hash(user1) → 3
hash(user2) → 7
hash(user3) → 11
Step 3 — Assign clockwise
Each key goes to the first server found by moving clockwise on the ring.
user1 (pos 3) → DB2 (pos 6)
user2 (pos 7) → DB3 (pos 10)
user3 (pos 11) → DB1 (pos 2 wrap)
Analogy: Pizza restaurants on a circular road. Each customer goes to the nearest clockwise restaurant. A new restaurant opens — only nearby customers switch. Everyone else stays put.

The problem: Servers may hash to positions that create uneven segments. One server might "own" 60% of the ring while another owns 10% — causing a hotspot.
Without virtual nodes
  • DB1 at position 2
  • DB2 at position 3
  • DB3 at position 10
  • DB3 owns positions 3→10 (huge segment)
  • DB3 gets overloaded — hotspot
With virtual nodes
  • Each server placed at multiple positions
  • DB1 → positions 2, 8, 11
  • DB2 → positions 4, 6, 10
  • DB3 → positions 1, 7, 9
  • Load distributed evenly across all servers
How it works: Instead of hashing hash(DB1), we hash hash(DB1-1), hash(DB1-2), hash(DB1-3) — placing the same physical server at multiple ring positions. The more virtual nodes per server, the more even the distribution.

CachingDistribute cache keys across nodes. Avoids full cache invalidation on resize.Redis Cluster, Memcached

Connection to other scaling concepts: Consistent hashing is the mechanism that makes horizontal DB sharding, distributed caching, and load balancing all work without full rebalancing. It's the glue of distributed systems.
Trade-off sandbox
Without consistent hashing
Adding a server triggers 100% key remapping. Millions of DB records need to move. Hours of downtime.
Without virtual nodes
Uneven ring segments cause hot spots. One server carries 60% load while others idle.
Server crashes
Only keys between the crashed server and the next server clockwise need to move. System stays stable.
Interview question: "How do distributed caches rebalance when nodes are added?"

Answer structure: Consistent hashing → hash ring → clockwise assignment → virtual nodes → only 1/N keys move.
