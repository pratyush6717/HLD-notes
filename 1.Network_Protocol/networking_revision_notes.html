Network Protocols
Definition: A network protocol is a set of rules two computers follow to communicate — a language + grammar for machines. Ensures reliable data exchange, ordering, and delivery.
Without protocols
  • Machines can't understand each other
  • Data arrives out of order
  • Packets are lost
  • No standard message format
With protocols
  • Standardised data format
  • Reliable delivery guaranteed
  • Ordered transmission
  • Clients and servers talk reliably

HTTP vs WebSocket
HTTP
Request–response
Client sends request → Server responds → Connection closes. Can't push data to client.
Analogy: Sending an email — one shot, conversation ends
WebSocket
Persistent connection
Single connection stays open. Both sides can send data at any time. No polling needed.
Analogy: A phone call — both parties talk freely
Evolution of polling
Polling
Client asks every N seconds. Thousands of useless requests. Server overload.
Long polling
Server holds request until data arrives. Better, but connection resets frequently. Still HTTP overhead.
WebSocket
Persistent connection. Server pushes instantly. No resets, no overhead per message.

Don't use WebSocket for simple APIs — connections stay open and consume memory. Use HTTP/REST unless you need real-time bidirectional communication.
TCP vs UDP
TCP
Reliable, ordered
Handshake → packets numbered → ACKs sent → missing packets retransmitted. Slower but guaranteed delivery.
Analogy: Courier with signature confirmation
UDP
Fast, unreliable
Send and forget. No connection setup, no ACKs, no reordering. Dropped packets are ignored.
Analogy: Live radio broadcast — lost signal is ignored

Why not TCP for video? If a packet is lost, TCP freezes and waits. In a video call, that gap is worse than just skipping the frame. QUIC (used by HTTP/3) builds reliability on top of UDP to get the best of both.
WebRTC
Problem it solves: Video through a server is expensive (all bandwidth passes through it). WebRTC enables direct browser-to-browser communication — server is only needed for setup.
3-stage connection process
1 — Signaling
Peers exchange IP, network capabilities, and metadata via a signaling server (WebSocket or HTTP). Not part of WebRTC itself — developer implements.
2 — ICE / STUN
Most devices are behind NAT. STUN server helps each peer discover its public IP + port. ICE finds the best network path.
3 — TURN (fallback)
If direct connection fails (strict firewall), a TURN server relays traffic. More expensive — avoided when possible.
Trade-off / edge caseWhat happens
No signaling serverPeers can't find each other — connection impossible
Strict firewallDirect path fails → TURN relay needed (expensive)

Why not WebRTC for chat? Managing many peer connections is complex, and there's no central storage for message history. Chat needs a server in the middle.

Why not WebSocket for video? At 2 Mbps per stream, routing 1M video calls through a server = 2 Tbps. That's where the server explodes.
