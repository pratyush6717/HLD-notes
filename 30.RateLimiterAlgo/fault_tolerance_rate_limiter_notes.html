Cascading failure: Product Service responds in 60s. Order Service threads all wait. Thread pool fills. New requests rejected. Order Service looks down to Client. Entire system collapses from one slow dependency.
Cascade failure
Client → many requests ↓ Order Service threads: all waiting 60s ↓ thread pool fills New requests rejected → HTTP 503 ↓ Product Service: slow but not dead ↓ but it killed Order anyway Entire system down
Without fault tolerance
  • One slow service blocks all threads
  • Thread pool exhaustion
  • Cascade spreads to callers
  • System collapses from one weak point
With fault tolerance
  • Rate limiter: control incoming traffic
  • Bulkhead: isolate failures per service
  • Time limiter: don't wait forever
  • Circuit breaker: stop calling failing service
  • Retry: try again with backoff
Definition: A fault-tolerant service survives even when its dependencies fail — by controlling failures and preventing them from spreading.
The order matters. Each tool in the chain serves a different purpose. Wrong order = tools fighting each other (e.g. Retry before RateLimiter = useless retries flooding the system).
The correct execution order
Rate Limiter → Bulkhead → Time Limiter → Circuit Breaker → Retry
Rate Limiter
Controls incoming traffic. Rejects excess before they enter. First line of defense.
Bulkhead
Isolates thread pools per downstream service. Product failure can't steal Order's threads.
Time Limiter
Caps maximum wait time. Product takes 60s? Timeout at 2s. Thread released immediately.
Circuit Breaker
Tracks failure rate. If 50%+ fail → open circuit → stop calling for 60s. Auto-resets and tests recovery.
Retry
On transient failure, retry N times with exponential backoff. Comes last — only retries after circuit is closed.
Why Retry is last: If Retry is before Circuit Breaker, a failing service gets retried 3× per request → 3× the load → makes outage worse. Circuit Breaker first stops retries from firing during an outage.
5 algorithms to know. Each handles the traffic control differently. Token Bucket is the most used — it allows controlled bursts while maintaining average rate.
Fixed WindowSimple but has edge case
Divide time into fixed blocks. Max N requests per window. Simple to implement.
✗ Edge problem: 5 requests at end of window + 5 at start = 10 in 1 second. Window boundary exploited.
Sliding LogAccurate but heavy
Store timestamp of every request. Count how many in last N seconds. Reject if over limit.
✔ Accurate — no edge case
✗ Memory heavy — must store all timestamps. Cleanup needed.
Sliding Window CounterBalance of accuracy + efficiency
Divide window into smaller sub-windows. Track counts per sub-window. Weighted average across current + previous window.
✔ More accurate than fixed window, less heavy than sliding log
✗ More complex to implement. More memory than fixed window.
Token BucketMost used
Bucket holds N tokens. Each request consumes 1 token. No token = rejected. Tokens refill at fixed rate (e.g. 5/sec).
✔ Allows controlled bursts (use saved-up tokens)
✔ Simple and smooth traffic control
✔ Used by AWS, Stripe, most production systems
✗ Wrong bucket size → burst overload
Analogy: Movie ticket counter — tokens = tickets. Ticket available = entry. Tickets gone = wait.
Leaky BucketStable output, no bursts
Requests queue up. Processed at fixed constant rate — like a bucket with a hole. Output is always smooth.
✔ No output spikes — perfectly smooth rate
✗ High latency (requests queue). Queue bottleneck under load.
Analogy: Queue outside a venue — everyone enters at same fixed pace.

What-if sandbox
No rate limiter
  • Traffic spike → threads fill up
  • System crashes under load
  • DDoS = instant outage
Token bucket too large
  • Burst traffic allowed beyond capacity
  • Downstream overloaded despite rate limiter
Queue too small (leaky bucket)
  • Requests rejected before processing
  • High error rate under normal traffic spikes
Queue too large (leaky bucket)
  • Requests wait too long in queue
  • High latency — user experience degrades
Rate limiter definition: Restricts the number of requests a service can handle in a given time window to prevent overload.

Token Bucket line: Token bucket is most commonly used because it allows controlled bursts while maintaining the average rate.
