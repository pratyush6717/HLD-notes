🔹 1. What is @Transactional?

👉 It means:

“Run this method inside a DB transaction.”

🧠 Simple meaning
Start Transaction
   ↓
Run all DB operations
   ↓
If success → COMMIT
If error → ROLLBACK
🔹 2. Real-Life Example (Best way to understand)

👉 Money transfer:

Debit from A
Credit to B
❌ Problem (without transaction)
A se paisa cut gaya ✔
B ko mila hi nahi ❌

👉 Data inconsistent 😵

✅ With @Transactional
Either BOTH succeed ✔
OR BOTH fail (rollback) ✔
🔹 3. Code Example
@Service
public class PaymentService {

    @Transactional
    public void transfer(Long from, Long to, double amount) {
        debit(from, amount);
        credit(to, amount);
    }
}
🧠 What Spring does internally

Because of AOP:

Call method
↓
Proxy intercepts
↓
START TRANSACTION
↓
Run method
↓
If success → COMMIT
If error → ROLLBACK
🔥 4. ACID Properties (Interview MUST)
🅰️ Atomicity

👉 All or nothing

Debit + Credit → both happen
Else → none happen
🅲 Consistency

👉 DB always valid state me rahe

Example:

Balance negative nahi hona chahiye
🅸 Isolation

👉 Multiple transactions ek dusre ko disturb na kare

Example:

2 users ek hi account update kar rahe
🅳 Durability

👉 Once committed → permanent

Even if system crash 💥

🔹 5. Default Behavior (VERY IMPORTANT ⚠️)
✅ Success case
No exception → COMMIT
❌ Failure case
RuntimeException → ROLLBACK
⚠️ BUT IMPORTANT

👉 Checked exception pe rollback nahi hota

Example
@Transactional
public void test() throws Exception {
    debit();   // runs
    throw new Exception("error"); // ❌ NO rollback by default
}

👉 Data commit ho jayega 😵

🔹 6. Fix for Checked Exception
@Transactional(rollbackFor = Exception.class)

👉 Now:

Any exception → rollback ✔
🔹 7. Very Important Interview Point

👉 Why only RuntimeException?

Because:

Spring follows EJB style default
Checked exceptions = “expected business scenario”
Runtime = “unexpected failure”
🔥 8. Connection with AOP

👉 Same concept you learned:

@Transactional = AOP

Spring internally:

Proxy → Start Tx → Call method → Commit/Rollback
🔹 9. Visual Flow
Client
  ↓
Proxy (Transactional)
  ↓
Start Transaction
  ↓
Your Method
  ↓
Commit / Rollback
⚡ Final Summary
@Transactional = wraps method in DB transaction
Success → commit
RuntimeException → rollback
Checked exception → no rollback (by default)
Uses AOP internally (proxy-based)
🧠 One-line understanding

👉 @Transactional ensures either everything succeeds or nothing is saved.

Bilkul sahi 👍 — normal JDBC ya old style me tum khud likhte ho.
Par Spring ka magic yahi hai: tumhe likhna nahi padta.

🔹 1. Tum jo bol rahe ho (Manual way)

Agar Spring use na kare, toh tum aise likhoge:

Connection con = dataSource.getConnection();

try {
    con.setAutoCommit(false);   // BEGIN TRANSACTION

    debit(con);
    credit(con);

    con.commit();               // COMMIT
} catch (Exception e) {
    con.rollback();             // ROLLBACK
}

👉 Yahi tum keh rahe ho ✔

🔹 2. Problem with this approach
Har method me same boilerplate ❌
Error handling complex ❌
Code messy ❌
Easy to forget rollback ❌
🔹 3. Spring ka approach (@Transactional)

Tum sirf ye likhte ho:

@Transactional
public void transfer() {
    debit();
    credit();
}
🧠 Behind the scenes kya hota hai

👉 Spring internally EXACTLY wahi karta hai jo tum manually likh rahe the:

Proxy intercepts
↓
BEGIN TRANSACTION
↓
Run your method
↓
If success → COMMIT
If error → ROLLBACK
🔥 4. Direct comparison
Manual JDBC	Spring
setAutoCommit(false)	Automatic
commit()	Automatic
rollback()	Automatic
try-catch	Handled by Spring
Boilerplate code	Clean code
🔹 5. Important understanding

👉 Tum transaction likh nahi rahe
👉 Tum declare kar rahe ho

"Is method ko transaction me chalao"

Spring bolta hai:

"Samajh gaya, baaki main handle karunga"

🔹 6. Proxy ka role yahi hai

Proxy hi internally ye karta hai:

BEGIN → METHOD → COMMIT / ROLLBACK

Tum directly nahi dekh pa rahe
👉 isliye confusing lag raha hai
