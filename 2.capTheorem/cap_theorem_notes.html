The nightmare: India DB shows balance = 800, USA DB shows 1000 after a network cut. Both users withdraw. Result: double spending and financial disaster. This is why CAP theorem exists.
CAP theorem: In a distributed system with replicated data, you can only guarantee two of these three properties at the same time.
C
Consistency
All nodes return the same data. Every read returns the most recent write — never stale data.
A
Availability
Every request gets a response — even if it's an error. The system never goes silent. It always replies.
P
Partition tolerance
The system keeps working even when the network between nodes breaks. Real networks always fail eventually.
Why can't we have all three? When a partition splits India DB and USA DB, a write reaches only one side. You must choose:

Return stale data → keeps Availability but breaks Consistency
Reject the request → keeps Consistency but breaks Availability

P is non-negotiable in real distributed systems — networks always fail. So the real choice is always CP or AP.
AP = Availability + Partition Tolerance. System keeps responding even when nodes disagree. Sacrifices strong consistency in favour of eventual consistency.
During partition

Server A = 6, Server B = 5. User hits Server A → gets 6. User hits Server B → gets 5. System stays online with temporarily different values.

After partition heals

Nodes synchronise and resolve the conflict. Data eventually converges to the correct value — this is called eventual consistency.

Eventual consistency: Data will become consistent — just not immediately. Like two supermarket branches selling from different stock counts, then syncing inventory at end of day.
DatabaseGood use cases
Cassandra DynamoDB CouchDB Riak Social media likes, messaging, recommendation feeds, large-scale analytics — anywhere a slight delay in consistency is acceptable.
CP = Consistency + Partition Tolerance. All nodes always return the same data. Sacrifices availability — during a partition the system may stop responding rather than return wrong data.
Core mechanism

Writes require majority agreement across nodes (Raft / Paxos / Zookeeper). Only after majority approves does the value change. Minority nodes stop responding during partition.

The bank example

Balance must be exact. Inconsistent data → double withdrawal → negative balance. CP systems reject the write rather than risk wrong data. Correctness over availability.

Why leader election? One node is elected leader — all writes go through it to maintain strict ordering. If the leader goes down, the system pauses until a new leader is elected (Raft / Paxos).
DatabaseGood use cases
HBase MongoDB (majority writes) Zookeeper etcd Banking, payments, inventory (avoid overselling), distributed locks — anywhere correctness is non-negotiable.
CA = Consistency + Availability. Works perfectly — but only when there is no network partition. Since partitions always happen in real distributed systems, CA is mostly a single-node concept.
How it works

Single source of truth. All clients read and write to the same database. No replication = no conflict. Data is always correct and the system always responds.

The limitation

The moment you add a second server, partitions become possible and you must choose CP or AP. CA is no longer achievable at distributed scale.
