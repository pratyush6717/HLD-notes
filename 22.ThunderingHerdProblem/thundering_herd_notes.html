Why auto scaling doesn't save you
Auto scaling timeline
  • Scale decision: ~30 seconds
  • New instance boot: ~1 minute
  • Total: ~90 seconds to help
Traffic spike timeline
  • Spike hits: second 0
  • System crashes: second 2–5
  • Auto scaling arrives: 90 seconds too late
System doesn't just fail — it collapses in a chain reaction. Retries make it worse, not better.
Step-by-step collapse
Step 1: 1M requests arrive at 12:00:00 ↓ Step 2: Thread pools full, queues overflow ↓ Step 3: Requests rejected — HTTP 503, timeouts ↓ users + browsers retry automatically Step 4: Retry storm — 1M → 2M → 3M requests ↓ Step 5: CPU 100%, memory exhausted, latency spikes ↓ 10ms → 2000ms Step 6: More timeouts → more retries → full crash
Retry storm = the killer. Each failed request triggers a retry. System receives 3× the original traffic. This is the cascading failure pattern — and it's why systems like IRCTC crash even though they have auto-scaling.

Solution 1 — Exponential backoff
Formula
wait = base × 2^n retry1 = 200ms retry2 = 400ms retry3 = 800ms retry4 = 1600ms
Problem with plain backoff
  • All clients use same formula
  • All retry at exactly same time
  • Still creates synchronized spike
Solution 2 — Jitter (randomness)
Formula
retry_time = random(0, base × 2^n) Client 1 → retry in 300ms Client 2 → retry in 450ms Client 3 → retry in 120ms
Traffic spreads → no synchronized spike → system recovers

Solution 3 — Rate limiter (most important)
Token bucket algorithm
Bucket has fixed tokens (= system capacity). Each request consumes one token. When bucket empty — excess requests wait or are rejected.
Capacity: 5000 req/sec 12:00:00 → 1M requests arrive Token bucket: allows 5000 Remaining 995,000 → wait / reject Backend is protected.
Why rate limiter works
  • Backend never receives more than capacity
  • Excess requests rejected cleanly (HTTP 429)
  • System stays stable under any traffic
  • Sits at API Gateway — before backend even sees request
Solution 4 — Virtual waiting room (BookMyShow technique)
User enters at 12:00 → assigned position: #25342
System allows 500 users/second into booking
User sees: "Estimated wait: 50 seconds"
No thundering herd — flow is controlled and metered
Production architecture for flash sale / concert booking. Every layer adds protection — CDN absorbs static traffic, rate limiter controls flow, queue smooths bursts, workers process at their own pace.
Naive architecture (crashes)
Users (1M) ↓ Load Balancer ↓ Ticket Service ↓ Database
All 1M hit DB → crash
Production architecture (survives)
Users (1M) ↓ CDN (static assets cached) ↓ API Gateway ↓ Rate Limiter (5000/sec max) ↓ excess → virtual queue Kafka / SQS Queue ↓ workers consume at own pace Booking Workers ↓ Database
