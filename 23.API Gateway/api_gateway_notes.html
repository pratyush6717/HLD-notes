API composition
Client sends 1 request. Gateway calls Product, Invoice, Review, Recommendation services internally. Merges responses. Returns combined result. Client sees no complexity.
Authentication
JWT verified once at gateway. Valid → forward to microservice. Invalid → reject immediately. No auth code needed in individual services.
Rate limiting
10 requests/minute per user. 11th request → HTTP 429. Prevents DDoS and API abuse. Token bucket algorithm. Per user, per IP, or per API key.
Service discovery
Microservice instances change dynamically. Gateway asks Eureka/Consul: "where is Order Service?" Gets current IP. Routes there. No hardcoded URLs.
Request / response transform
Client sends GET /orders. Gateway rewrites to GET /orders?userId=123. Response transformed to mobile-friendly format.
Caching + logging
Product catalog cached at gateway — no service call needed. Every request/response logged with latency, status codes. Central observability.

API Gateway itself must be highly available. It's the single entry point — if it goes down, everything goes down. Production: Gateway runs as a cluster, behind a DNS load balancer, across multiple regions and availability zones.
How gateway handles millions of requests
Client (global) ↓ DNS Load Balancer (Route53 / Cloudflare) ↓ routes to nearest region
API Gateway Cluster (Mumbai) ↓ Microservices (Mumbai)
API Gateway Cluster (Singapore) ↓ Microservices (Singapore)
Multi-region
Deploy gateways in multiple regions. DNS routes user to nearest region.
India user → Mumbai region
Singapore user → Singapore region
Lower latency. Disaster recovery.
Availability zones
Each region = multiple data centers. AZ1 fails → AZ2 takes over automatically.
