The core problem: In microservices, each service has its own DB. A single user action spans multiple services and DBs. If one step fails mid-way, the other steps have already committed — leaving the system in an inconsistent state.
E-commerce disaster
  • User buys item — 2 things must happen
  • Step 1: Order create → success
  • Step 2: Inventory reduce → failed
  • Result: Order exists, inventory not reduced
  • Product oversold
Banking disaster
  • Transfer ₹100 from A to B
  • Step 1: Debit A → success (A = ₹0)
  • Step 2: Credit B → failed (B = ₹50)
  • Result: ₹100 vanishes into thin air
  • Financial disaster
3 solutions — overview
2PC
Coordinator asks all services to prepare, then commits or aborts. Strong consistency. But coordinator crash = system blocked.
3PC
Adds a Pre-Commit phase so participants know the decision before final commit. Non-blocking. Complex — rarely used in practice.
Saga
Break big transaction into small local ones. Failure → run compensation (rollback) actions. Async via events. Used everywhere in microservices.
Travel booking analogy: Flight booked ✔ → Hotel booked ✔ → Taxi failed ✗. Saga compensation: cancel hotel → cancel flight. Each step has a forward action and a rollback action.
2PC = Two Phase Commit. A coordinator manages all participants. Two phases: Prepare (vote) and Commit/Abort (decide). Strong consistency but blocking on coordinator failure.
Phase 1 — Prepare (voting)
Coordinator ↓ "Can you commit?"
Order DB Inventory DB
↓ both reply
YES YES
Phase 2 — Decision
Coordinator ↓ all YES → sends COMMIT
Order DB ✔ Inventory DB ✔
Any NO → ABORT
Order DB ✗ Inventory DB ✗
The blocking problem — what-if sandbox
Coordinator crashes
Participants in "prepared" state — locks held. They can't decide alone. System blocked until coordinator recovers.
OK message lost
Coordinator thinks participant failed → sends ABORT. Transaction rolled back unnecessarily.
Commit message lost
Participants prepared but never receive COMMIT. They wait forever. Locks held indefinitely.
Pros
  • Strong consistency — all or nothing
  • Well understood protocol
  • Good for small number of services
Cons
  • Blocking — coordinator crash = system stuck
  • Locks held during both phases — latency
  • Not suited for large microservice systems
3PC = Three Phase Commit. Adds a Pre-Commit phase between Prepare and Commit. Now participants know the coordinator's decision before the final commit — so they can proceed even if coordinator crashes. Non-blocking.
1
Prepare — same as 2PC. Coordinator asks "can you commit?" Participants lock resources and vote YES/NO.
2
Pre-Commit — coordinator broadcasts the decision (PRE-COMMIT). Participants now know the outcome before final commit. If coordinator crashes here, participants can proceed independently.
3
Commit — final COMMIT signal sent. Participants commit and release locks.
Pros over 2PC
  • Non-blocking — participants can decide even if coordinator crashes
  • Participants already know decision after Phase 2
Why rarely used
  • Complex to implement correctly
  • Requires synchronous network assumptions
  • Saga solves the same problem more practically
In practice: 3PC is mostly theoretical. Real systems either use 2PC for strong consistency (banking) or Saga for distributed microservices. You mention 3PC in interviews to show you know the evolution — not because you'd implement it.
Saga = most used in microservices. Instead of one big transaction, break it into small local transactions. Each has a forward action and a compensation (rollback) action. Communication via events — no global lock.
Happy path — all success
T1: Order create ✔ ↓ event published T2: Inventory reduce ✔ ↓ event published T3: Payment ✔ ↓ event published T4: Shipping ✔
Failure path — compensation runs
T1: Order create ✔ ↓ T2: Inventory reduce ✔ ↓ T3: Payment ✔ ↓ T4: Shipping FAILED ✗ ↓ compensation triggers C3: Refund payment ↓ C2: Restore inventory ↓ C1: Cancel order
Choreography vs Orchestration
Choreography
Services react to events independently. No central controller.
Order Service → publishes event ↓ Inventory Service reacts ↓ Payment Service reacts
✔ Simple, no central dependency
✗ Hard to debug, circular dependencies possible
Orchestration
Central orchestrator calls each service in sequence. Controls rollback on failure.
Orchestrator ↓ calls each Order → Inventory → Payment ↓ failure → triggers rollback Compensation sequence
✔ Easier to monitor and debug
✗ Orchestrator = single point of concern

Final intuition: 2PC → strong consistency, blocking risk. 3PC → non-blocking improvement, rarely used. Saga → eventual consistency, event-driven, scales with microservices. Real production systems almost always choose Saga for distributed workflows.
