The core pain: Service writes to DB and publishes to Kafka. These are 2 separate operations. No atomicity. Either can fail independently → system goes inconsistent.
DB write ✔ but Kafka publish ✗
  • Loan created in DB
  • Kafka event never sent
  • Notification Service never notified
  • User doesn't get notified — silent failure
DB write ✗ but Kafka publish ✔
  • Kafka event fires
  • Other services react and process
  • But actual data never saved in DB
  • Ghost data / wrong system state
Why not 2PC? Kafka doesn't properly support 2PC. It's slow (multiple network roundtrips), and coordinator failure blocks the entire system. 2PC = theoretical solution, not practical.
Mental model:
Never: DB write + Kafka publish separately
Always: Single source of truth → then async publish

Consistency > immediacy

Core idea: Don't call Kafka directly. Store the event in the DB (Outbox table) in the same transaction as the business data. A separate poller reads the outbox and publishes to Kafka.
Outbox flow
Service receives request ↓ begin transaction
Same DB Transaction
User/Loan table INSERT Outbox table INSERT
↓ commit (atomic) Poller Service (async) ↓ reads outbox + marks published Kafka Event Published ↓ Notification Service consumes
Restaurant analogy
Wrong: Write order in register AND separately update delivery app (can mismatch). Right: Write order in register + delivery note in same register. One person then takes delivery note and updates the app. Always consistent.
Trade-offs to mention
✗Small delay — event not published immediately (poller runs every N ms)
✗Duplicate events possible if poller retries → fix with idempotency
✗Ordering issues with multiple pollers → fix with Kafka partition key
✔DB + Event always in sync — atomicity guaranteed
Core idea: Flip the flow. Don't write to DB first — write the event first. Then your own service consumes that event and writes to DB. Event = single source of truth.
Listen-to-Yourself flow
Service receives request ↓ step 1 Write event to Outbox ↓ Poller → Kafka ↓ service also listens to own topic Same service consumes event ↓ step 2 (DB write happens here) Write to DB
The aha moment
  • Event is the source of truth
  • DB is derived from events
  • Perfect for event-sourced systems
  • Decoupled write path
The problem
  • GET request arrives immediately after write
  • DB still empty — event not yet consumed
  • User sees: data not found
Fix: Write-through cache (Redis) — read from cache immediately, DB updated async
Core idea: Every DB write creates a log entry (binlog). CDC tool (Debezium) reads those logs and automatically creates Kafka events. No changes to application code needed.
CDC flow
Service writes to DB ↓ DB writes binlog (always happens) DB Binary Log (binlog) ↓ Debezium (CDC tool) reads log ↓ auto-creates event Kafka Event ↓ Downstream consumers
Why it's powerful
  • Zero application code changes
  • Event auto-generated from any DB write
  • Works with legacy systems
  • Captures all changes — inserts, updates, deletes
Problems
  • Duplicate events if Debezium retries
  • Ordering issues with high write volume
  • Slight latency — log write delay
  • Extra infrastructure (Debezium cluster)

Interview definition: Dual write problem occurs when we try to update two systems (DB and message broker) without atomicity, leading to inconsistent states.

Interview solution: Use Transactional Outbox — store both DB change and event in the same transaction, then a separate process publishes events asynchronously.
